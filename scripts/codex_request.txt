## Part 1: small attribution hardening patch

### Problem

Current canonical cost filter classification is likely too exact. It may identify:

```text
strategy_filters.cost_edge
pre_trade_economics.net_edge_below_minimum
```

but miss legitimate canonical variants such as:

```text
strategy_filters.filtered entry: cost_edge
strategy_filters.filtered_entry_cost_edge
strategy_filters.entry_edge_below_required
pre_trade_economics.edge_below_required
pre_trade_economics.net_edge_below_minimum
```

If those variants exist in stored contexts, the report may count the block layer correctly but fail to mark `blocked_by_cost_filter=True`.

### Required change

Add a narrow helper in `decision_attribution.py` such as:

```python
def _has_layer_reason_containing(
    all_block_reasons: Sequence[str],
    layer: str,
    *needles: str,
) -> bool:
    ...
```

It should:

* parse existing normalized `layer.reason` strings with `split_block_reason`
* match only within the given layer
* compare reason text case-insensitively
* ignore malformed reasons
* avoid broad matching across unrelated layers

Then update `canonical_cost_filter` so it remains contract-first but recognizes safe cost/edge variants within only these layers:

```text
strategy_filters
pre_trade_economics
```

Accept variants containing conservative terms such as:

```text
cost_edge
edge_below
edge
net_edge_below_minimum
```

Do **not** make every `strategy_filters.*` reason a cost filter. Only edge/cost-related reasons should set `blocked_by_cost_filter=True`.

### Required tests

Update `tests/test_decision_attribution.py` with targeted tests:

1. Cost edge phrase variant:

```python
{
    "raw_signal": "BUY",
    "final_signal": "HOLD",
    "all_block_reasons": ["strategy_filters.filtered entry: cost_edge"],
}
```

Expected:

```text
blocked_by_cost_filter=True
block_reason_counts includes strategy_filters.filtered entry: cost_edge
```

2. Underscore variant:

```python
{
    "all_block_reasons": ["strategy_filters.filtered_entry_cost_edge"]
}
```

Expected:

```text
blocked_by_cost_filter=True
```

3. Pre-trade economics edge variant:

```python
{
    "all_block_reasons": ["pre_trade_economics.edge_below_required"]
}
```

Expected:

```text
blocked_by_cost_filter=True
```

4. Non-edge strategy filter should not be misclassified:

```python
{
    "all_block_reasons": ["strategy_filters.market_regime_block"]
}
```

Expected:

```text
blocked_by_cost_filter=False
```

This is a hardening patch, not a redesign.

## Part 2: extract replayable SMA strategy config

After the small attribution hardening patch, add a config boundary for SMA strategy parameters.

### Goal

Create a stable configuration object so future replay/sweep can generate many strategy configurations without mutating environment variables or reading global `settings` inside the strategy path.

The goal is **not** to remove `settings` entirely. The goal is to move the boundary:

```text
CLI / runtime layer:
  settings → SmaStrategyConfig

strategy layer:
  SmaStrategyConfig values

future replay / sweep layer:
  many SmaStrategyConfig values → comparable DecisionAttribution summaries
```

### Add file

Prefer:

```text
src/bithumb_bot/strategy_config.py
```

Define:

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class SmaStrategyConfig:
    short_n: int
    long_n: int
    pair: str
    interval: str
    exit_rule_names: tuple[str, ...]
    exit_max_holding_min: int
    exit_min_take_profit_ratio: float
    exit_small_loss_tolerance_ratio: float
    slippage_bps: float
    live_fee_rate_estimate: float
    entry_edge_buffer_ratio: float
    strategy_min_expected_edge_ratio: float
    buy_fraction: float
    max_order_krw: float
```

Add a factory:

```python
def sma_strategy_config_from_settings(
    *,
    short_n: int | None = None,
    long_n: int | None = None,
) -> SmaStrategyConfig:
    ...
```

It should read from the repo’s existing settings object and preserve current defaults.

Normalize `exit_rule_names` as an immutable tuple. Reuse existing parsing logic if available, or move the parser to avoid duplication.

### Update `SmaCrossStrategy`

Adapt `src/bithumb_bot/strategy/sma.py` so `SmaCrossStrategy` can be constructed from this config without changing existing public behavior.

Acceptable approaches:

```python
@dataclass(frozen=True)
class SmaCrossStrategy:
    config: SmaStrategyConfig
```

or, less invasive:

```python
@dataclass(frozen=True)
class SmaCrossStrategy:
    short_n: int
    long_n: int
    ...
    
    @classmethod
    def from_config(cls, config: SmaStrategyConfig) -> "SmaCrossStrategy":
        ...
```

Choose the least disruptive option that still gives future replay/sweep a stable config object.

Existing call sites must continue to work. If the app currently instantiates:

```python
SmaCrossStrategy(short_n, long_n)
```

that should still work, or all call sites must be updated safely.

### Remove direct settings reads from entry intent context

Currently `_build_entry_intent_context()` may read:

```python
settings.BUY_FRACTION
settings.MAX_ORDER_KRW
```

This is sweep-relevant and should come from config.

Change from:

```python
_build_entry_intent_context(pair=pair)
```

to something like:

```python
_build_entry_intent_context(
    pair=pair,
    buy_fraction=config.buy_fraction,
    max_order_krw=config.max_order_krw,
)
```

or, if keeping the existing dataclass fields:

```python
_build_entry_intent_context(
    pair=self.pair,
    buy_fraction=self.buy_fraction,
    max_order_krw=self.max_order_krw,
)
```

The stored decision context should still show the same `budget_fraction_of_cash` and `max_budget_krw` values as before when constructed from settings.

### Do not move runtime-only live gates into replay config

Do not treat this as a replay policy:

```python
_live_armed_entry_fee_authority_blocks()
```

That function depends on runtime live mode safety settings. Leave it as runtime-only behavior. You may add a short comment clarifying that this is runtime/live safety policy and should not become a replay/sweep parameter in this patch.

Do not change live gate behavior.
