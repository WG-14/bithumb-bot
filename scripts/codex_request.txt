## Mission

Do **not** solve this by flattening the current BTC position, using `resume --force`, loosening all halt gates, deleting lots, hiding blockers, or adding another manual DB repair path.

The goal is to make the bot safer and more operational by allowing it to resume in a **position-management mode** when it already holds a broker-confirmed, accounting-confirmed, projection-confirmed open position.

Core principle:

```text
Open exposure present
≠ unsafe
≠ must flatten
≠ cannot run
```

A normal automated trading bot must be able to restart with an existing canonical position and continue managing that position. It may block additional BUY entries while a position is open, but it should not require manual flattening before it can run.

`RISK_STATE_MISMATCH` must not become a permanent blocker after the mismatch has been repaired. Add or update a safe revalidation path that clears a stale `RISK_STATE_MISMATCH` only when current evidence proves the system is coherent.

Candidate clear conditions:


### 2. Add or expose an explicit `OPEN_POSITION_READY` / position-management resume mode

A clean open position should be a normal operational state.

For this state:

```text
canonical_state=OPEN_EXECUTABLE
broker_portfolio_converged=true
```

```text
exit_allowed=true
new_entry_allowed=false or strategy/risk-controlled
closeout_allowed=true if normal exit authority is available
flatten_primary_recommendation=false
recommended_action=resume_position_management

```

This is not `resume --force`. It is normal resume after current evidence has been verified.

### 3. Separate run-loop permission from new-entry permission

Do not require the bot to be flat before the run loop can operate.

Expected semantics:

```text
run_loop_allowed:
  process can run and evaluate strategy/risk/exit logic

position_management_allowed:
  existing canonical position may be managed

new_entry_allowed:
  new BUY entries may be placed

exit_allowed / closeout_allowed:
  existing position may be reduced or exited through normal strategy/SELL authority

emergency_flatten_allowed:
  emergency exposure reduction path is available

flatten_primary_recommendation:
  flatten is the primary recommended next action only for true emergency/risk states
```

For clean open exposure:

```text
run_loop_allowed=true
position_management_allowed=true
exit_allowed=true
new_entry_allowed=false unless strategy explicitly permits pyramiding
emergency_flatten_allowed=true
flatten_primary_recommendation=false
```

### 4. Restrict `HALT_RISK_OPEN_POSITION` and flatten recommendation

`HALT_RISK_OPEN_POSITION` must not be emitted merely because open exposure exists.

Do not recommend `flatten-position` as the next action when all of the following are true:

```text
canonical open exposure exists
broker_portfolio_converged=true
accounting_projection_ok=true
lot_projection_converged=true
broker_qty_known=true
active fee issue=0
open orders=0
risk breach=0
```

Flatten may be primary only for:

```text
kill switch
daily loss limit
position loss limit
panic stop
explicit operator flatten request
normal position-management path impossible and emergency exposure reduction is required
```

Operator-facing guidance for clean open exposure should say:

```text
Canonical open exposure is present and accounted.
Resume in position-management mode is allowed.
New entries are blocked until the existing position is closed or risk policy allows additional entries.
Emergency flatten is available but not recommended as the primary action.
```

If blocked, guidance must say the actual blocker:

```text
current broker mismatch
active fee issue
unresolved order
projection drift
stale/missing broker snapshot
risk loss limit
panic stop
```

not merely:

```text
open exposure exists
```

### 5. Preserve fail-closed behavior for unsafe open exposure

Do not loosen safety gates globally.

Still block run/resume if any of these are true:

```text
broker/portfolio mismatch
active fee pending
unresolved local order
stray remote open order
submit_unknown
projection drift
broker snapshot stale/missing
risk loss limit breach
kill switch
panic stop
source conflict
```

In those cases, `recovery-report`, `repair-plan`, `restart-checklist`, and `health` should point to diagnosis/repair or emergency action according to cause.
