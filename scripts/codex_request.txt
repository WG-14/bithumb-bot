You are working in the `WG-14/bithumb-bot` repository.

Your task is to inspect the current `main` branch, verify the remaining risks directly in the code, and implement the next structural hardening patch. Do not treat this as a generic cleanup. Use this as the starting diagnosis, but verify it against the repository before changing code.

## Core problem statement

The repository now has typed paper/live evidence validation, but the validation may still trust `payload["thresholds"]` from the evidence artifact itself. If true, an evidence producer can weaken its own promotion criteria by writing thresholds such as:

```json
{
  "thresholds": {
    "min_observation_seconds": 1,
    "min_decision_count": 0,
    "min_closed_lifecycle_count": 0,
    "max_blocked_decision_ratio": 1.0,
    "max_execution_quality_breach_count": 999
  }
}
```

That would mean the repo has “typed evidence,” but not a repo-trusted promotion policy. This is not acceptable for an automated trading experiment system. Promotion policy must be owned by the repository/runtime contract, not by the evidence artifact being evaluated.

## Mission

Verify or falsify the hypotheses below, then implement the smallest coherent design change that closes the high-priority risks without weakening existing fail-closed behavior.

Do not center this patch on `strategy_performance.py`. That file should remain an operational closed-lifecycle guard unless repository evidence proves otherwise. This task is about **trusted evidence policy, semantic promotion safety, observability, and operator recovery**.

Do not remove typed evidence validation.

Do not auto-apply profiles to env files.

## Design requirements

### 1. Add a trusted evidence validation policy

Add a repo-owned policy model, preferably in `evidence_chain.py`.

Example shape:

```python
@dataclass(frozen=True)
class EvidenceValidationPolicy:
    min_observation_seconds: float
    min_decision_count: int
    min_closed_lifecycle_count: int
    max_blocked_decision_ratio: float
    max_execution_quality_breach_count: int
    require_execution_quality_applicable: bool = True
    require_db_data_fingerprint: bool = True
```

Use existing naming/style conventions if the repository has better patterns.

Define separate default policies for:

* paper validation evidence
* live readiness evidence

The exact numeric defaults should be conservative but testable. If production-level thresholds are not yet known, pick explicit minimal defaults and document them as policy defaults, not artifact-declared rules. Do not use zero thresholds unless there is a strong existing repository reason.

The evidence artifact’s `thresholds` field may remain as report metadata, but it must not be the sole source of truth.

Acceptable approaches:

* Ignore artifact thresholds for pass/fail and use only trusted policy.
* Or require artifact thresholds to be at least as strict as trusted policy, then use the stricter of the two.

Fail closed if artifact thresholds are weaker than trusted policy.

Add stable reason codes such as:

```text
paper_validation_evidence_policy_threshold_too_weak:min_observation_seconds
paper_validation_evidence_policy_threshold_too_weak:min_decision_count
paper_validation_evidence_policy_threshold_too_weak:min_closed_lifecycle_count
paper_validation_evidence_policy_threshold_too_weak:max_blocked_decision_ratio
paper_validation_evidence_policy_threshold_too_weak:max_execution_quality_breach_count

live_readiness_evidence_policy_threshold_too_weak:min_observation_seconds
live_readiness_evidence_policy_threshold_too_weak:min_decision_count
live_readiness_evidence_policy_threshold_too_weak:min_closed_lifecycle_count
live_readiness_evidence_policy_threshold_too_weak:max_blocked_decision_ratio
live_readiness_evidence_policy_threshold_too_weak:max_execution_quality_breach_count
```

### 2. Validate `db_data_fingerprint`

Make `db_data_fingerprint` required by default.

Reject:

* missing field
* empty string
* non-string value
* value that does not start with `sha256:`

Add reason code:

```text
paper_validation_evidence_schema_invalid:db_data_fingerprint
live_readiness_evidence_schema_invalid:db_data_fingerprint
```

If a policy allows fingerprint absence in some future compatibility mode, it must be explicit and tested. Do not silently allow absence by default.

For live readiness, reject `execution_quality_status=not_applicable` by default.

Reason code:

```text

```

For paper validation, only allow `not_applicable` if the trusted paper policy says so. If allowed, keep it visible in reports and do not mark it as stronger than a real pass.
