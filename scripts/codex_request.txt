## Core problem statement

The repository should not become a system where operators manually tune env values until a backtest looks good.

It should become a system where:

```text
hypothesis
→ manifest
→ deterministic dataset fingerprint
→ parameter-space evaluation
→ out-of-sample validation
→ real rolling walk-forward validation
→ stability checks
→ candidate profile hash
→ promotion integrity check
→ operator-reviewed artifact
→ paper validation consideration
```

The next patch must improve the research system’s validation power, auditability, recoverability, and operator visibility.

## Preserve priority order

Use this priority order throughout:

```text
P0: Live safety, correctness, live/paper isolation, wrong-order prevention, duplicate-order prevention, ledger consistency, restart/recovery behavior
P1: Reproducibility, deterministic artifacts, path policy, promotion integrity, operator observability, recoverability
P2: Strategy tuning, parameter search convenience, profitability
```

Any P2 change that weakens P0 or P1 is unacceptable.

## Keep these hypotheses explicit

### Hypothesis A

The initial research patch added the correct structural layer, but the current promotion gate is still audit-weak because it does not fully prove that the candidate profile being promoted is identical to the candidate profile evaluated in the report.

Test this by inspecting `src/bithumb_bot/research/promotion_gate.py`.

Expected structural fix:

* Recompute `sha256_prefixed(build_candidate_profile(candidate))` inside promotion.
* Refuse promotion if the recomputed hash does not equal the candidate’s stored `candidate_profile_hash`.
* Report the refusal reason as `candidate_profile_hash_mismatch`.

### Hypothesis B

The current walk-forward command is only a scaffold, not real rolling walk-forward validation.

Test this by inspecting `src/bithumb_bot/research/validation_protocol.py`.

Expected structural fix:

* Add optional `manifest.walk_forward` configuration.
* Generate rolling train/test windows.
* Evaluate candidates across multiple windows.
* Report per-window and aggregate walk-forward evidence.
* If `walk_forward_required=true`, promotion must not pass without real walk-forward evidence.

### Hypothesis C

The current parameter stability score is placeholder logic and can allow candidates to pass stability merely because the parameter grid has enough candidates.

Expected structural fix:

* `parameter_stability_required=true` must have real meaning.

* Unknown strategy names must fail clearly.
* Do not add new live trading strategies.

### Hypothesis E

The repo has documentation for research validation, but it still lacks an operator runbook that explains the practical route from research artifact to paper validation consideration.

Expected structural fix:

* Add `docs/runbooks/research-to-paper.md`.
* Explain exact commands, artifact review, manual env/profile consideration, paper/dry-run observation, and what must not be automated.

## Task 2: Replace placeholder parameter stability scoring

Inspect:

```text
src/bithumb_bot/research/validation_protocol.py
src/bithumb_bot/research/parameter_space.py
tests/test_research_backtest_reproducibility.py
```

Current suspected weakness:

```text
_parameter_stability_scores returns 1.0 for all candidate sets with len >= 3.
```

Replace this with a real neighbor-based scoring method.

Design guidance:

* Define neighboring candidates as candidates differing by one parameter-grid step where possible.
* Score should reflect whether nearby candidates also perform acceptably.
* A candidate that only wins as an isolated spike must receive a low score.
* A candidate in a broad stable plateau must receive a higher score.
* Use validation or OOS metrics for stability, not train-only performance.
* If stability cannot be computed, report `None` and fail when `parameter_stability_required=true`.

Acceptable scoring model:

```text
neighbor_score = number_of_neighbors_with_positive_validation_return_and_gate_compatible_metrics / total_neighbors
```

or a better deterministic equivalent.

Requirements:

* Keep the score deterministic.
* Include score in candidate artifacts.
* Include enough diagnostic fields to understand why the score passed or failed.
* Do not let `parameter_stability_required=true` pass simply because the candidate count is >= 3.

Add tests covering:

```text
stable plateau → high stability score / can pass
isolated spike → low stability score / fails when parameter_stability_required=true
too few neighbors → None or low score / fails when required
```

Suggested test file:

```text
tests/test_research_parameter_stability.py
```

## Task 3: Upgrade walk-forward to rolling windows

Inspect:

```text
src/bithumb_bot/research/experiment_manifest.py
src/bithumb_bot/research/validation_protocol.py
src/bithumb_bot/research/dataset_snapshot.py
examples/research/sma_filter_manifest.example.json
docs/research-validation.md
```

Add optional manifest config:

```json
"walk_forward": {
  "train_window_days": 180,
  "test_window_days": 30,
  "step_days": 30,
  "min_windows": 6
}
```

Validation requirements:

* All values must be positive integers.
* `min_windows` must be > 0.
* Invalid config must fail manifest validation.
* If `walk_forward_required=true`, missing `walk_forward` config should fail validation or fail the walk-forward gate with an explicit reason.

Implement rolling windows:

```text
window_001:
  train: start → start + train_window_days
  test: train_end + 1 → train_end + test_window_days

window_002:
  train/test shifted by step_days

repeat until no complete test window remains
```

Requirements:

* Use available manifest date ranges deterministically.
* Report each window’s train/test date ranges.
* Report candidate metrics per window.
* Report aggregate metrics:

  * window_count
  * pass_window_count
  * fail_window_count
  * mean_test_return_pct
  * median_test_return_pct if easy without new dependencies
  * worst_test_return_pct
  * return_consistency_pass
* If fewer than `min_windows` are available, fail with `walk_forward_insufficient_windows`.
* If `walk_forward_required=true`, candidate promotion must require real walk-forward evidence, not just train/validation consistency.

Add tests covering:

```text
valid walk_forward config parses
invalid walk_forward config is rejected
rolling windows are generated deterministically
walk_forward_required refuses missing evidence
insufficient windows fails clearly
candidate with repeated positive test windows passes aggregate walk-forward
candidate with inconsistent windows fails aggregate walk-forward
```

Suggested test file:

```text
tests/test_research_walk_forward.py
```
