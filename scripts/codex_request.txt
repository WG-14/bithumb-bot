## Design direction

Add a dedicated module, preferably:

```text
src/bithumb_bot/decision_attribution.py
```

This module should define a first-class attribution domain model. Suggested shape:

```python
@dataclass(frozen=True)
class DecisionAttribution:
    raw_signal: str
    final_signal: str
    decision_type: str
    base_reason: str
    entry_reason: str
    entry_block_reason: str | None
    primary_block_layer: str
    primary_block_reason: str
    blocked_by_cost_filter: bool
    blocked_by_fee_authority: bool
    blocked_by_position_gate: bool
    blocked_by_order_rule: bool
    blocked_by_performance_gate: bool
    gap_ratio: float | None
    required_edge_ratio: float | None
    signal_strength_label: str
    submit_expected: bool | None
    execution_block_reason: str | None
    target_block_reason: str | None
```

The exact fields may be extended if the repo already exposes useful canonical equivalents. However, do not remove the core diagnostic intent:

* distinguish raw signal from final signal
* distinguish strategy filter blocks from position/order/execution/performance blocks
* preserve edge economics
* preserve submit expectation
* support aggregation by cause

Add normalization functions along these lines:

```python
def normalize_decision_attribution_from_context(context: dict[str, Any]) -> DecisionAttribution:
    ...

def normalize_decision_attribution_from_row(row: sqlite3.Row) -> DecisionAttribution:
    ...

def summarize_decision_attributions(
    rows: Iterable[DecisionAttribution],
) -> DecisionAttributionSummary:
    ...
```

The normalizer must be defensive:

* tolerate missing keys
* tolerate older context versions
* distinguish unknown from false where possible
* never crash the report because one row has malformed JSON
* expose skipped/malformed row counts in the summary

The `interpretation` section should be conservative and rule-based. It must not overclaim. For example:

* If raw BUY is near zero, suggest `raw_buy_scarcity`.
* If raw BUY exists but final BUY is near zero and many are cost-filtered, suggest `entry_edge_insufficient_or_cost_filter_strict`.
* If final BUY exists but `submit_expected=false`, suggest `execution_intent_or_order_rule_block`.
* If gap is usually below required edge, suggest `gap_below_required_edge`.
* If malformed context is high, suggest `observability_schema_incomplete`.

Keep these as diagnostic labels, not trading advice.
