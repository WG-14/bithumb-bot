You are working in the GitHub repository `WG-14/bithumb-bot`.

Your task is to investigate and implement a structural fix for a repeated live-trading failure pattern observed on EC2. Do not treat the current diagnosis as proven fact. Treat it as a set of hypotheses to verify or falsify directly against the repository code, tests, and available runtime evidence. The goal is not to suppress the immediate symptom, but to evolve this repo into a more robust, reproducible, operable automated trading experiment system.

Context source: the current analysis summarized the incident and proposed hypotheses/design direction. Use it as starting context, not as a conclusion. 

The observed operational pattern is:

* Start the live bot.
* One BUY or SELL gets filled on Bithumb.
* The bot stops or becomes non-resumable.
* The system reports fee pending / accounting pending / `RISK_STATE_MISMATCH`.
* Operator or GPT-guided recovery uses `fee-pending-accounting-repair`, `rebuild-position-authority`, and sometimes `flatten-position`.
* Flattening can create another live SELL fill that again enters fee pending, causing the same class of mismatch in the opposite direction.
* Manual DB edits and repeated repairs have accumulated, making lot projection and recovery state harder to reason about.

The most important suspected structural issue is:

> The system currently appears to treat authoritative fee availability as a prerequisite for canonical fill accounting. When Bithumb provides fill principal evidence immediately but fee evidence is missing, ambiguous, or only available at order level, the bot defers applying the entire fill. The exchange balance changes immediately, but local `fills`, `trades`, `portfolio`, `orders.qty_filled`, and lot projection do not. This can produce broker/local mismatch, risk halt, authority projection drift, flatten loops, and repeated manual repair.

Keep this as a hypothesis until verified.

Your work should prioritize design correctness over a narrow symptom patch.

Claim: If a live fill has valid `qty`, `price`, `notional`, and identity, but fee is missing or `order_level_candidate`, the system records a diagnostic observation and marks the order `ACCOUNTING_PENDING` without applying principal to `fills`, `trades`, `portfolio`, and `orders.qty_filled`.

### Hypothesis 4: Position authority / lot projection is acting as an over-authoritative materialized view

Claim: `open_position_lots` / lot projection can drift from canonical holdings after repeated repairs, and the system treats projection drift as a blocker requiring manual rebuild.

Inspect:

* `position_authority_state.py`
* `position_authority_repair.py`
* `lifecycle.py`
* `runtime_readiness.py`

Verify whether every principal fill application deterministically updates or rebuilds lot projection. If not, implement either:

* an idempotent projection rebuild after principal application, or
* a safe incremental update plus invariant check.

Lot projection should be treated as a rebuildable materialized projection, not as a competing source of truth over canonical executed principal.

### Hypothesis 5: Flatten/closeout is being used as recovery and can create new accounting incidents

Claim: `flatten-position` is a live market action, not accounting recovery. When fee pending handling is flawed, flattening a position can create a new SELL with fee pending and cause the same mismatch class again.

* broker/local asset is converged,
* no unapplied principal fill exists,
* no unresolved order exists,
* sell quantity source is explicit and safe,
* fee-only pending is understood and will not block principal application.

Do not recommend or encode flatten as a normal response to unapplied principal accounting.

## Required design direction

Implement principal-first accounting.

The target flow should be:

```text
Broker fill observed
→ validate fill identity, side, price, qty, notional
→ apply fill principal immediately:
     - fills/trades or equivalent principal ledger row is recorded
     - orders.qty_filled is updated
     - portfolio cash/asset principal is updated
     - lot projection is updated or rebuilt
→ classify fee separately:
     - fee_finalized / validated_order_level_paid_fee
     - principal_applied_fee_pending
     - fee_validation_blocked
→ if fee is finalized later:
     - apply only the fee delta / finalization event
     - do not re-apply principal
```

Do not simply remove fee validation. Invalid fee evidence must remain visible and safe.

Preferred implementation approach:

1. Do not destructively rewrite `apply_fill_and_trade` as the first step.
2. Add explicit principal-first functionality, for example:

   * `apply_fill_principal_with_pending_fee(...)`
   * `finalize_fill_fee(...)`
   * or a cleaner equivalent that fits the repo architecture.
3. Extend DB schema or event tables if needed:

   * add fee status fields to `fills`, or
   * add a `fill_fee_events` / `fill_accounting_events` table.
4. Ensure idempotency:

   * applying the same principal fill twice must be a no-op,
   * finalizing fee twice must be a no-op or an explicit no-change result,
   * fee finalization must not duplicate trades or lots.
5. Preserve replayability:

   * accounting replay should reconstruct portfolio from principal + fee events,
   * lot projection should be rebuildable from canonical fill/trade events.
6. Preserve safety:

   * invalid negative fee, invalid price, invalid qty, identity mismatch, and overfill must still block or require operator review.
   * material zero fee should not be silently accepted as authoritative, but it also should not necessarily block principal if principal evidence is valid. Treat it as `principal_applied_fee_pending` or `fee_validation_blocked` according to evidence quality.

## Required readiness policy changes

Split fee/accounting pending states into at least these categories:

```text
unapplied_principal_pending
principal_applied_fee_pending
fee_validation_blocked
fee_finalized
```

Policy guidance:

* `unapplied_principal_pending`:

  * hard block new submissions,
  * auto-reconcile if possible,
  * operator review if bounded retry fails.
* `principal_applied_fee_pending`:

  * should not cause broker/local asset mismatch,
  * should not cause `RISK_STATE_MISMATCH`,
  * may run in degraded mode,
  * may block new entries if strategy policy chooses, but should not be treated as a hard accounting inconsistency.
* `fee_validation_blocked`:

  * operator review required,
  * principal should already be accounted if identity and principal evidence were valid.
* `fee_finalized`:

  * normal state.

Update `runtime_readiness.py`, recovery reports, and startup/restart behavior accordingly.

## Required tests

Add regression tests before or alongside the fix. Include tests that fail on current behavior and pass after the design change.

Minimum test cases:

1. Live BUY with order-level candidate fee applies principal immediately.

Expected:

* `orders.qty_filled` increases by fill qty.
* portfolio asset increases.
* portfolio cash decreases by notional, with fee handled according to pending/finalization model.
* broker/local asset convergence is possible.
* fee state is not lost.
* no `RISK_STATE_MISMATCH` caused solely by fee latency.

2. Live SELL with order-level candidate fee applies principal immediately.

Expected:

* asset decreases immediately.
* local holdings converge with broker holdings.
* fee remains pending/finalizable.
* no opposite-direction mismatch after flatten/closeout-like SELL.

3. Missing fee for material valid fill does not block principal.

Expected:

* principal applied,
* fee state is `principal_applied_fee_pending` or equivalent,
* operator observability contains fee warning,
* no duplicate fill when fee later finalizes.

4. Invalid fee evidence remains safe.

Cases:

* negative fee,
* invalid fee string,
* impossible qty/price,
* overfill,
* identity mismatch.

Expected:

* invalid principal evidence blocks principal.
* invalid fee evidence does not corrupt principal accounting.
* operator report clearly shows what is blocked and why.

5. Fee finalization is idempotent.

Expected:

* finalizing an already-finalized fee does not duplicate fills/trades/lots.
* finalizing a pending fee updates only fee delta/status.
* replay remains stable.

6. MyOrder private stream path matches REST reconciliation path.

Expected:

* `myorder_runtime.ingest_myorder_event` applies principal for valid fill events even when fee is pending.
* It records fee pending separately.
* It does not leave terminal executed volume unaccounted.

7. Lot projection converges after principal apply.

Expected:

* after BUY/SELL principal application, `open_position_lots` or its replacement projection converges with canonical holdings.
* projection rebuild is deterministic.
* repeated repair/rebuild does not inflate total qty.

8. Flatten is not recovery for unapplied principal.

Expected:

* flatten/closeout command refuses or warns when there is any `unapplied_principal_pending`.
* flatten is allowed only after principal convergence.
* flatten SELL uses principal-first accounting.
