## Incident context

On EC2, live trading repeatedly entered this failure cycle after real Bithumb BUY/SELL fills:

```text
Bithumb order fills successfully
→ local fee validation classifies the fill as fee_pending
→ local fills/trades/portfolio/lot accounting becomes incomplete or inconsistent
→ broker/local portfolio mismatch appears
→ startup/resume gates block live
→ recovery suggests flatten or manual repair
→ flatten itself creates another real fill
→ that fill goes through the same fee/accounting failure path
→ operator manually repairs fee/principal/lot state and sometimes bot_health
```

Runtime evidence from the incident:

```text
Actual Bithumb mobile-app fee rate: 0.04% = 0.0004 = ~4 bps

DB-observed examples:
flatten SELL fee_bps = 3.999784
latest BUY fee_bps   = 3.999902
previous SELL        = 3.998178
previous BUY         = 3.999148

Resolved setting during incident:
LIVE_FEE_RATE_ESTIMATE = 0.0025
LIVE_FILL_FEE_ALERT_MIN_NOTIONAL_KRW = 10000.0
```

The likely direct trigger was:

```text
configured expected live fee rate = 0.0025
actual Bithumb paid fee rate      = 0.0004
→ expected_fee_rate_mismatch
→ old behavior: order_level_candidate / fee_pending
→ accounting/recovery cascade
```

But do **not** reduce the fix to `LIVE_FEE_RATE_ESTIMATE=0.0004`. That is operational hygiene, not the repository-level fix. The repository-level fix is that coherent exchange `paid_fee` must outrank configured estimates for settlement accounting.
