## Mission

Investigate and fix the structural cause behind the current class of failures where:

* `portfolio` can be correct,
* `audit-ledger` / replay-based views can be correct,
* but `audit` still fails because the latest `trades.cash_after` / `trades.asset_after` diverge from the current portfolio after later repair/adjustment events.

Do **not** assume that the initial diagnosis is true. Treat it as a leading hypothesis and verify or falsify it against the repo.

The likely issue is not a single bad compare. The likely issue class is that the repo currently has **split accounting truth sources** across:

* execution facts (`fills`, `trades`)
* repair/adjustment events (`external_cash_adjustments`, `external_position_adjustments`, `manual_flat_accounting_repairs`, `fee_gap_accounting_repairs`, `position_authority_repairs`)
* mutable snapshot state (`portfolio`)
* reporting/audit code paths that do not all use the same replay rules

Your job is to verify that, then implement the smallest safe design change that moves the repo toward a **single authoritative accounting projection model**.

Do not trust these statements blindly. Confirm them from the codebase and adjust your plan if the code differs.

## Working hypotheses to test

Keep these as hypotheses unless verified:

### H1 — Most likely immediate failure mechanism

`audit` is using the wrong truth source. It is directly or indirectly comparing the latest trade after-state to the current portfolio, even when later repair/adjustment events have changed the projected authoritative state.

### H2 — Higher-level structural cause

The repository does not currently have one unambiguous accounting source of truth. Different commands and reports derive state from different event families and different replay rules.

### H3 — Fee lifecycle nondeterminism is forcing downstream accounting repairs

Fee data may arrive incomplete first (`missing`, `fee_pending`, `order_level_candidate`, operator-confirmed later), which can cause delayed accounting convergence and event-layer repairs.

### H4 — `trades.cash_after` / `trades.asset_after` are structurally fragile as an authority source

These fields may be valid only as execution-local snapshots, not as final authoritative portfolio state after later repairs.

### H5 — Recovery/resume readiness is ahead of accounting reproducibility

The system may be operationally resumable while still lacking a single deterministic, replayable accounting projection.

For each hypothesis:

* verify or falsify it with direct code inspection and tests
* document the evidence precisely
* do not overclaim

## Required outcome

Produce a fix that prioritizes **design correction over symptom masking**.

Minimum acceptable outcome:

1. `audit`, `audit-ledger`, and reporting paths that claim accounting truth must converge on the same replay/projection rules.
2. The repo must stop relying on latest-trade after-state as authoritative whenever later repair events exist.
3. Tests must cover the exact failure class: portfolio correct, replay correct, latest trade stale, old audit fails, new logic passes.
4. The resulting behavior must be deterministic and idempotent.

Preferred outcome:

* introduce a clearer accounting projection boundary that makes it obvious which events define authoritative state
* move the codebase toward a single append-only accounting event model, without requiring a risky full rewrite in one patch

## Scope and implementation guidance

### Phase 1 — Verify current architecture and failure path

Inspect all relevant code paths used by:

* audit
* audit-ledger
* cash-drift-report
* recovery-report
* health / readiness / reconcile paths
* repair commands

Build a concise internal map of:

* which functions compute portfolio/accounting state
* which tables/event families each command includes
* where latest `trades.cash_after` / `asset_after` are treated as authority
* where `portfolio` is directly mutated instead of projected

If the repo already has a reusable projection/replay abstraction, use it. If not, create one.

### Phase 2 — Unify authoritative accounting replay

Implement a single reusable accounting projection/replay function or module that:

* clearly defines included event families
* clearly defines omitted diagnostic-only event families
* can be used by audit/reporting/recovery code paths
* produces authoritative projected state from canonical inputs

At minimum, the code should make the following distinction explicit:

* **authoritative accounting projection**
* **diagnostic execution-local snapshots**
* **non-authoritative observations**

Do not leave this implicit.

### Phase 3 — Fix audit semantics

Change `audit` so it does **not** fail just because the latest `trades.cash_after` / `asset_after` are stale after later repairs.

Instead, `audit` should validate the right invariants, e.g.:

* projected portfolio state equals authoritative replay state
* projection freshness / metadata are valid
* unresolved fee/repair states are surfaced explicitly
* latest trade after-state mismatch is treated as a non-authoritative stale-snapshot condition if later repairs explain the delta

Do **not** silently discard useful diagnostics. If stale trade after-state exists, preserve that as an explicit warning / detail, not as the primary authoritative failure.

### Phase 4 — Improve recoverability and observability

Add or improve observability so operators can understand:

* which event families were included in projection
* whether post-trade repairs exist
* whether latest trade after-state is stale
* whether fee lifecycle is unresolved
* whether projection is deterministic / converged

Prefer structured outputs or explicit metadata over ambiguous logging.

### Phase 5 — Add deterministic tests and fixtures

Add tests that directly reproduce this problem family.

You must add repo-local tests that cover at least:

1. **Latest trade stale but projection correct**

   * trades last after-state differs from portfolio
   * external/manual repair events explain the delta
   * authoritative replay matches portfolio
   * old failure class is reproduced
   * new audit logic passes or downgrades to explicit stale-snapshot warning

2. **Cross-command consistency**

   * `audit`, `audit-ledger`, and accounting-related reporting use the same authoritative projection semantics
   * no silent divergence in included event families

3. **Repair idempotency**

   * repeated repair-related logic does not double-apply deltas

4. **Deterministic rebuild / replay**

   * same DB state replayed twice yields identical projection
   * if you add projection checksum/hash support, test it

5. **Fee lifecycle edge case**

   * incomplete fee first, later confirmed/applyable
   * system converges deterministically
   * unresolved fee state is visible

If there is no existing suitable fixture DB, create a minimal sanitized fixture or builder that reproduces the problem.

## Design preference order

When choosing among fixes, use this priority order:

1. **Unify the accounting source of truth**
2. **Make repair events first-class accounting inputs**
3. **Make reporting/audit consume the same projection**
4. **Preserve backward compatibility where practical**
5. **Do not do a broad schema rewrite unless needed for safety/correctness**

* authoritative vs diagnostic state separation
* de-authoritize `trades.cash_after` / `asset_after` where appropriate
