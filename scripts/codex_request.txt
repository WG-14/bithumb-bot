Your task is to perform the next hardening pass on the research/runtime SMA decision-equivalence system. The repository has improved: canonical decision validation, report-level binding, actual runtime order-rule hashing, research decision recording, and repo-owned export/replay commands appear to exist. However, the current implementation may still fail to form a promotion-grade closed loop:

```text
repo-owned research export
  -> repo-owned runtime replay
  -> decision-equivalence
  -> promotion evidence
```

The core risk remains: strategy semantics may still be distributed across research backtest logic, runtime strategy logic, canonical decision adapters, decision-equivalence comparison, and profile/evidence validation. If research export and runtime replay are not bound to the same approved profile, if runtime replay does not use the actual approved regime policy, if decision-equivalence accepts manually prepared canonical-looking JSON without validating export provenance, or if no positive golden equivalence test proves an aligned case can pass, then the repo can still fail to prove that “the strategy validated in research” is the same strategy replayed in runtime.

Do not turn this into general advice. Inspect, patch, and test the actual repository.

## Core objective

Make the repo-owned decision-equivalence loop actually usable as promotion-grade evidence, not merely fail-closed diagnostics.

A successful patch should make this true:

```bash
bithumb-bot research-export-decisions \
  --manifest <manifest.json> \
  --candidate-id <candidate_id> \
  --split validation \
  --profile <approved_profile.json> \
  --out <research_decisions.json>

bithumb-bot runtime-replay-decisions \
  --profile <approved_profile.json> \
  --db <paper_or_runtime.sqlite> \
  --through-ts-list <timestamps.json> \

bithumb-bot decision-equivalence \
  --research-decisions <research_decisions.json> \
  --runtime-decisions <runtime_decisions.json> \
  --profile-hash <same approved profile hash> \
  --market <market> \
  --interval <interval> \
  --data-fingerprint <dataset_or_db_hash>
```

The loop must either produce a promotion-grade passing report for explicitly supported aligned cases, or fail closed with precise, actionable reason codes when the current research model cannot represent the runtime state.
