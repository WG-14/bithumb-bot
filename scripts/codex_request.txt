## Required design changes

Do not merely set `broker_qty_known=True`.

When `accounts_v1_rest_snapshot` is available and valid, construct formal evidence equivalent to:

```text

```

### 2. Keep `open_position_lots` as projection, not canonical truth

Do not make `open_position_lots` authoritative.

The intended model is:

```text
canonical truth:
  exchange observations
  fills
  trades
  fee finalization/allocation events
  fee repair events
  external adjustments
  broker account snapshots

projection:
  open_position_lots
  trade_lifecycles
```

The safe full projection rebuild condition should be:

```text
canonical replay qty == portfolio qty == broker qty
fresh broker evidence present
open orders = 0
unresolved fee pending = 0
projection drift exists
→ rebuild projection safely
```

For the EC2 case, after evidence propagation is fixed, expected preview should become:

```text
needs_rebuild=1
safe_to_apply=1
repair_mode=full_projection_rebuild
broker_qty_known=1
broker_qty=0.000799820000
broker_portfolio_converged=1
accounting_projection_ok=1
unresolved_fee_pending=0
remote_open_order_count=0
recommended_command=uv run python bot.py rebuild-position-authority --full-projection-rebuild --apply --yes
```

### 4. Add safe multi-fill order-level paid_fee allocation

This is still P0 because otherwise the next live multi-fill order can recreate fee_pending and manual recovery.

Implement a safe deterministic authority path:

```text
per-fill fee present
→ trade-level fee authority
→ finalize

single-fill + order-level paid_fee present
→ exchange_order_paid_fee_single_fill
→ finalize

multi-fill + order-level paid_fee present + complete coherent fill set
→ exchange_order_paid_fee_allocated
→ finalize

incomplete or incoherent evidence
→ fee_pending
```

Do not use configured fee estimate as settlement authority. Configured fee may be used only as a diagnostic/coherence bound.

Required safe allocation gates:

```text
order-level paid_fee present and parseable
order terminal/filled
all fills known
fill IDs stable
side/market/order identifiers match
sum(fill funds) matches order executed_funds within tolerance
sum(allocated fees) equals order paid_fee after KRW rounding
allocated fee rate is coherent with configured/observed fee bounds
no duplicate fill application
```

Allocation should be deterministic and auditable. Prefer proportional allocation by `funds`/notional, rounded to KRW fee precision, with rounding residue assigned deterministically, for example to the largest-funds fill or stable sorted last fill. The sum of allocated fees must exactly equal order-level `paid_fee`.

Incident fixture:

```text
order_paid_fee=27.83
order_executed_funds=69595.47103

fill C0101000000983890060:
  price=115994000
  qty=0.00036902
  funds=42804.10588
  expected_allocated_fee=17.12

fill C0101000000983890061:
  price=115995000
  qty=0.00023097
  funds=26791.36515
  expected_allocated_fee=10.71
```

Expected:

```text
fee_status=validated_order_level_paid_fee_allocated
fee_authority=exchange_order_paid_fee_allocated
fee_source=order_level_paid_fee
fee_confidence=validated
accounting_decision=finalize
accounting_status=accounting_complete
no fee_pending
no unapplied_principal
```

### 5. Preserve fail-closed behavior for bad fee evidence

Continue to produce `fee_pending` when:

```text

identifier mismatch
duplicate fill
order not terminal

```

Diagnostic warnings are allowed for minor configured-rate mismatch if exchange evidence is coherent, but settlement authority must come from exchange evidence, not configured estimate.
