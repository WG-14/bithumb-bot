## Core problem

The repository currently separates `paper` and `live`, but it does not cleanly separate these execution intents:

```text
diagnostic-only
live-simulated-run
live-real-order
recovery-only
```

Current behavior appears to be:

```text
paper:
  strategy and DB experimentation possible
  but live order rules / live readiness / target-delta live semantics are not exercised

live + dry-run:
  health / ops / reporting diagnostics possible
  but run loop is blocked

live + real-order armed:
  run loop possible
  but real order risk exists
```

The missing layer is:

```text
live-simulated-run:
  use live market, live order rules, live readiness, target-delta, performance gate, and decision persistence
  but hard-disable all broker order submission
```

## Required design direction

Do not weaken `run` as a real-order command.

Add an explicit live dry-run execution surface. Prefer a separate command:

```bash
uv run bithumb-bot live-dry-run --short 7 --long 30
```

Alternative acceptable design only if cleaner in the current CLI:

```bash
uv run bithumb-bot run-live-dry --short 7 --long 30
```

Avoid making `run` behavior ambiguous. If you add a flag instead, it must be explicit and impossible to confuse with real-order arming, for example:

```bash
uv run bithumb-bot run --live-dry-run --short 7 --long 30
```

but a separate command is preferred.

## Required behavior for `live-dry-run`

The command must only run when:

```text
MODE=live
LIVE_DRY_RUN=true
LIVE_REAL_ORDER_ARMED=false
```

It must fail closed if:

```text
LIVE_REAL_ORDER_ARMED=true
LIVE_DRY_RUN=false
MODE is not live
DB_PATH is unsafe or missing
live runtime paths are inside the repository
startup/recovery safety gate is blocking
```

It must perform enough live-environment checks to be useful:

```text
market validation
order-rule snapshot / chance contract check
accounts preflight using dry-run/unarmed base-missing policy
runtime readiness snapshot
target_position_state resolution
strategy decision generation
target-delta submit plan generation
strategy performance gate evaluation
decision persistence
decision telemetry visibility
```

It must never submit live orders:

```text
No broker order submit
No order record that implies real submission
No fill application
No exchange order creation
No accidental submit through target_delta
No accidental submit through residual closeout
No accidental submit through strategy_position
```

If the code path needs an execution service, create an explicit no-submit execution service for this command rather than relying on `LIVE_DRY_RUN=true` deep inside broker code.

## Persistence requirements

`live-dry-run` should create enough records to validate decision flow:

```text

target_delta plan fields must be present when execution engine is target_delta
performance gate fields must be present
```

Do not create misleading real-order lifecycle artifacts:

```text
Do not create orders rows with PENDING_SUBMIT / NEW / FILLED
Do not create fills
Do not create trades
Do not mutate portfolio as if an order executed
```

If an existing paper/live dry-run path already creates order-like diagnostics, keep it clearly marked as diagnostic and ensure ops reports cannot confuse it with actual order submission.
