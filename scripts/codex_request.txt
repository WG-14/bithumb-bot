Treat this as an engineering task to make the live trading bot more robust, reproducible, and operable. Do not reduce this to a narrow symptom patch. Context:

The EC2 live bot has repeatedly stopped after exactly one BUY or SELL execution. The operator has had to manually resume the bot and sometimes manually repair the DB. The current observed pattern is:

* A live order is submitted.
* Broker/exchange fill is observed.
* Fee/accounting information is incomplete or delayed.
* The bot promotes this to `RECOVERY_REQUIRED`.
* Later, fee/accounting repair succeeds.
* Audit/recovery reports show no active blocking issue.
* Yet runtime remains paused or requires explicit `resume`.
* Recent DB rows showed terminal orders such as `FILLED` and `FAILED` while `orders.local_intent_state` remained `PENDING_SUBMIT`.

Known suspicious findings to verify, not blindly assume:

1. `live_submission_execution.py` appears to treat any fill whose accounting status is `fee_pending` as an application failure and transitions the order to `RECOVERY_REQUIRED`, with a reason like “broker fill observed but accounting is fee-pending.”
2. `oms.py` appears to include `RECOVERY_REQUIRED` in both open/risky order status sets and terminal status sets, which may give different components inconsistent semantics.
3. `fee_pending_repair.py` already contains logic to apply fee-pending repairs and transition projected order status, but this appears to be operator-command-oriented rather than a first-class idempotent reconciler in the live run loop.
4. Runtime readiness often knows the system is safe to resume, for example with no active blockers, no unresolved fee state, converged broker/portfolio/lot projection, and `safe_to_resume=1`, but the live runtime still requires an explicit operator resume.
5. Terminal orders can retain `local_intent_state=PENDING_SUBMIT`, which suggests the order state model is not being updated atomically or consistently.

Primary objective:

Redesign and implement the smallest coherent change set that prevents this entire class of failures: recoverable exchange/accounting latency must not be escalated into a manual halt/resume cycle unless there is a real unresolved risk or operator-review condition.

Do not merely add a workaround that auto-runs `resume` after the old broken flow. Prefer a state-machine and reconciler design that makes the system correct before it reaches halt/resume.

Working hypotheses to validate or falsify:

Hypothesis 1: Fee-pending is promoted to a hard recovery state too early.

Bithumb fill data may be eventually consistent. The live submission path currently appears to require immediate authoritative fee/accounting data. If a fill is observed but fee is missing/pending, it records a broker fill observation and transitions to `RECOVERY_REQUIRED`. That may be structurally wrong. A fee-pending fill should usually become an accounting-pending state with automatic retry/reconciliation, not an operator recovery incident.

* Search for `fee_pending_fills`, `live_application_fee_pending`, `manual fee resolution required before ledger apply`, `LiveFillFeeValidationError`, and `_mark_recovery_required`.
* Identify every code path that converts fee-missing/fee-pending into `RECOVERY_REQUIRED`, halt, pause, or resume gate.
* Determine whether the parser is actually failing to read available fee data, or whether the exchange genuinely does not provide authoritative fee immediately.
* Inspect `broker_fill_observations.raw_payload`, `fee_status`, `fee_source`, `fee_confidence`, and `fee_validation_reason` handling.

Hypothesis 2: The order state model is not a single coherent state machine.

`orders.status`, `orders.local_intent_state`, `order_intent_dedup.order_status`, `order_events.order_status`, runtime halt state, and recovery counters may be updated independently. This allows contradictory states such as `orders.status='FILLED'` with `local_intent_state='PENDING_SUBMIT'`.

Required tests:

* A terminal `FAILED`, `REJECTED`, or `CANCELED` order must not retain `local_intent_state=PENDING_SUBMIT`.
* `order_intent_dedup.order_status` must not disagree with `orders.status` for active dedup rows unless explicitly documented.

Hypothesis 3: `RECOVERY_REQUIRED` has overloaded semantics.

`RECOVERY_REQUIRED` appears to mean too many things at once: unresolved risky order, terminal historical recovery marker, operator-review-required state, and auto-repairable accounting pending state. If it is both “open” and “terminal,” startup gate and resume logic can disagree.

Validation:

* Search every use of `RECOVERY_REQUIRED` in `src`, `tests`, and `docs`.
* Classify each use into:

  * submit outcome
  * accounting pending/recovery
  * operator halt blocker
  * terminal historical marker
  * risk gate condition
  * report/display state
* Decide whether `RECOVERY_REQUIRED` must be split into more precise states, or whether its membership in open/terminal sets must be corrected.

Design preference:

Do not let a single string represent both an unresolved live risk and a completed historical repair marker. Introduce clearer distinctions if needed, such as:

* `ACCOUNTING_PENDING`
* `FEE_PENDING`
* `AUTO_RECOVERING`
* `OPERATOR_REVIEW_REQUIRED`
* `RECOVERY_REPAIRED`
* `HARD_HALTED`

The exact names may differ, but the semantics must be clear and tested.

Hypothesis 4: Repair succeeds but runtime policy refuses automatic recovery.

Design requirements:

1. Separate state axes instead of overloading one order status.

At minimum, distinguish:

* Submission state: intent created, submitting, submitted, acknowledged, rejected, submit unknown.
* Exchange execution state: open, partial, filled, canceled.
* Accounting state: not applied, fee pending, applied, repaired, invalid.
* Operator/risk state: ok, degraded, auto-recovering, operator-review-required, hard-halted.

This does not necessarily require a full DB migration in one PR, but the code paths touched by this incident must move toward these semantics and must not deepen the overload of `RECOVERY_REQUIRED`.

2. Treat fee-pending as asynchronous accounting, not immediate manual recovery.

Preferred flow:

* Broker fill observed with missing/pending fee.
* Record `broker_fill_observations` with `accounting_status='fee_pending'`.
* Mark order/accounting as fee/accounting pending, not hard recovery, unless there is another real risk.
* Retry broker order/fill reads for a bounded time/count.
* If authoritative fee becomes available, apply fill and trade idempotently.
* If only order-level fee is available and attribution is unambiguous, apply automatically with clear provenance.
* If account cash drift can infer fee and there is a single candidate, allow automatic repair with explicit evidence.
* If attribution is ambiguous, fee is invalid, or drift is large/unexplained, transition to `OPERATOR_REVIEW_REQUIRED`, not a generic overloaded recovery state.

3. Promote fee-pending repair into an idempotent reconciler.

Do not leave `fee_pending_repair.py` as only a manual CLI path. Extract or reuse its core logic in a reconciler that the live run loop/reconcile flow can call safely and repeatedly.

Suggested shape:

```python
class AccountingReconciler:
    def reconcile_pending_fees(self, conn, broker) -> ReconcileResult:
        ...
```

The exact API may differ, but it must be idempotent, testable, and observable.

4. Make runtime halt policy severity-based.

Use explicit severity levels or equivalent semantics:

* OK: continue normally.
* DEGRADED: continue but report warnings.
* AUTO_RECOVERING: retry/reconcile; optionally block new submissions if necessary.
* OPERATOR_REVIEW_REQUIRED: block new submissions, keep process observable, require operator action.
* HARD_HALTED: stop trading due to real risk, kill switch, loss limit, irreconcilable broker conflict, or unsafe unknown order.

Recoverable fee/accounting latency must not be treated as hard halt.

5. Preserve safety.

Do not allow duplicate orders, overfills, double application of fills, or silent fee omission. Idempotency is mandatory.

Must-have invariants:

* A fill is applied at most once.
* A fee repair is applied at most once per fill identity.
* Terminal order states cannot retain submit-pending local intent.
* Auto-recovery cannot place a new order while there is an unresolved live order that could still execute.
* Operator-review-required states must block new submissions.
* Hard risk states must remain hard blocked.
* Manual DB edits must not be part of normal recovery.
