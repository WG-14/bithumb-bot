# Part 3: Remove context list materialization in replay hot path

Change `replay_sma_strategy_decisions_from_candles()` so it no longer does this in the normal path:

```python
contexts = []
for candle:
    contexts.append(_replay_decision_context(...))

attributions = [normalize_decision_attribution_from_context(context) for context in contexts]
summary = summarize_decision_attributions(attributions)
```

Preferred structure:

```python
acc = DecisionAttributionAccumulator()
decision_count = 0

for candle:
    attribution = _replay_decision_attribution(...)
    acc.add(attribution)
    decision_count += 1

summary = acc.summary()
```

If you do not introduce the accumulator in replay directly, at least avoid the `contexts` list and build attributions one by one. But the target is streaming accumulation.

Keep the existing `StrategyReplayResult` shape unless you have a strong reason to change it.

# Part 5: Add max-operations / allow-large-sweep guard

`max_candles` prevents full-history DB scans, but it does not prevent large bounded grids.

Add CLI and plan support for operation budget.

## Update StrategySweepExecutionPlan

In `src/bithumb_bot/strategy_sweep.py`, extend `StrategySweepExecutionPlan` with:

```python
max_operations: int | None
allow_large_sweep: bool
```

Update `build_strategy_sweep_execution_plan()` to accept:

```python
max_operations: int | None
allow_large_sweep: bool
```

Guard behavior:

```text
if estimated_operations > max_operations and not allow_large_sweep:
    block
```

For live mode, this should be enforced by default.

Recommended default:

```text
max_operations = 300000
```

You may apply the default only in CLI, not necessarily in the library function, if that fits the repo style better.

Block reason example:

```text
estimated_operations_exceeds_max_operations
```

## CLI options

In `app.py`, add:

```text
--max-operations
--allow-large-sweep
```

Expected behavior:

* In live mode, if estimated operations exceed max operations and `--allow-large-sweep` is not provided, fail before replay execution.
* `--allow-large-sweep` permits execution and should be reflected in the plan.
* JSON output should include `max_operations` and `allow_large_sweep`.
* Human output should show estimated operations and max operations.

Do not confuse this with `--allow-full-history`; they solve different problems:

```text
--allow-full-history
  permits unbounded candle history

--allow-large-sweep
  permits high grid_count × candle_count operation count
```
