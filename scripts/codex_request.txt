You are working in the `WG-14/bithumb-bot` repository.

Your task is not to add another short-term residual/dust workaround. The priority is to verify and implement a structural simplification of the execution model.

Core problem hypothesis to verify, not assume blindly:

The current bot has grown complex because execution is still centered on lot-native authority, dust tracking, residual inventory policy, and recovery-specific branches. Entry and exit decisions are partially separated, but the repository does not have one clear execution-level definition of “current position.” As a result, real exchange leftovers after BUY/SELL cycles can become neither a normal strategy lot nor true dust. This creates states where BUY is blocked because BTC remains, normal SELL is blocked because there is no sellable strategy lot, and residual SELL requires separate policy gates. This is likely why dust/residual policy keeps expanding.

The desired long-term behavior is:

```text
The bot may buy once and sell once.
The exchange may leave residual BTC due to fills, fees, rounding, min quantity, or min notional constraints.
If the bot is stopped and restarted, it should read the current broker balance, validate safety conditions, compare current position against target position, and converge safely.
Residual BTC should not break the bot’s state machine.
Only below-exchange-minimum deltas should be treated as true dust.
Executable residual BTC should be treated as current position, not as a separate exceptional class.
```

Do not dilute this into a general refactor. Work directly in this repo and produce concrete code changes, tests, and a final report.

Keep the hypothesis falsifiable. If any of the above is wrong, explain exactly what the code actually does and adjust the design accordingly.

## Design objective

Promote target-position / delta execution from shadow telemetry toward a primary execution model behind a feature flag.

The new model should be:

```text
Signal Layer:
  Strategy emits BUY / SELL / HOLD.

Target Policy Layer:
  BUY  -> target_exposure_krw = TARGET_EXPOSURE_KRW if set, else MAX_ORDER_KRW.
  SELL -> target_exposure_krw = 0.
  HOLD -> maintain previous persisted target_exposure_krw.

Position Truth Layer:
  Primary execution current position = broker-verified current BTC quantity.
  This must be gated by existing safety checks.
  Do not blindly trust broker balance if readiness/recovery state is unsafe.

Delta Execution Layer:
  target_qty = target_exposure_krw / reference_price
  delta_qty = target_qty - current_broker_qty

Order Rule Layer:
  If abs(delta_qty) and abs(delta_notional) satisfy exchange min_qty and min_notional:
    submit BUY or SELL for the delta.
  Else:
    classify as true_dust / below exchange minimum and do not submit.

Accounting / Audit Layer:
  Existing lot-native authority, fills, fees, PnL, and recovery diagnostics remain important.
  But in target_delta execution mode, lot authority must not be the primary SELL sizing authority.
```

This is the architectural priority:

```text
current_position + target_position + delta
```

not:

```text
strategy lot + residual inventory + dust tracking + special close candidate
```

Add or use a setting such as:

Use existing settings if they already exist; avoid duplicate names.

Do not add more residual/dust policy branches unless absolutely necessary for migration compatibility. Prefer deleting or bypassing residual-specific logic in `target_delta` mode.

Do not remove existing residual behavior for `lot_native` mode in this task unless it is clearly dead code and tests prove it.

## Required implementation

### 1. Persist target position state

Add durable target state storage.

Minimum fields:

```text
pair
target_exposure_krw
target_qty
last_signal
last_decision_id
last_reference_price
updated_ts
```

Use the repo’s existing DB/migration style.

Rules:

```text
BUY:
  persist target_exposure_krw = TARGET_EXPOSURE_KRW if set, else MAX_ORDER_KRW

SELL:
  persist target_exposure_krw = 0

HOLD:
  preserve previous target_exposure_krw
```

If HOLD occurs and no previous target exists, fail closed in `target_delta` mode with a clear block reason such as:

```text
missing_persistent_target_state
```

Do not silently assume flat unless existing startup/recovery policy explicitly proves that flat target is safe.

### 3. Define position truth clearly

In `target_delta` mode:

```text
current position = broker_qty
```

But only when safety gates pass.

Block execution if any of these are unsafe:


Use existing readiness payload fields where possible. Do not invent an unrelated safety system.

If broker/local convergence can be checked in more than one way, centralize the decision and log which evidence source was used.

### 4. Implement target_delta execution path

When:

```text
EXECUTION_ENGINE=target_delta
```

the live execution summary and submission path should use target delta, not residual candidate logic.

Expected behavior:

```text
raw_signal=BUY:
  target_exposure_krw > 0
  compute delta vs current broker_qty
  submit BUY only for positive executable delta

raw_signal=SELL:
  target_exposure_krw = 0
  compute delta vs current broker_qty
  submit SELL for executable current broker quantity

raw_signal=HOLD:
  maintain previous target
  submit only if current broker_qty has drifted enough from target to produce executable delta
  otherwise no-op
```

If delta is below exchange minimum:

```text
submit_expected=false
target_dust_classification=true_dust
target_block_reason=delta_below_exchange_min
```

If delta is executable:

```text
submit_expected=true
target_dust_classification=executable_delta
target_block_reason=none
```

Do not call this residual. Do not require `RESIDUAL_LIVE_SELL_MODE=enabled` for target-delta SELL. That mode belongs to the old residual path.

### 5. Keep lot-native accounting, but demote it from execution authority in target_delta mode

Do not delete accounting.

In `target_delta` mode:

* lot-native records may still be used for:

  * audit
  * PnL
  * lifecycle diagnostics
  * recovery reports
  * reconciliation
* but SELL sizing should not require:

  * `sellable_executable_lot_count > 0`
  * strategy lot authority
  * residual sell candidate

If a broker balance exists and safety gates pass, the target delta determines the order.

### 6. Simplify dust semantics

In `target_delta` mode, dust should mean only:

```text
abs(target_qty - current_qty) is below exchange min_qty or min_notional
```

Executable leftover BTC is not dust. It is current position.

Update naming, comments, logs, and tests accordingly.

Keep existing dust classifications for old paths if necessary, but avoid extending them as the new execution model.

## Acceptance criteria

The work is acceptable only if all of these are true:

1. `lot_native` mode still works as before.
2. `target_delta` mode persists target state.
3. `target_delta` mode uses broker-verified current position as primary execution truth after readiness gates pass.
4. `target_delta` mode computes orders from `target_qty - current_qty`.
5. Executable leftover BTC after restart is treated as current position and can be sold when target is zero.
6. Below-minimum leftover BTC is treated as true dust and does not corrupt the state machine.
7. SELL in `target_delta` mode does not require strategy lot authority.
8. Residual policy settings are not required for target_delta SELL.
9. Unsafe broker/local/recovery states fail closed with clear block reasons.
10. Logs and decision context clearly show target/current/delta reasoning.
