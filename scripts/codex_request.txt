# Primary objective

Make machine-readable CLI modes actually machine-readable.

The rule is:

```text
When a command is invoked with --json, stdout must contain only the JSON payload.
Logs, diagnostics, live execution contract messages, warnings, and human-readable status lines must go to stderr or a log file.
```

This patch should prioritize channel separation, not more strategy logic.

---

# Required design principle

Adopt this repo-level CLI output contract:

```text
stdout:
  command result data

stderr:
  logs, diagnostics, warnings, live execution contract, progress/status messages

--json stdout:
  valid JSON only
```

This is not cosmetic. It is required so operators can safely do:

```bash
uv run bithumb-bot strategy-sweep ... --json > result.json
python3 -c 'import json; json.load(open("result.json"))'
```

# Part 3: Use a general solution, not a one-off sweep hack

Do not patch only `cmd_strategy_sweep()` if the root cause is logging configuration or global live contract logging.

Prefer one of these approaches, depending on repo structure:

## Option A: Logging goes to stderr globally

Configure Python logging so command diagnostics go to stderr, not stdout. This is the cleanest if compatible with existing tests.

## Option B: Machine-readable mode redirects diagnostics to stderr

Add a CLI-level mechanism such as:

```python
machine_readable_output = command_has_json_flag(args)
```

and ensure live contract diagnostics/logging use stderr for those commands.

## Option C: Command-specific safe wrapper

If global logging changes would break many tests, add a minimal wrapper around machine-readable commands, but document it as the CLI JSON contract.

Do not silently drop logs. Do not mix logs into JSON stdout.
