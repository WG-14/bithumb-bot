You are working in GitHub repository `WG-14/bithumb-bot`.

This is a production-safety and experiment-system hardening task. Do not treat this as a simple `broker_qty_unknown = 0` symptom patch. The goal is to make `live-dry-run` a reliable live-environment decision experiment mode by adding a correct evidence pipeline from `/v1/accounts` into runtime readiness / canonical position / target-delta decision context, while preserving all no-submit and real-order safety guarantees.

Use the current EC2 evidence as context, but keep the root causes falsifiable.

## Core problem

The latest repository already has a useful `live-dry-run` command and no-submit safety surface. Do not redo that work.

The current gap is:

```text
live-dry-run command
→ live order-rule / market / accounts preflight runs
→ accounts preflight allows missing BTC row as dry-run flat start
→ target-delta decision does not receive broker position evidence
→ target-delta blocks with broker_qty_unknown
→ live-dry-run cannot validate target-delta plan or performance-gate behavior
```

This is an evidence propagation problem, not an order submission problem.

## Required design direction

Do not patch this by blindly defaulting missing broker qty to zero.

Implement or strengthen a formal evidence pipeline:

```text
/v1/accounts payload
→ parsed account/balance evidence
→ live-dry-run broker position evidence
→ runtime readiness / canonical position input
→ target-delta decision input
→ strategy_decisions.context_json
→ decision-telemetry / ops-report
```

The code must preserve fail-closed behavior when evidence is not trustworthy.

## Concrete implementation requirements

### 1. Create a structured account evidence object

Add a small dataclass or equivalent structure, for example:

```python
@dataclass(frozen=True)
class LiveAccountEvidence:
    source: str
    observed_ts_ms: int
    execution_mode: str
    quote_currency: str
    base_currency: str
    quote_qty_known: bool
    quote_available: float
    quote_locked: float
    base_qty_known: bool
    base_available: float
    base_locked: float
    base_currency_missing_policy: str
    preflight_outcome: str
    flat_start_allowed: bool
    flat_start_reason: str
    evidence_policy: str
    evidence_reason: str
```

Use existing account parsing code where possible. Do not duplicate fragile parsing logic if `broker.accounts_v1` already has it.

For a valid live-dry-run accounts response with KRW row present and BTC row missing:

```text
base_qty_known=true
base_available=0.0
base_locked=0.0
base_currency_missing_policy=allow_zero_position_start_in_dry_run
preflight_outcome=pass_no_position_allowed
evidence_policy=missing_base_row_known_zero_for_live_dry_run
```

For transport/auth/schema failures:

```text
base_qty_known=false
preflight_outcome=fail_*
```

For real-order unsafe missing base row:

```text
base_qty_known=false
preflight_outcome=blocked_or_unknown
```

### 2. Do not mutate real broker state or portfolio from live-dry-run evidence

`live-dry-run` evidence is input to a simulated decision. It must not create orders, fills, trades, or portfolio mutations.

Allowed:

```text
strategy_decisions row
decision context
diagnostic runtime metadata if already intended for diagnostics
```

Not allowed:

```text
orders rows with PENDING_SUBMIT / NEW / FILLED
fills rows
trades rows
portfolio mutation as if execution occurred
exchange order submission
```

### 3. Feed evidence into target-delta decision

Modify the live-dry-run flow so target-delta receives known broker qty evidence.

Patch design can be either:

```text
A. Pass LiveAccountEvidence directly into build_execution_decision_summary / target-delta decision builder.
```

or:

```text
B. Convert LiveAccountEvidence into the existing canonical position / readiness structure before calling build_execution_decision_summary.
```

Prefer the least invasive design that preserves a single evidence contract.

Required result after patch:

```text
live-dry-run latest decision context has:
  broker_qty_known=1
  broker_qty=0.0
  balance_source=accounts_v1_rest_snapshot or equivalent
  balance_source_preflight_outcome=pass_no_position_allowed
  base_currency_missing_policy=allow_zero_position_start_in_dry_run
```

and no longer blocks with `broker_qty_unknown` under valid known-zero conditions.

### 4. Keep target-delta fail-closed for unsafe cases
