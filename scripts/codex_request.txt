Your task is to investigate and fix a recurring live-trading accounting failure. Do not treat the current diagnosis as unquestionable fact. ## Incident context

On EC2, the bot repeatedly entered this failure cycle after live BUY/SELL fills:

```text
Bithumb order fills successfully
→ local fee validation classifies the fill as fee_pending
→ local fills/trades/portfolio/lot accounting becomes incomplete or inconsistent
→ broker/local portfolio mismatch appears
→ startup/resume gates block live
→ recovery suggests flatten or manual repair
→ flatten itself creates another fill and repeats the same accounting failure
→ operator manually repairs fee/principal/lot state and sometimes bot_health
```

Recent runtime evidence showed:

```text
Actual Bithumb app fee rate: 0.04% = 0.0004 = ~4 bps

DB-observed examples:
flatten SELL fee_bps = 3.999784
latest BUY fee_bps   = 3.999902
previous SELL        = 3.998178
previous BUY         = 3.999148

Resolved setting:
LIVE_FEE_RATE_ESTIMATE = 0.0025
LIVE_FILL_FEE_ALERT_MIN_NOTIONAL_KRW = 10000.0
```

The direct trigger appears to be:

```text
configured expected live fee rate = 0.0025
actual Bithumb paid fee rate      = 0.0004
→ expected_fee_rate_mismatch
→ order_level_candidate
→ fee_pending
→ accounting/recovery cascade
```

However, do not stop at “set LIVE_FEE_RATE_ESTIMATE to 0.0004.” That is only an operational mitigation. The more important structural concern is:

```text
The repository currently appears to let a local estimated fee rate override exchange-provided paid_fee evidence as an accounting gate. If a single-fill order has exchange paid_fee, matching executed_volume/funds, and matching identifiers, the fill should generally be accounting-complete; expected fee rate mismatch should be a warning/diagnostic, not a reason to reject the exchange ledger data and destabilize accounting.
```

## Working hypotheses to verify or falsify

Keep these as hypotheses until tested.

### Hypothesis A — direct trigger: live fee-rate configuration drift

`LIVE_FEE_RATE_ESTIMATE` was unset in the live env and silently defaulted to `0.0025`, while actual Bithumb paid fees were ~`0.0004`. This caused valid exchange paid_fee values to fail `expected_fee_rate_match`.

Verify by inspecting `config.py`, tests, `.env.example`, and any startup/preflight behavior. Confirm whether `LIVE_FEE_RATE_ESTIMATE` is allowed to silently default in live real-order mode and whether that default is actually used for live fee accounting validation.

Do not merely change the default unless that is justified by tests. The deeper question is whether a fallback estimate should be allowed to block exchange paid_fee accounting at all.

### Hypothesis B — structural root cause: fee authority is incorrectly ordered

The fee authority hierarchy may be wrong. Current behavior appears to make local estimated fee rate a hard accounting gate for exchange order-level `paid_fee`.

Expected design direction:

```text
Authoritative or near-authoritative fee evidence:
- trade-level exchange fee
- order-level exchange paid_fee when single-fill evidence is present and executed_volume/funds/identifiers match
- operator_confirmed fee

Diagnostic or estimate evidence:
- LIVE_FEE_RATE_ESTIMATE
- FEE_RATE
- strategy/pretrade fee estimates
```

A local estimate should be useful for pretrade sizing, sanity warnings, and anomaly detection. It should not normally invalidate exchange-provided paid_fee for a single-fill order whose quantity, funds, and identifiers match.

Verify whether the current code violates this by requiring `expected_fee_rate_match` for `validated_order_level_paid_fee`.

### Hypothesis C — state coupling amplifies the fee issue

The system may couple fee finalization, principal application, portfolio projection, open-position lots, and runtime resume gates too tightly. A valid fill misclassified as `fee_pending` then cascades into `RISK_STATE_MISMATCH`, `HALT_STATE_UNRESOLVED`, `POSITION_AUTHORITY_PROJECTION_CONVERGENCE_REQUIRED`, and manual DB repair.

Verify in `live_submission_execution.py`, `execution.py`, `runtime_readiness.py`, `recovery.py`, and projection/lot tests whether pending fee is allowed to poison portfolio/lot truth or whether principal is applied idempotently and separately from fee finalization.

### Hypothesis D — live operability guardrails are insufficient

The system allowed live operation with a resolved fee estimate far from recent observed broker fee rates, and recovery suggestions steered the operator toward flattening before forensic diagnosis. This may indicate missing live preflight invariants and missing observability.

Verify whether health/recovery/startup outputs expose:

```text
configured fee rate
recent observed fee_bps median
fee-rate deviation
fee authority mode
new fee_pending count since last start
principal_applied_fee_pending count
repair counts
position_authority repair counts
whether flatten is suppressed when accounting root cause is unresolved
```

## Required design direction

Prioritize design changes that prevent the whole class of failures, not just this exact fee rate mismatch.

### 1. Separate fee authority, accounting decision, and diagnostics

Refactor or adjust the model so these are explicit concepts:

```text
Fee authority:
- exchange_trade_fee
- exchange_order_paid_fee_single_fill
- operator_confirmed
- configured_estimate
- unknown/missing

Accounting decision:
- authoritative_fee: safe to finalize accounting with this fee
- estimated_fee: not valid for live final accounting; allowed only for pretrade/diagnostics
- missing_fee: principal-only or recovery path, depending on policy

Diagnostics:
- expected_fee_rate_mismatch
- fee_ratio_outlier
- zero_fee_material
- missing_fee
- multi_fill_order_level_fee_ambiguous
```

The concrete expected behavioral change:

```text
For a single-fill Bithumb order:
- paid_fee is present and non-negative
- not a material zero-fee case
- executed_volume matches fill qty
- executed_funds matches fill funds
- client_order_id/exchange_order_id/fill_id evidence is present

Then order-level paid_fee should be accepted as accounting-complete even if expected_fee_rate_match is false.

The expected_fee_rate_match=false result should remain visible as a warning/diagnostic/anomaly, but it should not by itself force fee_pending.
```

Preserve fail-closed behavior for truly unsafe cases:

```text
missing/unparseable paid_fee
negative fee
material zero fee
multi-fill order-level fee ambiguity
executed_volume mismatch
executed_funds mismatch
identifier mismatch
```

### 2. Make live fee configuration drift observable and gated

Add or improve preflight/readiness diagnostics so live real-order mode refuses or loudly warns when configured estimates diverge from recent observed exchange-paid fee rates.

Minimum acceptable behavior:

```text
health/recovery-report/restart-checklist should display:
- LIVE_FEE_RATE_ESTIMATE
- observed_fee_bps_median over recent broker_fill_observations with non-null fee and material notional
- deviation between configured estimate and observed median
- count of recent expected_fee_rate_mismatch observations
- count of fee_pending observations
- count of fee_pending_accounting_repairs
```

For live real-order startup, add a guard or warning policy. Prefer a hard block only if it does not make recovery impossible; otherwise make it an explicit warning with a clear operator action. If the configured fee estimate is used only for diagnostics after the fee-authority fix, this drift should no longer block accounting, but it should still be visible.

### 3. Prevent recovery from recommending flatten before accounting root cause is addressed

If the active blocker is fee/accounting authority drift or unresolved accounting diagnostics, do not suggest flatten as the primary action unless there is actual market exposure that cannot be made safe otherwise.

Recovery output should distinguish:

```text
forensic/accounting root cause mode:
  no additional orders
  collect original broker fill/order evidence
  inspect fee authority and local accounting state
  produce repair plan

market-risk flatten mode:
  only when actual executable exposure is present and accounting evidence is sufficient
```

### 4. Improve repairability without manual DB edits

Do not rely on operators manually editing `bot_health` or multiple tables. If repair is necessary, it should be via auditable, idempotent commands.

Longer-term design target:

```text
bithumb-bot doctor --live --explain
bithumb-bot repair-plan --live
bithumb-bot repair-apply --plan-id ...
```

Each repair plan should show:

```text
preconditions
touched tables
before/after portfolio
before/after lot projection
idempotency key
rollback/backup path
expected audit-ledger result
why this is safe
```

You do not need to fully build the repair-plan engine in this task unless it is already close. But add tests, TODOs, or scoped structural changes that move the repo in that direction rather than adding another opaque manual repair shortcut.

### 5. Treat open_position_lots as a rebuildable projection, not unquestioned truth

Check whether `open_position_lots` is effectively used as source of truth. The desired direction is:

```text
canonical truth = fills + trades + finalized fee/adjustment events + external adjustments
open_position_lots = deterministic, rebuildable materialized view
```

If projection divergence is detected and broker/portfolio evidence is converged, the system should diagnose and rebuild deterministically where safe, not require repeated operator guessing. Do not over-scope this into a full accounting rewrite, but make sure your fee-authority fix does not worsen lot projection drift.
