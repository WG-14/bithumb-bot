You are working in the GitHub repository `WG-14/bithumb-bot`.

Your task is to implement the next structural step after the residual-only holdings patches. Do not treat the current diagnosis as unquestionable fact. Use it as the starting hypothesis, verify or falsify it against the current repo, then implement the smallest coherent design change that moves the bot from “residual diagnosis and recovery blocking” toward “residual-aware position management.”

The key point is this:

```text
The problem is not that residual BTC must be removed.
The deeper problem is that residual BTC is not yet modeled as normal inventory/exposure in BUY/SELL/HOLD policy.
```

The latest patches successfully improved diagnosis and operator safety:

```text
- residual inventory is visible through ResidualInventorySnapshot
- broker/portfolio/projection convergence is reported
- residual-only holdings are no longer misclassified as position-authority repair
- rebuild-position-authority and flatten are no longer recommended
- audit and audit-ledger now agree
- can_resume / run_loop_can_resume / tradeability_gate_blocked are scoped
```

Preserve those improvements.

However, do not continue down a “residual-closeout --apply” path as the primary design. A system that asks the operator to clear residual BTC every time it appears will repeat the same operational failure. Residuals are expected in automated trading because of partial fills, fees, minimum quantities, tick/quantity steps, and split BUY/SELL flows.

The target direction is:

```text
Residual-aware position management:
- If residual inventory is broker/portfolio/projection-converged and explainable, treat it as tracked inventory, not automatically as a live-blocking recovery incident.
- BUY sizing must account for residual inventory as existing exposure.
- SELL policy must be able to consider exchange-sellable residual inventory when the configured residual policy allows it.
- HOLD must keep residual inventory tracked without creating an incident.
- Unexplained, divergent, stale, or unsafe residuals must still fail closed.
```

## Required design changes

### P0.1 Add explicit `residual_inventory_mode`

Add a config setting or policy source, preferably with safe defaults:

```text
residual_inventory_mode=block | track | ignore_tiny
```

Semantics:

```text
block:
  current conservative behavior
  explainable residual-only holdings remain run-loop blocking
  recommended command may remain residual-closeout-plan

track:
  explainable broker-matched residual-only holdings become RESIDUAL_INVENTORY_TRACKED
  run loop may operate if all other recovery/accounting gates pass
  residual inventory counts as exposure
  BUY sizing must account for residual exposure
  SELL may create a residual sell candidate when policy and signal allow
  repair/flatten are not recommended

ignore_tiny:
  only exchange-unsellable tiny dust may be treated as effective-flat/tracked-only
  material exchange-sellable residuals should not be ignored
```

Default may remain `block` if you want maximum rollout safety. But tests must cover `track`.

### P0.2 Add `RESIDUAL_INVENTORY_TRACKED`

Do not replace fail-closed behavior. Split the states:

```text
RESIDUAL_INVENTORY_TRACKED:
  residual is broker/portfolio/projection-converged
  accounting_projection_ok=true
  residual rows are explainable
  broker evidence fresh/known
  no unresolved orders
  no active fee/accounting blocker
  residual_inventory_mode=track

NON_EXECUTABLE_RESIDUAL_HOLDINGS or RESIDUAL_INVENTORY_UNRESOLVED:
  residual exists but is not safe to track as inventory
  broker mismatch
  portfolio mismatch
  projection divergence
  stale/missing broker evidence
  active accounting issue
  unknown/unexplainable source/linkage
  policy mode is block
```

Expected readiness output in track mode:

```text
residual_class=RESIDUAL_INVENTORY_TRACKED
tradeability_reason=RESIDUAL_INVENTORY_TRACKED

tradeability_gate_blocked=false
operator_next_action=run_with_residual_inventory_tracking or equivalent

```

### P0.4 BUY sizing must include residual inventory exposure

When residual inventory is tracked, BUY sizing must treat it as existing exposure.

Minimum expected model:

```text

current_exposure_notional = current_exposure_qty * market_price
available_budget = target_or_allowed_budget - current_exposure_notional
```

If the residual already consumes the budget, BUY should be blocked or reduced.

Add explicit fields in decision context / execution context if needed:

## Safety constraints

Do not:

```text
- flatten residuals as primary action
- manually mutate DB to clear residuals
- infer strategy executable authority from aggregate residual qty
- make dust_tracking rows pretend to be open_exposure lots
- allow residual-aware run loop when broker/portfolio/projection are divergent
- allow residual-aware run loop with stale/missing broker evidence
- allow residual-aware run loop with active fee/accounting/order recovery issues
- allow BUY sizing to ignore residual exposure
- allow SELL residual submission without final pre-submit proof
```

Preserve:

```text
- fail-closed default
- lot-count based strategy SELL authority
- audit/audit-ledger convergence checks
- row-level residual explainability
- repair/flatten not recommended for broker-matched explainable residuals
- residual-closeout-plan read-only diagnostic behavior
```
