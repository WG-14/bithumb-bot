Codex task for `WG-14/bithumb-bot`

You are working directly in the `WG-14/bithumb-bot` repository. Do not turn this into a generic cleanup. Treat this as a live-mode recovery blocker with a narrowly scoped suspected bug in position authority readiness classification.

## Core problem statement

A live KRW-BTC bot instance is blocked from safe resume even though the external account evidence and local accounting evidence converge.

Observed live facts from the operator session:


`position_authority_projection_publications` is empty, so there is no current-state attestation proving the portfolio-anchored projection. `position_authority_repairs` contains historical repair evidence, but the readiness model explicitly treats repair events as historical evidence, not current-state proof.

Current CLI behavior:

And:

```text

```

The full projection gate report shows the safety evidence is otherwise good; the blocker is effectively only `full_projection_rebuild_not_required`.

The suspected bug is that `needs_full_projection_rebuild=false` is being derived even when the projection is clearly non-converged and the only stale state is fragmented historical materialized lots.

## Hypothesis to verify, not assume

The likely faulty logic is in `src/bithumb_bot/position_authority_state.py` around `historical_fragmentation_projection_drift`:

```python
historical_fragmentation_projection_drift = bool(
    not bool(projection_convergence.get("converged"))
    and "historical_fragmentation" in diagnostic_flags
    and lot_row_count > 1
    and projected_total_qty > portfolio_qty + _EPS
    and not projection_repair_covers_excess
    and other_active_qty > _EPS
)
```

In the observed state:

```text
projection_converged = false
lot_row_count = 14
projected_total_qty = 0.001788
portfolio_qty = 0.00099986
other_active_qty = 0.001788
portfolio_projection_publication_present = false
portfolio_projection_repair_event_status = recorded_but_not_current_state_proof
portfolio_projection_repair_recorded = true
projected_qty_excess = 0.00078814
```

But `diagnostic_flags` contains only:

```text
["projection_diverged"]
```

So `historical_fragmentation_projection_drift` remains false, and the safe official rebuild path is blocked as “not required.”

The hypothesis is: this is a classification blind spot. Projection is non-converged due to stale fragmented materialized lots, but the full projection rebuild path is gated on the presence of a pre-existing `historical_fragmentation` diagnostic flag that is not set for this state.

## Desired design behavior

When the state is:

```text
projection_converged = false
projected_total_qty > portfolio_qty
lot_row_count > 1
other_active_qty > 0
projection_repair_covers_excess = false
portfolio_projection_publication_present = false
broker_portfolio_converged = true
accounting_projection_ok = true
no open/unresolved/pending/submit_unknown/recovery_required orders
no unresolved fee pending
```

and the projection divergence is from stale fragmented materialized `open_position_lots`, then:

```text
needs_full_projection_rebuild = true
repair_action_state = inspect_only or safe_to_apply only after full rebuild gate passes
rebuild-position-authority --full-projection-rebuild preview should report safe_to_apply=1 only if all full rebuild gates pass
recommended_command should be rebuild-position-authority --full-projection-rebuild --apply --yes
```

The normal non-full rebuild path must remain blocked for this case if it cannot safely interpret the existing lots.

## Suggested implementation direction

Do not blindly apply this patch if a better repo-native design exists, but evaluate this direction carefully.

Introduce an explicit concept such as:

```python
projection_excess_with_materialized_fragmentation = bool(
    not bool(projection_convergence.get("converged"))
    and lot_row_count > 1
    and projected_total_qty > portfolio_qty + _EPS
    and not projection_repair_covers_excess
    and other_active_qty > _EPS
    and not portfolio_projection_publication_present
)
```

Then make `historical_fragmentation_projection_drift` true only for a narrow, evidence-rich subset, for example when:

```python
projection_excess_with_materialized_fragmentation
and (
    "historical_fragmentation" in diagnostic_flags
    or portfolio_projection_repair_recorded
    or portfolio_projection_repair_event_status == "recorded_but_not_current_state_proof"
)
```

If this condition is met and `historical_fragmentation` is not already in `diagnostic_flags`, append a diagnostic flag or add a more precise new flag such as `materialized_projection_fragmentation`. Choose the naming that best preserves existing semantics.

Be careful: diagnostic flag naming affects tests and operator reports. Prefer adding more explicit observability rather than hiding the condition.
