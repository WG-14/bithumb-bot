## Hypothesis 1: target_delta BUY can bypass the performance gate

### Concern

The previous patch added `evaluate_strategy_performance_gate()` and appears to call it in `broker/live.py` for live real-order BUY entry. However, target-delta submit plans may be handled in an early target-delta branch inside `_determine_live_execution_intent()`.

If that branch returns before the normal BUY branch, then this case may bypass the performance gate:

```text
MODE=live
LIVE_DRY_RUN=false
LIVE_REAL_ORDER_ARMED=true
EXECUTION_ENGINE=target_delta
performance_gate.allowed == false
target_submit_plan.side == BUY
target_submit_plan.submit_expected == true
```

This must not be allowed. A negative-expectancy strategy must not quietly create a BUY rebalance via target_delta just because it used a different execution path.

### Required behavior

When all of these are true:

```text
MODE=live
LIVE_DRY_RUN=false
LIVE_REAL_ORDER_ARMED=true
target-delta final submit side == BUY
strategy performance gate is enabled and not allowed
```

then the target-delta BUY submit must be blocked before broker submission.

The block must:

```text
- preserve target-delta diagnostics
- keep SELL / flatten / target-closeout / panic-stop / reconcile / recovery paths available
- emit an explicit reason code
- be visible in order suppression / decision telemetry / ops-report or health where appropriate
```

Use existing reason-code patterns where possible, but the resulting operator-facing reason must clearly indicate strategy performance gating, e.g.:

```text
STRATEGY_PERFORMANCE_BLOCKED
STRATEGY_EXPECTANCY_NEGATIVE
STRATEGY_SAMPLE_INSUFFICIENT
STRATEGY_NET_PNL_NEGATIVE
STRATEGY_PROFIT_FACTOR_LOW
STRATEGY_FEE_DRAG_EXCESSIVE
```

### Tests to add

Add focused tests covering at least:

```text
test_target_delta_buy_rebalance_blocked_by_strategy_performance_gate_when_live_armed
test_target_delta_hold_rebalance_buy_blocked_by_strategy_performance_gate_when_live_armed
test_strategy_performance_gate_does_not_block_target_delta_sell
test_strategy_performance_gate_does_not_block_recovery_or_flatten_commands_if_applicable
```

The tests should simulate enough lifecycle rows or monkeypatch the performance gate so the block is deterministic.

Do not only assert that the gate function returns blocked. Assert that the target-delta BUY submit path itself becomes non-submitting.

Expected assertions should include:

```text
submit_expected == false
final_action is a block action, not REBALANCE_TO_TARGET
block_reason includes strategy performance gate reason
target_submit_plan remains present for observability
target_submit_plan.side == BUY
target_sizing / invariant_status are still visible if they were computed
```

Also include a positive control:

```text
target_delta SELL remains allowed when performance gate is blocked
```

because poor strategy performance must not prevent risk-reducing exits.

## Hypothesis 2: SELL floor remainder can create ambiguous target-state semantics

### Current known behavior

The updated tests appear to confirm this kind of behavior:

```text
desired SELL qty: 0.0004998
qty_step: 0.0001
final submitted qty: 0.0004
rejected remainder: about 0.0000998
dust_policy: exchange_step_remainder_tracked
invariant_status: passed
```

This is a reasonable sizing policy. The remaining issue is what happens after the fill.

If target_delta wants target exposure 0, but the actual broker/local state still has a below-step or below-min residual, the next loop must not repeatedly attempt closeout or present inconsistent “flat” state without evidence.

### Required behavior

After target-delta SELL floors a requested closeout and leaves a remainder:

```text
- the rejected remainder must be observable
- the remainder must be classified as tracked dust / true dust / below-min residual according to existing dust and runtime-readiness semantics
- the next HOLD cycle with target_exposure_krw=0 must not submit another SELL if the remaining qty is below exchange minimum / below step / non-executable
- ops-report / health / decision telemetry must make the reason clear enough for an operator
```

Do not hide the residual by pretending the system is literally flat if broker/local quantity remains nonzero. It is acceptable to treat it as effective-flat only if the dust model supports that and reports it clearly.

### Tests to add

Add focused tests covering at least:

```text
test_target_delta_sell_floor_remainder_updates_or_reports_dust_target_state
test_target_delta_sell_floor_remainder_does_not_repeat_submit_when_remainder_below_min
test_target_delta_sell_floor_remainder_appears_in_ops_report_or_decision_context_as_tracked_dust_or_true_dust
```

Use existing DB helpers and state models where possible. If full live fill simulation is too large for this patch, build the smallest realistic test around the decision / execution / readiness layer and document the remaining integration boundary.

The tests must prove:

```text
- rejected_remainder is present and nonzero
- dust_policy is explicit
- next target_delta HOLD / SELL logic does not create an endless closeout loop for non-executable remainder
- operator-facing context distinguishes:
  literal flat
  effective flat due to dust
  tracked dust / true dust / non-executable residual
```

## Implementation guidance

Prefer a design-level fix over local symptom suppression.

If performance gate enforcement currently lives only in the normal BUY branch, factor it into a small helper that can be called from both:

```text
- normal BUY entry path
- target_delta BUY rebalance path
```

The helper should answer:

```text
should_block_new_entry_or_buy_rebalance(...)
```

It should not block:

```text
SELL
flatten-position
target-closeout
panic-stop
cancel-open-orders
reconcile
recover-order
resume safety evaluation
```

If target-delta remainder handling is currently only diagnostic, make sure the state transition or next-loop behavior is safe. If a full state model change is too broad, at least add explicit observability and no-repeat-submit guard coverage.
