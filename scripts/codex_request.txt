## Primary problem statement

The repo already appears to use `GET /v1/orders/chance` as the authority for order capability, side-specific min_total, and fee fields (`bid_fee`, `ask_fee`, `maker_bid_fee`, `maker_ask_fee`). However, there is a strong structural risk that fee handling is split across multiple truth sources:

* exchange-derived fee/rule sync from `/v1/orders/chance`
* local fallback rule resolution
* env/config-based estimated fee inputs such as `LIVE_FEE_RATE_ESTIMATE`
* post-trade fee observation / fee-gap repair flows

Do **not** weaken this core problem framing. The likely higher-order issue is not “one wrong formula” but **distributed fee authority** across rule sync, execution sizing, strategy gating, accounting, and repair.

### Hypothesis B — fallback can mask stale or wrong fee authority

When `/v1/orders/chance` sync fails, persisted snapshots or local fallback may allow continued operation while fee authority becomes stale, degraded, or ambiguous.

**Verify by:**

* tracing `source_mode`, `fallback_used`, snapshot age, and fee provenance through rule resolution

* adding tests for fallback vs merged snapshot behavior specifically for fee-sensitive execution paths

**Falsify by:**

* proving that degraded or fallback fee authority cannot drive normal entry decisions in armed live mode

### Hypothesis C — current architecture is repair-heavy rather than prevention-first

Zero-fee or fee-pending observations are handled mainly through downstream classification and repair flows, meaning the system expects fee inconsistency rather than preventing it upstream.

**Verify by:**

* tracing how fee_pending / zero-fee / fee-gap debt is created
* identifying whether upstream execution/accounting could have avoided those states
* adding scenario tests from rule sync → execution → fill accounting → fee observation → repair preview

**Falsify by:**

### Hypothesis E — fee parser is too strict for future/real-world fee states

The code may assume all fee fields from `/v1/orders/chance` must be strictly positive, which could be structurally brittle if zero-fee or exceptional maker cases ever appear.

**Verify by:**

**Falsify by:**

## Expected design direction

Prefer a repository-wide design change that establishes a **single canonical fee authority** and explicit degraded behavior.

### Target design goals

1. Introduce a canonical runtime fee authority object or equivalent abstraction, e.g. `FeeAuthoritySnapshot`, with at least:

   * bid fee
   * ask fee
   * maker bid fee
   * maker ask fee
   * provenance/source mode
   * freshness / staleness metadata
   * degraded/fallback status
   * operator-facing diagnostic summary

2. Ensure that the same canonical fee authority flows through:

   * rule resolution
   * execution sizing
   * strategy cost-edge / expected edge
   * post-trade accounting
   * fee observation
   * operator reporting / diagnostics

3. Redefine `LIVE_FEE_RATE_ESTIMATE` or equivalent:

   * it must **not** remain a silent parallel runtime authority in normal mode
   * either remove it from normal runtime decision paths, or demote it to an explicitly labeled degraded-mode fallback only
   * if retained, its precedence and allowable use must be explicit and test-covered

4. Make degraded mode explicit:

   * if fee authority is stale / fallback-only / unknown in live armed mode, new entry should be blocked or tightly restricted
   * existing position management may be handled differently, but the mode must be explicit, testable, and operator-visible

5. Improve numeric consistency:

   * move fee/min_total/price_unit handling toward `Decimal`-safe behavior where this affects boundary correctness
   * avoid partial conversion that leaves decision points still dependent on `float` drift

6. Preserve and improve repair/reporting, but make them downstream safety nets rather than primary operating assumptions

7. Add raw evidence preservation for fee incidents where appropriate, e.g. an incident bundle containing:

   * raw `/v1/orders/chance` response or normalized authority snapshot
   * submit payload details
   * fill payload / normalized fill evidence
   * fee observation status
   * repair basis / rationale
   * enough metadata to reproduce and audit the incident
