The objective is to move the bot toward a more robust, reproducible, and operable automated trading experiment system by introducing target-position execution, initially in shadow/telemetry mode, without weakening current live safety gates.

Do not add more residual-specific branches as the primary solution. The core problem is deeper:

```text
The bot currently behaves like a lot-native order lifecycle engine with residual/dust/recovery policies layered on top.

  current_position
  target_position
  delta = target_position - current_position

Because of this, real-world partial fills, fees, step-size rounding, and residual BTC become exceptional states. Each exception adds another policy branch: dust_only_remainder, residual_inventory, closeout candidate, residual live sell mode, residual buy sizing mode, recovery gates, etc.

This is making the repo safer in small patches but structurally more complex and harder to reason about.
```

## Context from the live incident

Do not treat this as a one-off bug. The live state showed the structural issue:

```text
Recent orders:
  BUY  qty_req=0.0005  qty_filled=0.00049998
  SELL qty_req=0.0004  qty_filled=0.0004

Portfolio:
  asset_qty=0.0004998

Observed behavior:
  raw_signal=SELL
  final_action=CLOSE_RESIDUAL_CANDIDATE
  submit_expected=0
  pre_submit_proof_status=passed
  execution_block_reason=residual_live_sell_mode_telemetry
  residual_inventory_state=RESIDUAL_INVENTORY_TRACKED

Also:
  BUY can be blocked by dust_only_remainder / residual inventory.
  Normal strategy SELL can be blocked because sellable_executable_lot_count=0.
  Residual SELL can be recognized but blocked by residual policy mode.
```

This means the bot is not executing a simple model of:

```text
BUY signal  -> target_position = configured target exposure
SELL signal -> target_position = 0
Execute     -> order delta between current position and target position
```

Instead, it is operating through separate concepts:

```text
strategy lot authority
open_position_lots
dust_tracking_lot_count
residual inventory tracking
residual close candidate
BUY entry gate
SELL lot authority gate
recovery/reporting gates
```

These safety surfaces are valuable, but they should not remain the primary execution model if the intended experiment is repeated BUY/SELL cycles with clean target exposure behavior.

## Design goal

Introduce a target-position execution engine behind a feature flag, starting with shadow telemetry only.

Do not replace the current live execution path in this patch unless explicitly safe and tested. The first goal is to make the target engine observable and comparable against the current lot-native engine.

The target engine should model:

```text
Signal layer:
  BUY / SELL / HOLD from the existing strategy

Target layer:
  BUY  -> target_exposure_krw = MAX_ORDER_KRW or configured target exposure
  SELL -> target_exposure_krw = 0
  HOLD -> maintain previous target exposure

Position truth layer:
  current_qty = broker/local converged BTC qty
  current_exposure_krw = current_qty * reference_price

Delta layer:
  target_qty = target_exposure_krw / reference_price
  delta_qty = target_qty - current_qty
  delta_notional_krw = delta_qty * reference_price

Execution classification:
  delta_qty > order threshold  -> BUY candidate
  delta_qty < -order threshold -> SELL candidate
  abs(delta) below threshold   -> no-op / true dust
  inconsistent state           -> blocked recovery
```

Important: lot-native state must remain available for accounting, PnL attribution, audit, and compatibility. But in the target engine, lot-native SELL authority should not be the primary execution quantity source.
