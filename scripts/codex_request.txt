You are Codex working directly in the GitHub repository `WG-14/bithumb-bot`.

Your task is to investigate and implement a structural fix for a live EC2 recovery failure in the Bithumb trading bot. Do not treat the current diagnosis as proven fact. Use it as the starting hypothesis, then verify or falsify it from the repository code, tests, and a minimal reproducible fixture.

## Problem context

A live EC2 bot instance stopped after a final SELL fill. The initial trigger was a fee-pending SELL accounting issue, but that was repaired manually. After that repair:

* Orders were no longer unresolved.
* `recovery_required_count=0`.
* Broker and local portfolio balances converged.
* Full projection rebuild succeeded in matching total lot projection to portfolio.
* Resume was still blocked by:

```text
POSITION_AUTHORITY_CORRECTION_REQUIRED
target_lot_qty_fill_mismatch=target_qty=0.000399850000,fill_qty=0.000599990000
```

The relevant live sequence observed on EC2 was:

```text
trade 27 BUY  qty=0.00059999 BTC
trade 28 SELL qty=0.00040000 BTC
current broker/local BTC balance = 0.00039985 BTC
```

The remaining `open_position_lots` row after full projection rebuild looked like:

```text
entry_trade_id=27
entry_client_order_id=live_1777042500000_buy_ae2ac30c
qty_open=0.00039985
position_state=dust_tracking
dust_tracking_lot_count=1
internal_lot_size=0.0004
lot_rule_source_mode=full_projection_rebuild_portfolio_anchor
entry_decision_linkage=degraded_recovery_unattributed
```

The working hypothesis is:

> A `full_projection_rebuild` created a portfolio-anchor projection row that correctly matched current broker/portfolio quantity, but the subsequent readiness/authority assessment treated that synthetic portfolio-anchor row as a normal fill-native BUY residual. It then compared the anchor row’s `qty_open=0.00039985` to the original latest BUY fill quantity `0.00059999`, generated `target_lot_qty_fill_mismatch`, and blocked resume even though broker, accounting, and projection totals had converged.

Do not blindly patch around `target_lot_qty_fill_mismatch`. The goal is not to suppress a symptom. The goal is to separate truth-source semantics so the recovery system remains safe, reproducible, and operational.

## Hypotheses to verify or falsify

Keep these as hypotheses, not assumptions.

### Hypothesis A: Portfolio-anchor projection rows are being misclassified as fill-native correction targets

`_replace_with_portfolio_anchored_projection()` appears to create `open_position_lots` rows with:

```text
lot_rule_source_mode=full_projection_rebuild_portfolio_anchor
```

These rows are anchored to a recent BUY only as provenance. Their `qty_open` may represent the current portfolio/broker balance, not the natural residual of that single BUY fill.

Verify whether `build_position_authority_assessment()` later treats such rows as normal fill-native lots and applies fill-native invariants such as:

```text
target_total_qty == fill_qty
target_total_qty == expected_residual_qty
```

Confirm or refute by building a minimal test scenario and/or by tracing these fields:

```text
target_trade_id
target_client_order_id
target_total_qty
target_dust_qty
fill_qty
portfolio_qty
projection_convergence.converged
portfolio_projection_publication_present
lot_rule_source_mode for the target row
needs_correction
safe_to_correct
blockers
repair_action_state
reason
```

### Hypothesis B: `full_projection_rebuild` conflates “matching current portfolio quantity” with “reconstructing fill-native position authority”

Investigate `_apply_full_projection_rebuild()` and `_replace_with_portfolio_anchored_projection()`.

Verify whether the full rebuild first replays lifecycles, then deletes/replaces `open_position_lots` with a portfolio-anchored projection that is quantity-converged but no longer a true per-fill lifecycle decomposition.

A valid full projection rebuild may still be safe, but only if subsequent readiness logic understands the row’s projection kind and applies projection-specific invariants.

### Hypothesis C: The “latest BUY only” authority assessment is structurally fragile with accumulated residual/dust

`build_position_authority_assessment()` appears to focus on the latest accounted BUY and its rows. This may fail when the current residual portfolio is a combination of older residual plus the latest BUY’s residual.

Reproduce or falsify with a test fixture:

```text
prior residual exists, e.g. 0.00019986
latest BUY qty = 0.00059999, executable_lot_count=1, internal_lot_size=0.0004
latest SELL qty = 0.00040000
current portfolio/broker qty = 0.00039985
full projection rebuild creates one portfolio-anchor dust row
assessment should not require this portfolio-anchor row’s qty to equal latest BUY fill qty
```

### Hypothesis D: The recovery state machine lacks an actionable “safe_to_apply=false” diagnostic path

When `safe_to_apply=false`, the operator currently gets blocked but may need to read code to understand whether this is:

* missing broker evidence,
* missing projection publication,
* unsafe open orders,
* stale repair event,
* fill-native mismatch,
* portfolio-anchor projection mismatch,
* or a code/model limitation.

Inspect `recovery-report`, `restart-checklist`, and health output paths. Improve observability if needed.

## Required design direction

Prioritize a design change over a narrow symptom patch.

The system should distinguish at least these row semantics:

```text
fill_native_lot
  - created from a specific BUY fill’s executable/dust split
  - fill quantity, lifecycle SELL quantity, and residual quantity are valid invariants

portfolio_anchor_projection_lot
  - created to attest current broker/portfolio quantity after recovery
  - entry_trade_id is provenance anchor only
  - fill quantity equality must not be required
  - must require strong broker/accounting/projection/publication evidence

manual_external_position_adjustment_lot
  - created from external adjustment/manual repair
  - should not be interpreted as a natural BUY-fill residual
```

Do not necessarily add a large migration unless necessary. It is acceptable to start by centralizing provenance classification using existing columns such as `lot_rule_source_mode`, but the code must make the semantic split explicit, named, and tested.

## Minimum expected implementation

Implement a robust fix that satisfies all of the following:

1. Add or refactor a small explicit helper/model that classifies `open_position_lots` provenance or projection kind, at least recognizing:

```text
full_projection_rebuild_portfolio_anchor
ledger / fill-native rows
manual/external adjustment rows if already represented
unknown rows fail closed
```

2. Update `build_position_authority_assessment()` so a converged portfolio-anchor projection row is not treated as a fill-native correction target.

A portfolio-anchor projection may be considered converged only under strict conditions. Suggested minimum conditions:

```text
target lot row source/provenance is full_projection_rebuild_portfolio_anchor
projection_convergence.converged is true
target_total_qty matches portfolio_qty
portfolio_projection_publication_present is true, or another current-state attestation exists and is verified
unresolved_open_order_count == 0
recovery_required_count == 0
unresolved fee pending state is false
no executable exposure is being inferred from qty alone
row remains dust_tracking if executable_lot_count == 0
```

Use the repository’s existing available data paths. If not all checks belong inside `position_authority_state.py`, keep the separation clean and document the boundary.

3. Do not globally ignore `target_lot_qty_fill_mismatch`.

For fill-native lots, this mismatch must still block unless a valid partial-close residual normalization applies.

4. Strengthen the full projection rebuild postcondition.

After `_apply_full_projection_rebuild()` publishes a portfolio-anchor projection, it should not leave the system immediately in `AUTHORITY_CORRECTION_PENDING` due to misclassifying its own output.

If the post-assessment still requires correction, the full rebuild should either:

* fail with a precise postcondition error, or
* classify the portfolio-anchor projection as converged and resume-safe according to the stricter portfolio-anchor invariants.

5. Improve operator diagnostics for `safe_to_apply=false`.

At minimum, ensure the preview/report surfaces:

```text
target row provenance/projection kind
whether the blocker is fill-native or portfolio-anchor specific
which evidence is missing
whether manual DB update is unsafe
what command or code path is next
```

Do not add noisy logs in normal operation. Prefer structured report fields or existing report output.

## Non-goals and safety constraints

Do not:

* simply remove the `target_lot_qty_fill_mismatch` blocker;
* allow qty-only inference of executable exposure;
* make resume possible when broker/portfolio/accounting/projection evidence is not converged;
* add a manual DB UPDATE workaround as the primary fix;
* weaken fail-closed behavior for unknown row provenance;
* overfit to exactly `0.00039985` or the specific EC2 order IDs.

The fix must generalize to the problem family:

```text
fee repair + partial close + accumulated dust/residual + full projection rebuild + broker/portfolio reconciliation
```
