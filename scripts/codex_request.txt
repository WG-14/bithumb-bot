## Core problem statement

The likely direct cause is that a historical portfolio-anchored repair materialized an invalid current projection, but the more important question is whether the repo has a deeper structural flaw:

* multiple truth layers may coexist (`orders/fills/trades/portfolio`, materialized `open_position_lots`, repair-event history)
* read paths and write paths may not share the same invariants
* repair events may be treated as historical evidence in readiness logic while simultaneously acting as effective state writers in repair logic
* `open_position_lots` may be functioning like a long-lived source of truth instead of a disposable derived projection
* tests may recognize this incident class but still fail to enforce production-grade convergence guarantees

Your task is to determine which of those are real, which are not, and then implement the highest-priority design correction.

## Priorities

Keep these priorities in this order unless repo evidence forces a different order:

1. **Design correction over symptom patch**

   * do not start by patching row-5-specific behavior only
   * do not ship a narrow “ignore this one mismatch” fix
   * do not delete rows ad hoc without a design reason

2. **Shared invariants and deterministic convergence**

   * unify the write-path and read-path truth relations
   * ensure any published projection is either converged or rejected

3. **Derived-state discipline**

   * prefer making `open_position_lots` a disposable derived projection if repo evidence supports that direction
   * separate evidence logging from current-state publication

4. **Reproducibility and operability**

   * incident must become reproducible in tests
   * recovery semantics must become explainable and observable
   * operator-facing diagnostics must reflect the actual state model

## Desired design direction

Favor changes in this direction if repo evidence supports them:

* treat `orders/fills/trades/portfolio` plus explicit accounting adjustments as canonical truth
* treat `open_position_lots` as a rebuildable/materialized projection, not an independent truth source
* separate:

  * repair evidence append
  * projection rebuild request
  * projection publish / attestation
* ensure every repair/apply path uses one convergence oracle and one publish gate
* prevent any repair path from persisting a projection that the readiness layer will later reject as non-current or non-converged
* make incident-grade replay possible from captured DB/runtime state

Do **not** force this architecture if repo evidence decisively proves a better alternative. But if you choose a different direction, explain why.
