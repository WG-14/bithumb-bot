Continue working directly in the GitHub repository `WG-14/bithumb-bot`.

Do not treat this as a generic CLI beautification task. The EC2 smoke run showed that the research pipeline correctly generated artifacts, correctly failed all candidates, and correctly prevented promotion. The remaining issue is operator observability: the machine-readable reports contain useful diagnostic evidence, but the immediate CLI summary does not explain why no candidate can be promoted.

This patch must improve the repository as a reproducible and operable automated-trading research system. It must not change strategy results, gate logic, promotion logic, live execution behavior, broker behavior, path policy, env loading, or artifact hash semantics.

## Core problem statement

The direct EC2 symptom was:

```text
research-backtest and research-walk-forward produced reports
gate_result=FAIL
best_candidate_id=null
operator had to use jq repeatedly to understand why
```

The deeper structural issue is:

```text
Research artifacts are diagnostically rich,
but operator-facing diagnosis is too weak.
```

The CLI currently prints experiment id, hashes, candidate count, best candidate, gate result, artifact paths, content hash, and warnings, but it does not summarize candidate failure reasons, window failure reasons, promotion eligibility, or next safe action.

This can lead to operator mistakes:

```text
- lowering gates blindly
- attempting promotion when best_candidate_id is null
- inspecting the wrong experiment report
- confusing backtest failure with walk-forward failure
- confusing no-trade windows with negative-expectancy strategy failure
```

Your task is to add an explicit research run summary layer and wire it into the research CLI.

    ...

The function must be defensive:

## Summary fields to compute

Compute at least:

```text
candidate_gate_counts
top_fail_reasons
promotion_allowed
nearest_failed_candidate_id
nearest_failed_candidate_fail_reasons
walk_forward_window_count
walk_forward_pass_window_count
walk_forward_fail_window_count
top_window_fail_reasons
next_action
```

### candidate_gate_counts

Count candidate `acceptance_gate_result`.

Example:

```text
PASS:0,FAIL:8
```

If no candidates exist:

```text
none
```

### top_fail_reasons

Aggregate every candidate’s `gate_fail_reasons`.

Example:

```text
profit_factor_failed:8,walk_forward_failed:8,min_trade_count_failed:4
```

### promotion_allowed

Must be true only when the report itself indicates there is a promotable candidate.

Suggested rule:

```text
promotion_allowed = bool(report.best_candidate_id) and report.gate_result == "PASS"
```

Do not infer promotion allowed from a failed or nearest candidate.

Print as:

```text
promotion_allowed=1
```

or:

```text
promotion_allowed=0
```

### nearest_failed_candidate_id

When no PASS candidate exists, show a separate diagnostic candidate id.

This must not replace `best_candidate_id`.

Use the first candidate in the sorted report list if it exists, since reports are already sorted by candidate rank.

Example:

```text
best_candidate_id=none
nearest_failed_candidate_id=candidate_004_1e322c70
nearest_failed_candidate_fail_reasons=profit_factor_failed,walk_forward_missing
```

Important invariant:

```text
best_candidate_id remains none when all candidates fail.
nearest_failed_candidate_id is diagnostic only and must never be used as a promotion candidate.
```

### walk_forward_window_summary

If any candidate has `walk_forward_metrics`, compute aggregate window counts from the first available candidate’s walk-forward metrics.

Example:

```text
walk_forward_window_summary=window_count:5,pass:0,fail:5
```

### top_window_fail_reasons

Aggregate window-level `fail_reasons` across all candidates and all walk-forward windows.

Example:

```text
top_window_fail_reasons=test_metrics_gate_incompatible:40
```

### next_action

Add a simple deterministic next action.

Suggested mapping:

```text
promotion_allowed=1
→ next_action=review_promotion_candidate

no candidates
→ next_action=inspect_dataset_or_manifest

top_fail_reasons includes walk_forward_missing
→ next_action=run_walk_forward_before_promotion

top_fail_reasons includes walk_forward_failed
→ next_action=do_not_promote_review_walk_forward_windows

top_fail_reasons includes profit_factor_failed or min_trade_count_failed
→ next_action=do_not_promote_revise_strategy_hypothesis

otherwise gate_result=FAIL
→ next_action=inspect_report_or_adjust_hypothesis_not_promote
```

Keep this conservative. Do not suggest lowering gates.
