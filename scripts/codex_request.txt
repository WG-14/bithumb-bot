## Primary objective

Verify and, if confirmed necessary, patch promotion-time calibration lineage binding.

The suspected gap:

`src/bithumb_bot/research/lineage.py` has `execution_calibration_artifact_hash` in research lineage and `research-reproduce` can detect missing or mismatched calibration hashes later. `promotion_gate.py` also records `execution_calibration_artifact_hash` in the promotion artifact. However, `promote_candidate()` may not explicitly pass the candidate calibration hash into `build_promotion_lineage()` or fail immediately when the base research lineage calibration hash disagrees with the candidate calibration hash.

This means a bad or stale base lineage calibration hash could possibly survive until `research-reproduce` or approved-profile verification catches it. That is too late. Promotion should fail closed before writing the promotion artifact.

Keep this as a hypothesis until verified.

## Required patch: promotion-time calibration lineage fail-fast

If H3 or H4 is confirmed, patch the code so promotion fails closed before writing a promotion artifact when candidate calibration hash and base lineage calibration hash disagree.

Preferred implementation shape:

1. Extend `build_promotion_lineage()` in `src/bithumb_bot/research/lineage.py` with an optional parameter:

```python

```

2. Inside `build_promotion_lineage()`:

* validate the base lineage first as it currently does
* read existing base lineage `execution_calibration_artifact_hash`
* normalize empty strings to `None`
* if both base lineage calibration hash and supplied candidate calibration hash exist and differ:

  * raise `LineageValidationError("lineage_execution_calibration_artifact_hash_mismatch")`
* if supplied candidate calibration hash exists:

  * set `lineage["execution_calibration_artifact_hash"] = supplied_hash`
* if no supplied hash exists:

  * preserve existing base lineage hash if present
* recompute `lineage_hash` after the update.

3. In `promote_candidate()`:

* compute `candidate_calibration_hash = _candidate_calibration_hash(candidate)` once.
* use that value for:

  * promotion artifact `execution_calibration_artifact_hash`
  * `build_promotion_lineage(..., execution_calibration_artifact_hash=candidate_calibration_hash, ...)`
* avoid repeated `validate_production_calibration_policy(candidate)` calls if possible; store the result once for clarity and deterministic artifact construction.
* if `build_promotion_lineage()` raises `LineageValidationError`, surface it as:

  * `promotion refused: lineage_execution_calibration_artifact_hash_mismatch`
  * or the existing `promotion refused: {exc}` pattern if that is already the repo convention.

5. Preserve current `research-reproduce` calibration hash checks. Do not weaken them.
