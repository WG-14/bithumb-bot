## Design objective

Add a safe repair path for **verified flat-state stale lot projection**.

This repair should activate only when the system has strong evidence that the real position is flat and the only remaining mismatch is stale `open_position_lots` projection.

The repair should not weaken startup gates. It should produce an auditable repair event and make the post-repair materialized projection converge to broker/portfolio truth.

## Required safety preconditions

The new repair path may be considered only if all are true:

```text
1. broker_qty_known = true
2. broker_qty == 0 within asset epsilon
3. portfolio_qty == 0 within asset epsilon
4. broker_portfolio_converged = true
5. no unresolved open orders
6. no remote open orders
7. no PENDING_SUBMIT
8. no SUBMIT_UNKNOWN
9. no RECOVERY_REQUIRED orders
10. accounting projection is consistent with portfolio, or any inconsistency is explicitly reported and blocks repair
11. latest terminal order is SELL and FILLED, or there is otherwise strong terminal SELL evidence
12. latest trade after that SELL has asset_after == 0
13. open_position_lots projected qty > 0
14. open_position_lots rows are only stale materialized projection rows, preferably dust_tracking-only for this initial implementation
15. no executable broker exposure exists
```

For the concrete EC2 case, the first implementation may require:

```text
open_position_lots rows with qty_open > 0 are all position_state='dust_tracking'
```

If there are open_exposure rows as well, either handle them only if the same flat proof is strong enough, or block with a clear reason. Do not silently broaden the repair.

## Required repair behavior

When the preconditions pass, the repair must:

```text
1. Create a position_authority_repair event.
2. Include complete evidence in repair_basis:
   - broker_qty
   - portfolio_qty
   - latest SELL order id
   - latest SELL trade id
   - latest SELL qty
   - latest trade asset_after
   - open_position_lots rows before repair
   - projected_total_qty before repair
   - startup gate / readiness blocker reason
   - operator command and timestamp
3. Neutralize stale open_position_lots projection.
4. Record a position_authority_projection_publication or equivalent audit record if that is the existing pattern.
5. Verify postconditions:
   - projected_total_qty == 0
   - portfolio_qty == 0
   - broker_qty == 0
   - projection_converged = true
   - semantic contract still passes
   - no new startup gate blocker remains from position authority
6. Make recovery-report show that this blocker is cleared.
```

Neutralization should use repository repair patterns. Prefer a repair function over direct command-level SQL.

Acceptable approaches:

```text
- delete stale open_position_lots rows only inside a repair transaction with a position_authority_repair event and full before/after evidence
- or mark/replace projection through an existing projection publication mechanism if that is the repository’s canonical pattern
```

Unacceptable approaches:

```text
- manual SQL instructions as the fix
- hidden deletion without audit event
- bypassing startup gate
- treating this as harmless dust
- changing target_delta behavior
- changing strategy SELL/BUY logic
```
