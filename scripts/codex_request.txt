## Background problem statement to preserve

The current repository likely does **not** treat Bithumb `myOrder` as a strict exchange-truth event contract. It appears to normalize it into a loose internal event shape first, which may be causing broader operational failures, including but not limited to fee issues.

The attached `myOrder` reference indicates that the important official fields include at least:

* `ask_bid`
* `order_type`
* `state`
* `uuid`
* `trade_uuid`
* `price`
* `volume`
* `executed_volume`
* `remaining_volume`
* `paid_fee`
* `reserved_fee`
* `remaining_fee`
* `executed_funds`
* `trade_timestamp`
* `order_timestamp`
* `timestamp`

The repository may currently be misaligned in ways such as:

* using `side` / `order_side` / `type` instead of official `ask_bid`
* using `trade_id` / `fill_id` / `uuid` / `order_id` instead of official `trade_uuid`
* not treating official `state=trade` as a fill-producing event
* preferring `executed_volume` over `volume` for per-trade fill quantity
* preferring general `timestamp` over `trade_timestamp` for fill event time
* collapsing order-state semantics and fill-event semantics into one loose internal status model
* treating order-level fee fields as if they were always safely usable at trade level
* making stream-path and REST/recovery-path parse and reason about fills using different contracts

You must verify these concerns directly in repo code before changing anything.

#### Hypothesis E

### 3. Make design-first corrections

If the hypotheses are confirmed, do not merely patch the local symptom. Refactor toward a clearer architecture.

Prioritize the following changes if repo evidence supports them:

#### A. Introduce a strict `myOrder`-specific canonical contract

Do not keep using a loose generic normalizer if that is what the repo is doing now.

Create a documented, typed, exchange-contract-aware representation for Bithumb `myOrder`, separating at least:

* order lifecycle updates
* trade/fill events

This model must explicitly preserve official fields and distinguish:

* order UUID vs trade UUID
* per-trade quantity vs cumulative executed quantity
* trade timestamp vs delivery/event timestamp
* trade-level fee evidence vs order-level fee evidence

#### B. Separate fill observation from fill accounting

If current code blocks ledger application whenever fee evidence is incomplete, preserve safety, but do not lose the observed fill fact.

Refactor so the system can represent:

* fill observed
* fill accounted
* fee pending
* reconciliation pending
* manual recovery required

Do not let incomplete fee evidence erase or collapse the raw trade event truth.

#### C. Unify stream and recovery contracts

Ensure `myOrder` stream ingestion and REST/recovery lookup paths produce the same canonical fill evidence shape and apply the same reasoning rules.

Avoid having two different parsers with two different truth models for the same broker event family.

#### D. Make status semantics explicit

Do not force exchange `wait/trade/done/cancel` into an overly lossy internal model if it destroys meaning.

In particular, preserve that:

* `trade` means a fill event occurred
* `done` is an order terminal state, not necessarily a new fill
* order status and fill occurrence are related but not interchangeable

#### E. Improve observability and replayability

Preserve raw `myOrder` payloads and ensure the parsing/canonicalization layer is replayable.

If appropriate, introduce canonicalization versioning or equivalent replay-safe metadata so future parser corrections can be applied to historical event data.
