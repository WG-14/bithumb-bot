Current suspected cause:

* `target_delta` executed a terminal SELL for the full broker-verified position quantity.
* `lot-native` lifecycle projection consumes only `open_exposure` lots during SELL lifecycle accounting.
* `dust_tracking` lots are deliberately treated as operator evidence and are never matched in the ordinary SELL path.
* Therefore the real account became flat, but the materialized `open_position_lots` projection retained stale dust.

The repo currently contains a comment in `src/bithumb_bot/lifecycle.py` stating that SELL lifecycle consumes only the sellable open_exposure path and that `dust_tracking` lots remain operator evidence and are never matched. Verify this in code before changing behavior.

## Primary objective

Fix the structural mismatch between target-delta terminal SELL semantics and lot-native lifecycle projection semantics.

The correct fix is not “delete row 357.” The correct fix is to make the lifecycle/projection system understand a target-delta terminal flat SELL when sufficient evidence proves the SELL closed both executable open exposure and dust remainder.

## Non-goals and safety constraints

Do not:

* Manually mutate the live DB as the primary solution.
* Disable safety gates.
* Force resume.
* Make all SELLs consume `dust_tracking` rows unconditionally.
* Treat all dust as tradable inventory.
* Weaken operator-evidence protection for ordinary dust.
* Hide non-convergence by ignoring `open_position_lots`.
* Add a broad repair command that can erase dust without strong terminal-flat evidence.

Preserve:

* Fail-closed behavior when evidence is incomplete.
* Existing operator-evidence semantics for ordinary `dust_tracking`.
* Safety around unresolved orders, submit unknown, recovery-required orders, fee/accounting active issues, and broker/portfolio mismatch.
* Reproducibility of the lifecycle projection from trades/fills/orders.

### Hypothesis A: target-delta terminal SELL quantity includes dust, but lifecycle SELL only consumes open_exposure

### Hypothesis B: submit quantity authority and projection close quantity authority are different

Evidence to verify:

* Submit path uses `target_position_delta` and broker-verified total position.
* Projection close path uses sellable executable lot count / open exposure lots only.
* `submit_lot_count=1` and `sell_submit_lot_count=1` coexist with `final_submitted_qty=0.00059997`, implying one executable lot plus dust was submitted but projection treated one lot as close authority.

Falsification condition:

* If projection close quantity actually uses submitted qty but later reclassifies dust incorrectly, adjust the fix to that path.

### Hypothesis C: `dust_tracking` conflates active inventory residue, operator evidence, historical evidence, and stale terminal-sell residue

Evidence to verify:

* `dust_tracking` is included in projection totals and startup gate calculations.
* In ordinary cases it should not be consumed by SELL because it is operator evidence.
* In this terminal target-delta case it no longer represents actual inventory because broker/portfolio/trades are flat.

Falsification condition:

* If `dust_tracking` is not supposed to represent current projection and startup gate is incorrectly treating it as holdings, fix the gate/projection summary rather than lifecycle.

### Hypothesis D: repair-plan lacks a clear terminal-flat projection repair path even when broker/portfolio/trades prove flat

Evidence to verify:

* Top-level recommended command loops back to `uv run python bot.py repair-plan`.
* Reporting does not clearly separate current broker/portfolio truth from materialized projection drift.

Falsification condition:

* If a safe terminal-flat repair path exists but the user missed the correct command, surface it and add better reporting/tests.

## Design requirements

### 1. Add an explicit terminal-flat SELL lifecycle path

Implement a narrowly scoped path that can consume or close dust_tracking rows only when strong terminal-flat evidence exists.

Acceptable evidence may include:

* SELL trade has `asset_after == 0` within tolerance.
* SELL trade qty is at least `open_exposure_qty + dust_tracking_qty - tolerance`.
* SELL order / submit evidence identifies `target_delta_rebalance` or `sell_qty_basis_source=target_position_delta`.
* Broker/portfolio quantity is zero or converged to zero when the repair/reporting path is used.
* No unresolved orders, no submit_unknown, no recovery_required orders, no active fee/accounting issue.

Do not clear dust_tracking for ordinary lot-native SELLs that lack terminal-flat evidence.

### 2. Preserve ordinary dust operator-evidence semantics

The existing rule that dust_tracking is not normally matched by SELL is valid for ordinary cases. The fix must be conditional.

Add a negative test proving that a normal lot-native SELL or partial SELL without terminal-flat evidence does not consume dust_tracking.

### 3. Introduce or prepare for typed quantity authority

Prefer a typed helper or dataclass over more unstructured dict probing.

A possible shape:

```python
@dataclass(frozen=True)
class ExecutionQuantityAuthority:
    source: str  # e.g. "lot_native", "target_delta", "broker_verified_position"
    submitted_qty: float
    open_exposure_qty: float | None
    dust_tracking_qty: float | None
    terminal_asset_after: float | None
    terminal_flat: bool
    decision_reason_code: str | None
    evidence_source: str
```

If a full refactor is too large, implement a small forward-compatible resolver that can derive this authority from `orders`, `order_events.submit_evidence`, `trades`, and/or `fills`, and use it only for the terminal-flat path. Leave TODOs pointing to a fuller quantity-authority unification.

### 4. Repair-plan / reporting must become operationally clear

Improve repair-plan and/or recovery-report so operators can distinguish:

* current broker qty
* current portfolio qty
* latest trade asset_after
* materialized lot projection qty
* dust rows considered stale after terminal SELL
* whether terminal-flat stale-dust repair is safe
* exact blockers if not safe
* whether the command is inspect-only or mutating

Avoid self-loop guidance such as recommending only `repair-plan` again without a concrete next action.

Desired reporting fields:

```text
current_broker_qty=0
current_portfolio_qty=0
latest_trade_asset_after=0
materialized_lot_projection_qty=0.00019997
terminal_flat_sell_detected=1
stale_dust_rows_to_clear=[357]
safe_to_apply_terminal_flat_projection_repair=true|false
unsafe_reason=<specific blockers>
recommended_command=<only if safe and intentionally mutating>
```

### 5. Recovery must remain explicit and auditable

If the fix includes a repair action that mutates `open_position_lots`, it must:

* require an explicit `--apply --yes` style command if run as an operator repair,
* record an event in `position_authority_repairs` and/or `position_authority_projection_publications`,
* include preconditions and evidence in the repair record,
* have a postcondition that projection converges to broker/portfolio/trades,
* be safe to preview without mutation.

If the lifecycle fix prevents future drift but does not repair existing row 357 automatically, clearly state the safe post-patch operator flow.
