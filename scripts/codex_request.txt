## Primary hypothesis to verify

### Hypothesis A: Live SMA entry gating is not actually enforcing promoted candidate regime policy

There may be a function such as `evaluate_live_regime_policy`, and `src/bithumb_bot/strategy/sma.py` may import it, but `SmaCrossStrategy.decide()` may not actually load a promoted candidate artifact/profile and apply the allowed/blocked regime policy to block new BUY entries.

Verify whether the live/replay decision path actually does all of the following:

```text
1. Builds current_market_regime_snapshot at decision time.
2. Loads the currently promoted candidate policy, candidate profile, or equivalent artifact.
3. Reads regime_classifier_version.
4. Reads allowed_live_regimes or allowed_regimes.
5. Reads blocked_live_regimes or blocked_regimes.
6. Calls evaluate_live_regime_policy(...).
7. If base_signal or entry_signal is BUY and the current regime is not allowed, returns HOLD.
8. If current regime is explicitly blocked, returns HOLD.
9. If the candidate artifact has no regime policy, fails closed for live use.
10. Emits decision context/logging fields that make the block auditable.
```

If this is already fully implemented and tested, do not duplicate it. Instead, add only missing tests or documentation if needed.

If it is not fully implemented, implement it.

## Required design direction

This must be a structural live-operability patch, not another report-only patch.

The desired chain is:

```text
research regime attribution
→ regime acceptance gate
→ promoted candidate artifact with allowed/blocked regimes
→ live/replay current regime classification
→ live/replay BUY blocked unless current regime is explicitly allowed
→ auditable decision context explaining why
```

The system must not silently treat missing regime policy as “allowed everywhere.”

For live use, missing regime policy must fail closed unless there is an explicit, documented, tested override. Prefer no override unless the repo already has a standard safe-override pattern.

### 1. Add a clear promoted regime policy loading path

### 2. Fail closed for missing or invalid live regime policy

For live/replay entry gating, missing candidate policy must result in:

```text
entry_signal = HOLD
regime_decision = OFF
regime_block_reason = regime_policy_missing
```


Use the repo’s existing error/warning style.

It should not block exits. A SELL needed for risk reduction should not be prevented just because the current regime is not allowed.

If the strategy has an existing position and an exit rule fires, keep exit behavior intact.
