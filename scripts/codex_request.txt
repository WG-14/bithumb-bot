You are working in GitHub repository `WG-14/bithumb-bot`.

This is a production-safety and experiment-system hardening task. Do not treat this as a cosmetic `broker_qty_unknown` suppression patch. The current problem is that `live-dry-run` successfully runs without submitting orders, but the live account evidence read during `/v1/accounts` preflight is still not connected to the actual target-delta decision path.

Use the EC2 acceptance result as context, but keep all root causes falsifiable.

## Core problem

The problem is not that `LiveAccountEvidence` does not exist. It does exist.

The problem is that the live account evidence pipeline is not connected end-to-end:

```text
cmd_live_dry_run()
→ /v1/accounts preflight succeeds
→ accounts evidence is visible in logs
→ decision generation path still uses dry_run_static
→ build_live_account_evidence() does not receive the real accounts snapshot/diagnostics
→ target-delta receives broker_qty_known=0
→ target-delta blocks with broker_qty_unknown
```

The goal of this patch is to connect the existing evidence object to the actual `live-dry-run` decision path.

Do not broaden real-order behavior. Do not make missing BTC row globally mean zero. Known-zero is valid only under explicitly safe live-dry-run conditions.

## Required design direction

Do not add another evidence object. The evidence object already exists.

Connect the existing evidence object into the live-dry-run decision path:

```text
actual /v1/accounts snapshot
→ build_live_account_evidence(snapshot, diagnostics)
→ evidence.as_broker_position_evidence()
→ runtime readiness / target-delta input
→ build_execution_decision_summary()
→ _apply_live_dry_run_no_submit_policy()
→ record_strategy_decision(context_json)
```

The final live-dry-run decision must be based on `accounts_v1_rest_snapshot`, not `dry_run_static`.

## Concrete implementation requirements

### 1. Acquire actual account evidence inside `cmd_live_dry_run`

Inside the `live-dry-run` command path, fetch a live account snapshot using the same live account source / broker diagnostic path that powers the `/v1/accounts` preflight.

Required behavior:

```text
MODE=live
LIVE_DRY_RUN=true
LIVE_REAL_ORDER_ARMED=false
command=live-dry-run
```

must use account evidence from:

```text
accounts_v1_rest_snapshot
```

not:

```text
dry_run_static
```

If account snapshot acquisition fails, fail closed or record explicit account evidence failure. Do not quietly fall back to `dry_run_static` as broker evidence for target-delta.

### 2. Build `LiveAccountEvidence` from the actual snapshot and diagnostics

Use the existing function:

```text
build_live_account_evidence(snapshot, diagnostics)
```

For EC2’s current case:

```text
/v1/accounts payload has KRW row
BTC row is missing
execution_mode=live_dry_run_unarmed
base_currency_missing_policy=allow_zero_position_start_in_dry_run
preflight_outcome=pass_no_position_allowed
local position flat
unresolved orders none
```

the resulting decision context must contain:

```text
balance_source=accounts_v1_rest_snapshot
balance_source_preflight_outcome=pass_no_position_allowed
balance_source_execution_mode=live_dry_run_unarmed
base_currency_missing_policy=allow_zero_position_start_in_dry_run
broker_qty_known=1
broker_qty=0.0
broker_qty_evidence_policy=missing_base_row_known_zero_for_live_dry_run
broker_qty_evidence_reason=valid_accounts_snapshot_missing_base_row_allowed_for_unarmed_live_dry_run
```

### 3. Feed evidence into target-delta input

The evidence must affect target-delta behavior, not only reporting.

`build_execution_decision_summary()` or the equivalent target-delta input path must receive:

```text
broker_qty_known=1
broker_qty=0.0
balance_source=accounts_v1_rest_snapshot
broker_qty_evidence_policy=missing_base_row_known_zero_for_live_dry_run
```

Under the valid known-zero case, `execution_block_reason` must not remain `broker_qty_unknown`.

Acceptable outcomes:

```text
target_delta_side=NONE
target_block_reason=no_delta / already_at_target / no_target
```

or, if target exposure requires BUY:

```text
target_delta_side=BUY
target_would_submit=1
target_submit_allowed=0
submit_expected=0
final_action=BLOCK_LIVE_DRY_RUN_NO_SUBMIT or STRATEGY_PERFORMANCE_BLOCKED
```

`orders/fills/trades` must remain unchanged.

### 4. Preserve no-submit invariants

`live-dry-run` must still never:

```text
submit broker orders
create orders rows with PENDING_SUBMIT / NEW / FILLED
create fills
create trades
mutate portfolio as if an execution occurred
attach exchange_order_id
apply fills
```

Only diagnostic decision rows are allowed.

### 5. Preserve real-order safety

Do not make this inference globally true:

```text
missing BTC row == broker_qty=0
```

Known-zero from missing base row is valid only when all safe conditions hold:

```text
MODE=live
LIVE_DRY_RUN=true
LIVE_REAL_ORDER_ARMED=false
execution_mode=live_dry_run_unarmed
base_currency_missing_policy=allow_zero_position_start_in_dry_run
accounts response is valid
quote row is present and parseable
local position is flat or explicitly harmless/effective-flat
unresolved/open/recovery-required orders are absent
startup gate is not blocking
```

Unsafe cases must remain fail-closed:

```text
LIVE_REAL_ORDER_ARMED=true
LIVE_DRY_RUN=false
accounts transport/auth/schema failure
quote row missing
local position nonzero and not harmless/effective-flat
unresolved/open/recovery-required orders present
runtime readiness blocks resume
```

### 6. Make evidence visible in decision telemetry and ops-report

`decision-telemetry` must show at least:

```text

balance_preflight
```


from:

```text

broker_position_evidence=unknown

```

Current EC2 failure showed:

```text

broker_qty=0.00000000
broker_qty_evidence_policy=missing
balance_source=-
local_position_state=FLAT
```

After patch, valid live-dry-run known-zero should not display broker evidence as unknown.

Do not weaken:

```text
live-dry-run no-submit invariant
```

Do not make missing BTC row globally equal zero.
