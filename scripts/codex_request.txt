## Core problem to address

The structural risk is still this:

> Strategy semantics may remain distributed across research backtest logic, runtime strategy logic, canonical decision adapters, decision-equivalence comparison, and promotion evidence validation. If the repo can mark incomplete or manually prepared decision payloads as canonical, or if it cannot generate research/runtime decision lists internally, then “research-validated strategy” can still drift from “paper/live runtime strategy” while appearing validated.

A passing `pytest` run is not enough. The test suite must cover the failure modes that matter for this problem class.

## Hypotheses to validate or falsify

Keep these as hypotheses until verified in the current code.

### H1. Canonical decision completeness is too weak

The current code may classify a payload as canonical when it only has `decision_contract_version >= 1`. If so, `normalize_canonical_decision()` may fill missing semantic fields with `""`, `None`, or falsey values. Two incomplete decision lists could then compare equal and produce a passing canonical decision-equivalence report.

Validate by inspecting `is_canonical_decision()`, `normalize_canonical_decision()`, and `compare_decision_equivalence()`.

Falsify only if the repo already has a promotion-grade canonical validation path that fails on missing required semantic fields.

### H2. Decision-equivalence may not bind each decision to the requested report-level profile, market, interval, and data fingerprint

The comparator may compare research decisions to runtime decisions, but not verify that each decision payload is bound to the `profile_hash`, `market`, `interval`, and `data_fingerprint` passed into `compare_decision_equivalence()`.

Validate by creating tests where research/runtime decisions agree with each other but both carry the wrong profile hash, market, interval, or data fingerprint.

Falsify only if such cases already fail with explicit reason codes.

### H3. `order_rules_hash` may not reflect actual runtime order rules

The runtime canonical adapter may compute `order_rules_hash` from `position_gate["order_rules"]`, while `strategy/sma.py` may not put order rules into `position_gate`. If true, runtime canonical decisions may use a hash of `{}` for order rules and fail to detect order-rule drift.

Validate by inspecting `_build_position_gate_context()` and `runtime_decision_to_canonical_event()`.

Falsify only if runtime canonical events already contain a non-empty, deterministic, actual order-rule hash or equivalent order-rule snapshot hash.

### H4. Research backtest still does not directly emit canonical decisions

`BacktestRun` may still contain metrics/trades/warnings/regime/execution summaries but no canonical decision list. If so, `export_research_decisions()` may only normalize externally prepared dicts instead of making research decision generation repo-owned and deterministic.

Validate by inspecting `BacktestRun`, `run_sma_backtest()`, and validation protocol output artifacts.

Falsify only if the research engine already records canonical-compatible decision events at signal boundaries.

### H5. Repo-owned decision generation CLI is still incomplete

`cmd_decision_equivalence()` may still only compare two JSON files. If there are no repo-owned commands for exporting research decisions and replaying runtime decisions, then operators still have to manually prepare JSON decision lists.

Validate by inspecting `profile_cli.py`, `research/cli.py`, and CLI dispatch.

Falsify only if commands equivalent to the following already exist and are tested:

```bash
bithumb-bot research-export-decisions ...
bithumb-bot runtime-replay-decisions ...
bithumb-bot decision-equivalence ...
```

### H6. Golden research/runtime equivalence tests are still missing

Existing tests may be synthetic: one test builds a fake `StrategyDecision`, another normalizes a fake research dict. That is useful but not sufficient. The repo needs tests that run both the research decision path and the runtime replay path on the same candle/state/profile fixtures.

Validate by inspecting tests.

Falsify only if there are golden tests that execute both sides and compare canonical events across entry, blocked entry, exit, deferred exit, max holding, dust/position, and candle-boundary cases.

### H7. Exit-rule scope is probably already sufficiently clarified, but verify it

The previous patch appears to document and test that the current SMA runtime exit rules are exactly `opposite_cross` and `max_holding_time`, and that standalone `stop_loss` / `take_profit` are not implemented. Verify this remains true. Do not add stop-loss or take-profit unless the strategy/profile contract explicitly requires it.
