# CLI requirements

## Command name

Add:

```text
strategy-sweep
```

to the existing `bithumb-bot` CLI.

## Options

Support at minimum:

```text
--short
--long
--edge-buffer
--min-expected-edge
--slippage-bps
--pair
--interval
--from
--to
--through
--json
```

### List parsing

These should parse comma-separated values:

```text
--short 5,7,10
--long 20,30,50
--edge-buffer 0,0.0005,0.001
--min-expected-edge 0,0.0005,0.001
--slippage-bps 0,5
```

Parsing requirements:

* trim whitespace
* reject empty lists
* reject invalid integers/floats with useful CLI errors
* preserve deterministic order
* do not deduplicate unless the repo style already does so

### Pair and interval

Create a base config from existing settings:

```python
sma_strategy_config_from_settings()
```

Then use `dataclasses.replace()` or equivalent to override:

```text
pair
interval
```

when CLI options are supplied.

Do **not** mutate global `settings`.

### Time range

Support `--from`, `--to`, and `--through`.

Use the repo’s existing date/time parsing style if available. If `decision-attribution` uses a helper, reuse it.

If the repo only supports epoch ms easily, document the accepted format in help text and tests. Prefer consistency with existing commands over inventing a new parser.

Map these options into:

```python
run_sma_strategy_sweep(
    conn,
    base_config=...,
    grid=...,
    from_ts_ms=...,
    to_ts_ms=...,
    through_ts_ms=...,
)
```

### JSON output

For `--json`, emit a deterministic JSON array of summary rows.

Each row should include exactly the attribution-based fields from `StrategySweepSummaryRow`:

```text
config_id
short_n
long_n
entry_edge_buffer_ratio
strategy_min_expected_edge_ratio
slippage_bps
decision_count
raw_buy
final_buy
submit_expected_buy
raw_sell
final_sell
blocked_by_cost_filter_ratio
gap_lt_required_ratio
primary_issue
```

If you use `dataclasses.asdict()`, ensure the output is stable:

```python
json.dumps(..., sort_keys=True, ensure_ascii=False)
```

Do not include PnL fields.

### Human output

For non-JSON output, print a compact operator-readable report. Example:

```text
[STRATEGY-SWEEP]
rows=12

config_id          short long edge_buffer min_edge slippage decisions raw_BUY final_BUY cost_block gap_lt_required primary_issue
abc123...          5     20   0.0005      0.0010   0.0      120       14      3         0.42       0.77            gap_below_required_edge
```

The exact formatting can follow repo style, but it must expose the same core fields.


