## Core problem statement to preserve

Do not reduce this task to “apply a 2 KRW repair.” The current highest-priority design problem is:

```text
The live ingestion parser and aggregation path now appear to preserve fee authority better,
but the recovery layer still does not convert deterministic accounting evidence into a safe,
idempotent, concrete repair-plan action.

Recovery-report can diagnose:
  deterministic_allocation_available=1
  operator_confirmation_required=0
  fee_delta=2.0

But repair-plan cannot yet emit:
  safe_to_apply=1
  exact client_order_id
  exact fill_id
  exact fee
  exact fee_provenance
  exact recommended command
```

This means the system still depends on a human manually copying diagnostic values into a repair command. That is not deterministic recovery. It is manual repair with better diagnostics.

The next design-level fix should make recovery-report evidence and repair-plan action share the same source of truth.

## Required safety invariants before marking safe_to_apply=1

At minimum, validate these before recommending an apply command:

```text
1. affected pending fill still exists in broker_fill_observations or fill accounting incident projection
2. affected fill accounting status is not already complete
3. complete fill set is available
4. order_paid_fee is available
5. fee_delta equals order_paid_fee - sum_observed_or_applied_fill_fee within KRW fee rounding tolerance
6. proposed fee is positive and reasonable under configured fee rate tolerance
7. proposed fee plus already applied/observed fees equals order_paid_fee within tolerance
8. no existing fee_pending_accounting_repair idempotency record for this exact client_order_id/fill_id/fee
9. no unresolved open orders, submit_unknown, or recovery_required orders
10. broker/portfolio asset exposure is compatible with applying only a fee/accounting repair, not a position mutation
11. the repair command must use normal accounting/repair event paths, not direct row edits
```

Do not weaken startup gates. Do not auto-resume trading.

## Long-term design note to include in docs or final report

Include a concise roadmap toward this architecture:

```text
OrderSnapshot
  exchange_order_id
  client_order_id
  executed_volume
  executed_funds
  order_paid_fee

FillExecution
  fill_id
  qty
  price
  funds

FeeEvidence
  fill_id
  source: trade_level | order_level_allocated | operator_confirmed | estimate
  authority: authoritative | validated | inferred | missing
  accounting_status
  provenance
  validation_checks

LedgerPosting
  source_fill_id
  principal_cash_delta
  asset_delta
  fee_cash_delta
  evidence_id
```

Long-term principle:

```text
canonical ledger = sum of fill-level postings
aggregate fill = reporting/projection/view only
```

Target replay flow:

```text
raw exchange snapshot
+ parser version
+ accounting version
+ existing ledger postings
=> replay expected postings
=> diff against current DB
=> classify:
   auto-apply safe
   operator confirmation required
   unsafe / code bug
```

Manual `fee_pending_accounting_repair` should remain an explicit, audited, idempotent event path, but repair-plan should fill in the exact command when evidence is deterministic. Manual value selection should be the exception, not the normal recovery workflow.
