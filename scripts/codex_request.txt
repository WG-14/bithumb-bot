# Part 1: close the remaining SMA config boundary

## Required patch

```python
settings.STRATEGY_ENTRY_SLIPPAGE_BPS
settings.ENTRY_EDGE_BUFFER_RATIO
```

replace that hidden global dependency with explicit parameters.

Change `_load_position_context()` to accept:

```python
slippage_bps: float
entry_edge_buffer_ratio: float
```

Then replace uses like:

```python
exit_slippage_bps=float(settings.STRATEGY_ENTRY_SLIPPAGE_BPS)
exit_buffer_ratio=float(settings.ENTRY_EDGE_BUFFER_RATIO)
```

with:

```python
exit_slippage_bps=float(slippage_bps)
exit_buffer_ratio=float(entry_edge_buffer_ratio)
```

Update the call from `SmaCrossStrategy.decide()`:

```python
position, exposure, position_state = _load_position_context(
    conn,
    pair=self.pair,
    candle_ts=ts_list[-1],
    market_price=float(closes[-1]),
    signal_context=signal_context,
    slippage_bps=float(self.slippage_bps),
    entry_edge_buffer_ratio=float(self.entry_edge_buffer_ratio),
)
```

This is a small but important structural fix. It ensures `SmaStrategyConfig` controls the cost parameters used not only in entry intent and edge filtering, but also in position/executable-lot interpretation.

# Part 2: add candle-based decision replay

## Objective

Add an initial replay module that answers this question:

> Given historical candles and an explicit `SmaStrategyConfig`, how would SMA raw/final/block decision attribution change?

This is **not** a full backtest. This is **not** a PnL engine. This is **not** a trading simulator.

The replay result must flow through the existing attribution path:

```text
replay decision context
→ DecisionAttribution
→ summarize_decision_attributions()
→ DecisionAttributionSummary
```

This is the point of the architecture: live dry-run, historical decisions, and future sweep results must be comparable using the same attribution model.

## Add file

Prefer:

```text
src/bithumb_bot/strategy_replay.py
```

## Suggested types

Use names close to these:

```python
@dataclass(frozen=True)
class StrategyReplayConfig:
    strategy_config: SmaStrategyConfig
    from_ts_ms: int | None = None
    to_ts_ms: int | None = None
    through_ts_ms: int | None = None
```

```python
@dataclass(frozen=True)
class StrategyReplayResult:
    config_id: str
    attribution_summary: DecisionAttributionSummary
    decision_count: int
```

You may include additional diagnostic fields if useful:

```python
insufficient_candle_count: int
candle_count: int
```

Keep the JSON/reporting surface deterministic.

## Replay scope

Initial replay should do only this:

```text
load candles for pair/interval
recompute SMA raw signal for each replayable candle window
compute gap_ratio
compute required_edge_ratio / cost_floor_ratio using SmaStrategyConfig
apply cost/edge filter
produce replay decision contexts
normalize to DecisionAttribution
summarize with summarize_decision_attributions()
```

The replay should generate a decision for each candle index where enough prior candles exist.

Minimum context fields per replay decision should include enough for attribution:

```text
raw_signal
final_signal
decision_type
base_reason
entry_reason
entry_block_reason
gap_ratio
required_edge_ratio
blocked_by_cost_filter
signal_strength_label
primary_block_layer
primary_block_reason
all_block_reasons
experiment_fingerprint or config_id
```

For cost-filtered BUY/SELL candidates, set canonical block data consistently, for example:

```python
primary_block_layer = "strategy_filters"
primary_block_reason = "cost_edge"
all_block_reasons = ["strategy_filters.cost_edge"]
blocked_by_cost_filter = True
```

For non-filtered candidates, do not invent execution success. Since this replay is not an execution engine, be conservative with `submit_expected`. It can be `None` unless the repo already has a clear replay convention.

## Config ID

Create a deterministic `config_id`.

Acceptable implementation:

```text
stable JSON of SmaStrategyConfig + replay range → sha256 prefix
```

Do not include runtime timestamps.

Do not include nondeterministic object reprs.

## Explicit non-goals

Do not implement:

```text
strategy-sweep CLI
PnL
net PnL
max drawdown
fee drag
order submission simulation
fill simulation
broker reconciliation
cash drift replay
accounting repair replay
lot-native lifecycle replay
performance gate replay
```

This replay is a decision-attribution replay only.

## Safety requirements

The replay must be read-only.

It must not:

```text
insert into strategy_decisions
insert/update orders
insert/update fills
insert/update portfolio
insert/update open_position_lots
call broker APIs
call recovery/repair functions
submit/cancel orders
change runtime state
```

Use only candle reads and pure calculations.

If you need to query SQLite, use read-only SELECTs only.

## Tests for strategy replay

Add:

```text
tests/test_strategy_replay.py
```

Required test cases:

### 1. Insufficient candles

Given fewer than `long_n + 2` candles, replay returns:

```text
decision_count == 0
attribution_summary.sample_count == 0
```

or an equivalent explicit empty result. It must not crash.

### 2. Golden cross produces raw BUY

Use a simple deterministic candle fixture that creates a golden cross.

Expected:

```text
candidate_funnel.raw_BUY >= 1
raw_signal_counts includes BUY
```

### 3. High edge buffer turns BUY candidate into cost-filtered HOLD

Run the same candle fixture with two configs:

* low/no edge buffer
* high edge buffer

Expected for high buffer:

```text
raw_BUY >= 1
final_BUY lower than raw_BUY, or final_BUY == 0 if all BUY candidates filtered
blocked_by_cost_filter_ratio > 0
block_reason_counts includes strategy_filters.cost_edge
edge_stats.gap_lt_required_ratio is not None
```

This test is critical. It proves replay is useful for comparing “too loose vs too strict” settings without mutating `.env`.

### 4. Determinism

Run replay twice with the same candles and same config.

Expected:

```text
same config_id
same decision_count
same attribution_summary.as_dict()
```

### 5. Replay uses DecisionAttributionSummary

Assert the result’s summary is an instance of the existing `DecisionAttributionSummary`, not a parallel bespoke summary.

### 6. Replay is read-only

Create tables such as:

```text
orders
fills
portfolio
open_position_lots
```

with sentinel rows or row counts. Run replay. Assert row counts and sentinel values are unchanged.

If those schemas are difficult to create exactly, create minimal tables only if replay does not require them. The point is to ensure replay does not write.

### 7. Config-driven cost parameters

Use two `SmaStrategyConfig` values that differ only in `entry_edge_buffer_ratio` or `slippage_bps`. Assert the replay attribution changes accordingly while global `settings` remains unchanged.
