## Incident context

On EC2, the live system now appears accounting-ready according to `audit-ledger`, `repair-plan`, and `health`, but `fee-diagnostics` still reports an active fee-pending startup blocker.

Observed EC2 state:

```text
broker_fill_observations.accounting_status='fee_pending' rows = 13
fee_pending_accounting_repair_count = 8
fill_accounting_repaired_incident_count = 8

broker_fill_latest_unresolved_fee_pending_count = 0
unresolved_fee_state = 0
fill_accounting_active_issue_count = 0

blocking_incident_class = NONE
```

Yet `fee-diagnostics` prints something equivalent to:

```text
fee_pending_count=13
diagnostic_only_vs_startup_blocking=startup_blocking_due_to_active_fee_pending
startup_impact=active_fee_pending_blocks_resume
operator_action=resolve_fee_pending_before_resume
```

This is wrong if the 13 rows are historical/repaired/stale observations and canonical accounting says there are no active unresolved fee issues.

## Current likely bug

Verify this from the repo, especially:

```text
src/bithumb_bot/reporting.py
build_fee_rate_drift_diagnostics()
```

The suspected current behavior is:

```text
SELECT recent rows from broker_fill_observations
count rows where accounting_status == "fee_pending"
if fee_pending_count > 0:
    startup_impact = active_fee_pending_blocks_resume
    diagnostic_only_vs_startup_blocking = startup_blocking_due_to_active_fee_pending
    operator_action = resolve_fee_pending_before_resume
```

That is the wrong authority boundary.

`broker_fill_observations.accounting_status='fee_pending'` is an observation-layer/historical signal. It may include stale, duplicated, repaired, or previously pending observations. It must not be the startup-blocker authority.

Startup/restart blocker authority should come from canonical accounting/replay/readiness state, such as:

```text
compute_accounting_replay(conn)
unresolved_fee_state
broker_fill_latest_unresolved_fee_pending_count
fill_accounting_active_issue_count
fill_accounting_repaired_incident_count
fee_pending_accounting_repair_count
```

or the same canonical source used by `audit-ledger` / runtime readiness.

## Required design changes

### 1. Split historical observation counts from active blocker counts

In `build_fee_rate_drift_diagnostics()`, stop using `broker_fill_observations.accounting_status='fee_pending'` as the active startup-blocker authority.

Add or expose fields equivalent to:

```text
historical_fee_pending_observation_count
recent_fee_pending_observation_count
active_unresolved_fee_pending_count
repaired_fee_pending_incident_count
fee_pending_accounting_repair_count
unresolved_fee_state
fill_accounting_active_issue_count
broker_fill_latest_unresolved_fee_pending_count
```

Recommended semantics:

```text
historical_fee_pending_observation_count:
  count of historical/recent observation rows with accounting_status='fee_pending'

recent_fee_pending_observation_count:
  count within the current observation window

active_unresolved_fee_pending_count:
  canonical active unresolved fee issue count derived from accounting replay / audit-ledger authority

repaired_fee_pending_incident_count:
  count of repaired fee-pending incidents from canonical accounting replay

fee_pending_accounting_repair_count:
  count from fee_pending_accounting_repairs
```

### 2. Use canonical replay/readiness authority for startup impact

Use `compute_accounting_replay(conn)` or the same canonical source used by `audit-ledger`.

Suggested logic:

```text
active_unresolved_fee_pending_count =
  max(
    broker_fill_latest_unresolved_fee_pending_count,
    fill_accounting_active_issue_count,
    1 if unresolved_fee_state else 0
  )
```

Then:

```text
if active_unresolved_fee_pending_count > 0:
    startup_impact = active_fee_pending_blocks_resume
    diagnostic_only_vs_startup_blocking = startup_blocking_due_to_active_fee_pending
    operator_action = resolve_fee_pending_before_resume
    recommended_command = uv run python bot.py recovery-report

elif mismatch_count > 0 or historical_fee_pending_observation_count > 0:
    startup_impact = diagnostic_only_without_active_fee_pending
    diagnostic_only_vs_startup_blocking = diagnostic_only
    operator_action = review_fee_diagnostics
    recommended_command = uv run python bot.py fee-diagnostics

else:
    startup_impact = no_recent_fee_rate_drift_detected
    diagnostic_only_vs_startup_blocking = none
    operator_action = none
    recommended_command = none
```

For the EC2 case described above, the expected output is:
