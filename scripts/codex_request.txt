Use the incident-analysis context below as background, but do **not** treat it as proven fact. Verify every claim directly in `WG-14/bithumb-bot` before changing code. The current EC2 evidence and prior repo inspection suggest this is no longer a single “fee bug.” It appears to be a two-part evidence-granularity failure: Bithumb provides order-level `paid_fee` for multi-fill orders while the bot requires per-fill fees, and the bot’s position-authority rebuild gate cannot consume the same broker account evidence that other health surfaces can see.

## Mission

Do not add another isolated manual repair command. Do not patch symptoms by hiding blockers. Do not loosen safety gates broadly.

Your task is to make the live trading system more robust, reproducible, and operational by fixing the evidence model that connects:

```text
exchange evidence
→ canonical accounting
→ projection repair
→ operator recovery policy
```

The core principle to preserve:

```text
Observed evidence
≠ canonical accounting authority
≠ projection
≠ repair gate authority
```

The current failure loop is:

```text
1. Bithumb multi-fill order provides no per-fill fee, but provides order-level paid_fee.
2. Bot treats multi-fill order-level paid_fee as ambiguous.
3. Each fill is stored with fee=None / fee_pending.
4. Principal is not applied.
5. Operator manually repairs fees.
6. fills/trades/portfolio converge.
7. open_position_lots projection remains drifted.
8. rebuild-position-authority refuses to apply because broker_qty evidence is missing.
9. live_ready remains 0.
10. Operator is trapped in repeated manual recovery after every fill.
```

The next patch must break this loop structurally.

## Required design changes

### 1. Add a safe multi-fill order-level paid_fee allocation authority

Do not loosen fee validation blindly. Add a specific safe path for this evidence shape:

```text
per-fill fee missing
order-level paid_fee present
all fills for the order known
fill IDs stable
sum(fill funds) matches order executed_funds within tolerance
sum(allocated fees) equals order paid_fee after KRW rounding
allocated fees are coherent with configured/observed fee bounds
side/market/order identifiers match
no duplicate fill application
```

Expected successful outcome:

```text
fee_status=validated_order_level_paid_fee_allocated
fee_authority=exchange_order_paid_fee_allocated
fee_source=order_level_paid_fee
fee_confidence=validated
accounting_decision=finalize
accounting_status=accounting_complete
```

Expected failure behavior:

```text
fee_status=order_level_candidate
fee_authority=configured_estimate_or_missing_fee or equivalent pending authority
accounting_decision=pending_fee_validation
accounting_status=fee_pending
reason must explain exact failed gate:
  incomplete_fill_set
  executed_funds_mismatch
  order_paid_fee_missing
  allocated_fee_sum_mismatch
  identifier_mismatch
  duplicate_fill
  suspicious_fee_rate
```

Do not use configured fee estimate as settlement authority. Configured fee rate may be used as a diagnostic/coherence bound only.

Allocation rule should be deterministic and auditable. Prefer notional/funds proportional allocation with KRW rounding, assigning rounding residue deterministically, for example to the largest-funds fill or last deterministic sort key. The sum of allocated fees must equal the order-level `paid_fee`.

Use the incident fixture:

```text
order_paid_fee=27.83
fill A funds=42804.10588 qty=0.00036902 expected_allocated_fee=17.12
fill B funds=26791.36515 qty=0.00023097 expected_allocated_fee=10.71
```

This should no longer produce `fee_pending`.

### 2. Preserve fail-closed behavior for incomplete or incoherent multi-fill evidence

This is critical. Do not make all order-level paid_fee evidence automatically authoritative.

Must still fail closed when:

```text
not all fills are known
sum(fill funds) does not match order executed_funds
order-level paid_fee is missing, negative, zero for material notional, or unparseable
allocated fee sum cannot equal paid_fee after rounding
identifiers do not match
duplicate fills are present
fee rate is wildly outside configured/observed bounds
order is not terminal
```

### 3. Propagate broker account snapshot evidence into position-authority rebuild gates

Find the canonical place where `accounts_v1_rest_snapshot` is created or consumed.

The position rebuild gate should be able to consume a formal evidence object equivalent to:

```text

cash_locked=0
broker_qty=asset_available + asset_locked, or explicit policy-defined broker position qty

```

Then this scenario should be safe:

```text
portfolio.asset_qty=0.00079982
accounting_replay.qty=0.00079982
broker_qty=0.00079982
broker_qty_known=true
broker_portfolio_converged=true
open_orders=0
unresolved_fee_state=0

```

Expected:

```text

  needs_rebuild=true
  safe_to_apply=true
  repair_mode=full_projection_rebuild
  broker_qty_known=true
  broker_qty=0.00079982
  broker_portfolio_converged=true
```

Still refuse rebuild when:

```text
broker snapshot missing
broker snapshot stale
base/quote currency missing or mismatched
asset_locked > 0, unless explicitly allowed by policy
open orders present
unresolved fee pending
portfolio/replay mismatch
broker/portfolio mismatch
```

If rebuild is blocked, `repair-plan`, `recovery-report`, and/or `health` must show whether account evidence is missing entirely or merely not propagated to the repair gate.


### 5. Keep open_position_lots as projection, not canonical truth

Do not make `open_position_lots` a source of truth.

The intended model:

```text
canonical truth:
  exchange order/trade observations
  fills
  trades
  fee finalization / allocation events
  fee repair events
  external adjustments
  broker account snapshots

projection:
  open_position_lots
  trade_lifecycles
```

Projection drift should be repairable only when canonical accounting and broker evidence agree.

## Scope boundaries

Do:

```text
- Add safe deterministic multi-fill order-level paid_fee allocation.
- Preserve single-fill order-level paid_fee behavior.
- Preserve fail-closed behavior for incomplete/incoherent evidence.

- Improve repair-plan/recovery/health output for missing broker evidence.
```

```text

- Automatically trust partial fill sets.

- Delete historical lots without audited repair event.
- Resume live trading in tests or code paths.
```
