You are working in the GitHub repository `WG-14/bithumb-bot`.

Your task is to directly inspect, modify, and test the repository so that the SMA research/backtest/live pipeline evolves from “overall backtest return is enough” into a reproducible, regime-aware automated trading experiment system.

Do not reduce this to a cosmetic report-table patch. The core problem to investigate is:

> Overall-period profitability is not sufficient. Strategy candidates must be verified, approved, promoted, and operated under explicitly defined market-regime conditions.

The attached Korean analysis says the current likely issue is not merely “SMA gets whipsawed in sideways markets,” but a higher-level structural weakness: the research verification unit and the live operation decision unit may not yet speak the same regime language. Treat this as a hypothesis, not a proven fact. Verify it in the repo.

## Primary objective

Implement a design-oriented change that makes market-regime attribution and regime-aware acceptance a first-class part of the SMA experiment pipeline.

The final system should support this chain:

```text
shared MarketRegimeClassifier
→ per-candle/per-trade regime attribution in research backtests
→ regime-specific performance and coverage tables
→ regime-aware acceptance gate
→ promoted candidate artifact containing allowed/blocked live regimes
→ live strategy on/off decision based on current regime and promoted evidence
```

## Do not assume these hypotheses are true

Investigate and either support or refute each hypothesis with code evidence and tests.

### Hypothesis 1: Overall performance is hiding regime concentration

The SMA backtest may show acceptable total return, MDD, profit factor, or win rate while most gains come from a narrow subset of regimes, and losses accumulate in `chop`, `sideways`, `low_vol`, `downtrend`, or `volume_down` regimes.

Verification or refutation method:

Add regime attribution to trades and compute, at minimum:

```text

profit_factor
win_rate
expectancy
max_drawdown
max_consecutive_losses

```

Refute this only if validation/final_holdout/walk-forward results show stable expectancy across relevant regimes and no excessive PnL or loss concentration.

### Hypothesis 2: Research backtest and live strategy use different regime definitions or different decision contracts

The live strategy may already have a function like `classify_sma_market_regime`, while the research backtest may rely directly on `gap_ratio`, `range_ratio`, or similar filters instead of the same versioned regime contract.

Verification or refutation method:

For the same candle sequence and SMA parameters, compare research decisions and live/replay decisions by logging or testing:

```text

volatility_ratio
price_regime
volatility_bucket
volume_bucket
composite_regime
allows_sma_entry
final_signal
block_reason
regime_classifier_version
```

Refute this only if the same input candles produce the same regime snapshot, same allow/block decision, and same final signal under the same classifier version.

### Hypothesis 3: Existing regime classification may be too SMA-self-referential

If the only regime classifier is based almost entirely on SMA relationships, it may be repackaging “SMA signal strength” as “market regime.” That can make the research look more robust than it is.

Verification or refutation method:

Introduce or support independent market-regime axes that are not only SMA-cross/spacing based:

```text
price_regime: uptrend / downtrend / sideways
trend_strength_bucket: weak / normal / strong
volatility_bucket: low / normal / high
volume_bucket: decreasing / normal / increasing
liquidity_bucket: thin / normal / thick
composite_regime: e.g. uptrend_high_vol_volume_increasing
```

Compare SMA-specific regime outcomes against independent regime buckets where feasible. Do not remove useful SMA-specific checks, but separate “market regime” from “strategy entry permission.”

### Hypothesis 4: Walk-forward validation checks time stability but not regime coverage stability

The existing walk-forward result may pass because windows are favorable or dominated by uptrend periods. It may not prove that the strategy was tested across enough relevant market regimes.

Verification or refutation method:

Add walk-forward regime exposure and per-window regime metrics, for example:

```text

```

Refute only if walk-forward windows have adequate regime coverage and regime-specific results remain stable.

### Hypothesis 5: Volume and liquidity states are under-modeled

The requested analysis explicitly requires:

```text
rising market
falling market
sideways market
high volatility
low volatility
volume increasing
volume decreasing
```

If the repo currently models trend and volatility but not volume/liquidity as first-class regime dimensions, that is a structural gap.

Verification or refutation method:

Add or expose volume bucket calculations using defensible, deterministic rules such as moving-average volume ratio, quote-volume/turnover if available, or volume z-score. Add tests covering volume increasing, normal, decreasing, and missing/zero-volume edge cases.

## Design requirements

### 1. Create or refactor a shared market-regime module

Prefer a shared module such as:

```text
src/bithumb_bot/market_regime/
  __init__.py
  schema.py
  classifier.py
  thresholds.py
  metrics.py
```

Adjust the exact path to fit the repo’s existing layout.

The output must be a versioned structured snapshot, not only a string.

Example shape:

```python
@dataclass(frozen=True)
class MarketRegimeSnapshot:
    version: str
    price_regime: str
    trend_strength_bucket: str
    volatility_bucket: str
    volume_bucket: str
    liquidity_bucket: str
    composite_regime: str
    allows_sma_entry: bool
    block_reason: str
```

Use enums or literal constants if the repo style prefers them.

The classifier must be deterministic, testable, and reusable by both research/backtest code and live/replay strategy code.

### 2. Preserve compatibility while replacing duplicated logic

If the repo already has `classify_sma_market_regime`, do not blindly delete it. Refactor it into the shared classifier or make it a thin compatibility wrapper around the shared classifier.

Add tests proving the wrapper produces the expected historical labels or intentionally document changed semantics.

### 3. Add regime attribution to the research SMA backtest

The SMA backtest result should include first-class regime outputs.

Target shape, adjusted to repo style:

```python
@dataclass(frozen=True)
class BacktestRun:
    metrics: ResearchMetrics
    trades: tuple[dict[str, object], ...]
    candle_count: int
    warnings: tuple[str, ...]
    regime_performance: tuple[RegimePerformanceRow, ...]
    regime_coverage: tuple[RegimeCoverageRow, ...]
```

Each closed trade should include at least:

```text

exit_regime
entry_regime_snapshot
exit_regime_snapshot
net_pnl

```

If feasible without excessive complexity, also include:

```text

```

If holding-regime attribution is too large for this pass, implement entry/exit attribution now and leave a clear follow-up note in the final report.

Extend the existing acceptance gate instead of creating a disconnected checker.

Add config support for something like:

```json

  "min_trade_count_per_required_regime": 10,
  "required_regimes": ["uptrend", "high_vol", "volume_increasing"],

  "blocked_regime_max_trade_count": 0,
  "blocked_regime_max_net_pnl_loss_krw": 0,
```

The exact schema may differ, but the behavior must support:



### 7. Make promoted candidates carry live regime policy

A promoted candidate artifact must include regime evidence and policy, for example:

```json

  "candidate_id": "...",
  "regime_classifier_version": "market_regime_v2",
  "allowed_regimes": ["uptrend_normal_vol", "uptrend_high_vol_volume_increasing"],
  "blocked_regimes": ["sideways_low_vol", "downtrend", "low_vol_volume_decreasing"],
  "regime_evidence": {
    "uptrend_normal_vol": {
      "trade_count": 42,
      "profit_factor": 1.45,
      "expectancy": 123.4

    "sideways_low_vol": {
      "trade_count": 0,
      "policy": "blocked"
    }

```

The live strategy should be able to answer:

```text
current regime = sideways_low_vol_volume_decreasing
promoted candidate allowed_regimes = [uptrend_normal_vol, uptrend_high_vol_volume_increasing]
decision = OFF / block new entry
reason = current_regime_not_in_candidate_allowed_regimes
```

Do not rely only on environment variables or ad hoc live thresholds for this decision if candidate artifacts already exist or can be extended.
