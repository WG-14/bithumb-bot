## Primary design objective

Refactor the repo so that **operator-facing readiness/blocker semantics come from one canonical incident/verdict model**, with historical evidence clearly separated from active operational blockers.

Your implementation should strongly consider introducing a canonical internal object or equivalent layer with semantics close to:

* `incident_kind`: `none | active_fee_pending | active_fee_gap_unrepaired | historical_fee_gap_repaired | authority_missing | dust_only_tracked | open_executable | ...`
* `incident_scope`: `active_blocking | active_advisory | historical_context`
* `resolution_state`: `unresolved | repaired | not_applicable`
* `evidence`: counts, timestamps, adjustment totals, raw reconcile metadata, repair lineage details

This does **not** need to use these exact names if the repo has better-native terminology, but the semantic separation must be explicit and testable.

## What to change

### 1) Establish a canonical readiness/incident verdict layer

Implement or refactor so that:

* one canonical computation determines the current operational verdict for a given DB snapshot
* repair previews, runtime readiness, and operator reports all render from that same verdict layer rather than independently inferring state from overlapping raw metadata
* raw metadata such as `fee_gap_recovery_required` is treated as **evidence/input**, not automatically as the top-level operator verdict

### 2) Separate historical evidence from active blockers

Ensure the code can distinguish at least these cases cleanly:

* **Active fee-gap incident**

  * requires operator action
  * blocks resume or blocks closeout according to policy
* **Historical fee-gap evidence, already repaired**

  * still visible in evidence
  * does not present as an active unresolved incident
  * does not surface as “current recovery required”
* **Dust-only tracked residue**

  * execution-flat but not accounting-flat
  * should remain policy-distinct from fee-gap state
  * should not be conflated with active fee-gap debt

If needed, split existing reason/state concepts into separate concepts. For example, separate:

* current verdict/state
* reason code / causal evidence
* repair lineage / historical contamination marker

### 3) Reduce semantic dependence on `last_reconcile_metadata`

Do not remove it if it is still useful, but make sure:

* snapshot metadata is treated as recent reconcile evidence or cache,
* while canonical readiness/verdict is derived primarily from durable ledger + repair lineage + authoritative lot/exposure state,
* and operator surfaces do not treat stale or historical metadata as current truth.

### 4) Make operator-facing output machine- and LLM-readable

Health, recovery-report JSON/text, ops-report, restart-checklist, and fee-gap repair preview should clearly separate:

* verdict
* active issue status
* resume/closeout blocking policy
* next operator action
* historical context
* evidence

A result like “historical repaired fee-gap evidence exists” must not read like “manual recovery required now”.
