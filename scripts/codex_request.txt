You are working in the `WG-14/bithumb-bot` repository. Treat this as a repo-first engineering task, not a generic debugging exercise.

Context from production investigation points to a likely but not yet fully proven failure mode:

* A live Bithumb BUY order was actually filled on the exchange.
* The local DB kept the order in `orders` as `NEW` with `qty_filled=0`.
* No corresponding rows were created in `broker_fill_observations` or `fills`.
* `order_events` stopped after `exchange_order_id_attached` / `status_changed NEW`.
* Reconcile/health showed the run loop blocked by unresolved/open-order safety gates.
* There was also a reconcile exception consistent with `FILLED -> RECOVERY_REQUIRED` being disallowed.
* Current code suggests `fill_reading.py` maps `FillReadPolicy.ACCOUNTING_STRICT` to `parse_mode="strict"`, and `bithumb_order_queries.py:get_fills()` reads `/v1/order.trades` first.
* `_extract_fill_fee_observation()` appears to raise on materially sized fills with missing trade-level fee keys in strict mode.
* `recovery.py` appears to use `_mark_recovery_required_with_reason()` to directly downgrade order state to `RECOVERY_REQUIRED`.

Do **not** assume that this is the full root cause. Treat it as a leading hypothesis and investigate whether there is a higher-level structural cause. Your goal is to make the system more robust, reproducible, and operable as an automated trading experiment platform — not just suppress this symptom.

## Mission

Verify or falsify the likely root-cause hypotheses directly in the repo, then implement the smallest useful set of changes that moves the architecture in the right direction:

1. Prefer **design corrections over symptom patches**.
2. Preserve or improve **recovery safety**.
3. Improve **observability**, **reproducibility**, and **testability**.
4. Avoid baking in assumptions that only fit this one incident.

## Working hypotheses to evaluate

Keep these as hypotheses until tested:

### H1. Broker contract assumption is too strong

The code assumes trade-level fee keys are present for materially sized fills, but real `/v1/order` payloads may omit trade-level fee while still exposing fee elsewhere (for example at order level, or via another exchange surface). If true, strict accounting is rejecting a valid fill because the broker contract is modeled too narrowly.

### H2. Evidence collection and accounting are too tightly coupled

A strict accounting failure may prevent even `broker_fill_observations` from being recorded, causing the system to lose evidence and making recovery harder. If true, the repo needs a cleaner separation between evidence ingestion and ledger application.

### H3. Recovery state machine is non-monotonic / non-idempotent

If reconcile can see an order as effectively filled but then attempt a `FILLED -> RECOVERY_REQUIRED` transition, the recovery state model is structurally unsafe. If true, order truth and recovery incidents need to be separated.

### H4. Config-policy drift exists

The repo contains fee strictness-related settings (for example `LIVE_FILL_FEE_STRICT_MODE`, `LIVE_FILL_FEE_STRICT_MIN_NOTIONAL_KRW`, `LIVE_FILL_FEE_ALERT_MIN_NOTIONAL_KRW`), but the actual rejection path may be driven by different logic than operators would expect from those setting names. If true, configuration semantics need simplification or realignment.

## Deliverables

### 2. Design-first remediation

Implement changes that address the broader class of failures, not just this one payload.

Prioritize the following design direction unless the repo proves it inappropriate:

#### A. Separate evidence ingestion from accounting application

A broker read that yields partial/ambiguous fill evidence should still produce a durable observation record, even if accounting must be deferred.

A good outcome is:

* raw/normalized broker evidence is recorded
* parse warnings / fee status / confidence are retained
* ledger apply can be blocked without losing the evidence

#### B. Introduce explicit fill evidence quality states

Instead of collapsing everything into success vs exception, represent cases like:

* `trade_fee_complete`
* `order_level_fee_candidate`
* `fee_missing_material`
* `schema_variant_unknown`

These can be implemented with current types or modest type extensions, but the end state should allow policy decisions without destroying evidence.

#### C. Make recovery incident-driven instead of status-downgrade-driven

If practical within the current repo shape, move toward a model where exchange/order truth stays in states like `NEW/PARTIAL/FILLED/CANCELED`, while recovery concerns are tracked separately as incidents or observation/accounting statuses.

If a full incident model is too large for this task, still eliminate unsafe downgrade behavior:

* no `FILLED -> RECOVERY_REQUIRED` downgrade
* repeated reconcile should be idempotent
* recovery should preserve terminal truth and attach incident context elsewhere

#### D. Clarify or simplify fee strictness configuration

Either:

* wire strictness settings so they clearly govern the actual strict rejection path,
  or
* remove/deprecate misleading config if it is not part of the effective behavior.

### 3. Tests

Add or update tests so this class of problem becomes reproducible and hard to regress.

At minimum include:

#### Contract / parser tests

Use fixtures that model `/v1/order` variants for:

* single trade with trade-level fee present
* single trade with missing trade-level fee but order-level fee present
* single trade with missing fee everywhere
* multi-trade order with only order-level fee
* zero fee vs missing fee
* materially sized vs non-material fills

#### Reconcile / recovery tests

Cover scenarios where:

* exchange view is FILLED but local DB is NEW
* fill evidence is incomplete but present
* reconcile is run more than once on the same data
* no invalid status downgrade occurs
* evidence is recorded even when accounting is deferred

#### Regression fixture

If possible, create a redacted fixture derived from the incident shape so that:

* `get_fills(strict)`
* `get_fills(salvage)`
* reconcile/recovery behavior
  can be reproduced deterministically in tests

Do not invent exchange payload fields casually. If you need to model a payload shape, make it minimal and clearly justified by current parser expectations.

### 4. Operability / observability

Improve operator ergonomics where it materially helps.

Strongly consider adding or improving a diagnostic command/path so an operator can inspect one order end-to-end without manual SQLite spelunking. A useful result would include:

* raw broker payload or redacted raw evidence
* normalized order
* normalized fills
* observation rows
* accounting/incidents/recovery blockers
* next safe operator action

If a new CLI command is too large, at least improve logging/diagnostics in the existing relevant paths.
