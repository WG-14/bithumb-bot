Core problem to investigate and fix:

`run_sma_backtest()` appears to compute SMA signals using the current candle close and then simulate execution using that same candle close as `ExecutionRequest.reference_price`, with `signal_ts=candle.ts` and `decision_ts=candle.ts`. In this repository, `candles.ts` appears to be the candle bucket start timestamp, not the candle close timestamp. If true, the current backtest is not only same-close optimistic; it also records close-derived decisions at the candle start timestamp. This undermines execution realism, calibration, promotion, and reproducibility claims.

Preserve this priority: the goal is not “make the backtest less optimistic by adding slippage.” The goal is to introduce a repository-wide contract for when a signal is knowable, when a decision is made, when an order is submitted, what market evidence is used as the fill reference, and whether a candidate is promotable under that execution realism level.

Hypotheses to keep as hypotheses until verified:

H1. The root issue is not only “same close fill”; it is the absence of a time-axis contract. `candles.ts` may be bucket start, while backtest decisions based on candle close are currently stamped at that same bucket start.

H2. `StressExecutionModel.latency_ms` may currently be metadata only for price purposes: it records `submit_ts_assumption`, but it may not shift the fill reference to a later candle or quote.

H3. Top-of-book data may currently be execution metadata only, not the actual fill anchor. The execution models may still compute price from `reference_price` rather than BUY=ask and SELL=bid.

H4. Dataset quality and CLI output may expose limitations such as `execution_reference_price=candle_close`, but promotion may not fail closed on candle-close execution realism.

H5. Research backtest semantics may diverge materially from live/paper execution semantics. Live/paper paths appear to distinguish closed candles, orderbook bid/ask, submit/fill evidence, and latency more explicitly.

H6. Current data granularity may not support full-depth or intra-candle path realism. This is acceptable, but the system must make that limitation explicit and non-promotable where appropriate.

Implementation goals, in priority order:

P0. Introduce an explicit execution timing and fill-reference contract.

Add an execution timing policy concept to the research layer. It must be explicit in manifests or derived with clear legacy behavior. It should support at least:

* `signal_basis`: `closed_candle`
* `decision_time`: `candle_close` or `candle_close_plus_guard`
* `decision_guard_ms`: integer, default 0 unless configured
* `fill_reference_policy`:

  * `candle_close_legacy`
  * `next_candle_open`
  * `first_orderbook_after_decision`
  * `latency_adjusted_orderbook`
* `quote_selection`: at minimum `first_after_or_equal`
* `max_quote_wait_ms`: integer
* `missing_quote_policy`: `fail`, `skip`, or `warn`
* `allow_same_candle_close_fill`: boolean
* `min_execution_reality_level_for_promotion`: optional, for promotion checks

Do not silently preserve optimistic behavior as if it were realistic. If backward compatibility is necessary, retain legacy behavior only behind an explicit or clearly derived legacy policy, and label it in reports as optimistic and not promotable for live-ready profiles.

P1. Separate signal timing from execution timing in the backtest engine.

Refactor `run_sma_backtest()` so that signal generation and execution simulation are no longer an implicit same-candle action. Add small, testable data structures or helper functions for:

Signal event:

* `signal_candle_start_ts`
* `signal_candle_close_ts`
* `decision_ts`
* `side`
* `signal_reference_price`
* `signal_reference_source`
* `feature_snapshot` or equivalent minimal metadata
* `regime_snapshot`

Execution reference event:

* `submit_ts_assumption`
* `fill_reference_ts`
* `fill_reference_price`
* `fill_reference_source`
* `quote_ts`
* `quote_age_ms`
* `quote_source`
* `best_bid`
* `best_ask`
* `spread_bps`
* `execution_reality_level`
* `intra_candle_policy`
* failure reason if no valid reference exists

For candle-only safe mode, implement `next_candle_open`. If a signal is generated on candle `i`, the fill reference must be candle `i+1.open`; do not create a fill for the final candle if there is no next candle. `decision_ts` must be at or after `signal_candle_close_ts`, not equal to bucket start unless the interval semantics truly prove that `ts` is already a close timestamp.

Derive interval milliseconds from the dataset interval, using existing helpers where possible. Do not hard-code one-minute behavior except in tests.

P2. Add quote-aware execution reference selection.

For top-of-book based policies, do not use “nearest quote around candle timestamp” as a fill reference. Implement a directional selector for execution:

* Choose the first top-of-book snapshot with `quote.ts >= decision_ts` for `first_orderbook_after_decision`.
* Choose the first top-of-book snapshot with `quote.ts >= submit_ts_assumption` for `latency_adjusted_orderbook`.
* Enforce `max_quote_wait_ms`.
* Respect `missing_quote_policy`.
* Record quote age and source in every execution metadata object.
* BUY fill reference must be `best_ask`.
* SELL fill reference must be `best_bid`.

Keep the existing nearest-join behavior only for dataset quality/reporting if needed, but do not use it as execution fill evidence unless explicitly documented as a non-promotable legacy mode.

P3. Make latency affect price reference, not only metadata.

When a stress model has non-zero latency and the selected timing policy is latency-aware, the effective fill reference must be chosen after `decision_ts + latency_ms`. If latency changes and there are different later quotes or candle references, fill price must be able to change. Keep seeded randomness deterministic.

* `intra_candle_policy`

Update research reports, candidate profiles, and relevant hashes so that execution timing policy and execution reality level are part of reproducible lineage. Candidate profile hashes should change when execution timing policy changes.

Promotion must refuse candidates that are only supported by `candle_close_legacy` / same-close optimistic execution when the candidate is being promoted as paper/live-ready. Add a fail-closed gate such as:

* reject if `execution_reality_level` is below configured minimum

Tests to add or update:

1. `test_sma_signal_close_executes_next_candle_open_not_same_close`

Create a synthetic candle dataset where the signal candle close is 100 and the next candle open is 130. With `fill_reference_policy=next_candle_open` and zero slippage/fees, a BUY fill must use 130, not 100. Verify execution metadata includes `fill_reference_source=next_candle_open`.

2. `test_decision_ts_is_after_signal_candle_close`

For a dataset where `candles.ts` is bucket start and interval is `1m`, assert that `decision_ts >= signal_candle_start_ts + 60_000` for closed-candle signals. Also verify `signal_ts` is not misleadingly used as both bucket start and decision time unless explicitly documented.

3. `test_orderbook_policy_uses_first_quote_after_decision_not_nearest_before`

Create top-of-book snapshots before and after decision time. Ensure execution selects the first quote after decision, not the nearest quote before decision. Verify BUY uses ask and SELL uses bid.

4. `test_latency_changes_fill_reference_quote`

Create multiple top-of-book snapshots after decision time. Run with latency 0 and with a larger latency. Assert that `submit_ts_assumption`, selected quote, and fill reference can differ when latency crosses quote timestamps.

5. `test_top_of_book_model_uses_ask_for_buy_bid_for_sell`

Use close=100, ask=120, bid=80, slippage=0. BUY must fill from ask=120; SELL must fill from bid=80. A fill at 100 should fail the test.

6. `test_promotion_refuses_candle_close_execution_for_live_ready_candidate`

Construct a candidate/report that otherwise passes but uses `execution_reality_level=candle_close_optimistic` or `fill_reference_source=candle_close`. 7. `test_execution_metadata_includes_reference_source_and_quote_age`

Every trade execution metadata object must include reference source, reference timestamp, decision timestamp, submit timestamp, execution reality level, quote age, and quote source fields. Candle-only policies should use null quote fields but still include the keys.

8. `test_signal_event_quote_coverage_reported`

For a dataset with some signal events missing valid quotes, assert report-level signal quote coverage fields are present and deterministic.
