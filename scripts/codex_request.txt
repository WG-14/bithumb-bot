You are working in the `WG-14/bithumb-bot` repository.

This is not another dust/residual workaround task. The core issue is now clear: the repository does not have one coherent position policy. It currently mixes at least three different meanings of “position”:

```text
1. Strategy position:
   - SMA strategy emits BUY / SELL / HOLD.
   - Exit rules evaluate only when strategy thinks it is in position.

2. Lot-native position:
   - open_position_lots
   - executable_lot_count
   - dust_tracking_lot_count
   - sellable_executable_lot_count
   - entry_allowed / exit_allowed

3. Broker/target position:
   - broker_qty
   - target_position_state
   - target_qty - current_qty = delta
```

These are not aligned. That is the root cause. Do not treat this as a small bug in SELL sizing.

## Core problem

The repo keeps getting tangled because it alternates between these incompatible models:

```text
lot-native:
  Only the strategy lot created by the bot is a real position.
  Non-lot BTC becomes dust/residual/recovery state.

target-delta:
  Broker current BTC is the current position.
  Orders are computed from target position minus current broker position.
```

This has produced the failure mode we just observed:

```text
broker has BTC
local portfolio has BTC
but open strategy lot is absent
sellable_executable_lot_count = 0
dust_tracking_lot_count > 0
entry_allowed = false
exit_allowed = false

Result:
  BUY is blocked because BTC remains.
  Strategy SELL is blocked because no strategy lot exists.
  Residual/dust policy becomes the escape hatch.
```

The target-delta execution engine is directionally correct, but it was used incorrectly when we manually inserted `target_position_state target=0`. That caused the bot to sell existing BTC even though strategy signal was HOLD.

That SELL was technically correct under target_delta, but structurally wrong as a default startup policy:

```text
raw_signal=HOLD
final_signal=HOLD
reason=sma no crossover

target_position_state target = 0
broker_qty = 0.0004998
delta = -0.0004998
=> target_delta SELL
```

This is not an ideal automatic trading structure. It is a closeout behavior disguised as normal startup behavior.

## New design center

The next patch must implement a **startup position adoption policy**.

Do not add more residual sell modes.
Do not add more dust classifications.
Do not make target_delta always close current BTC to zero.
Do not confuse explicit closeout with strategy SELL.

The correct model is:

```text
Strategy changes target.
Execution converges current broker position to target.
Startup adopts existing broker position unless explicit policy says close it.
```

## Required policy

Implement and test this policy in `WG-14/bithumb-bot`:

```text
1. target_position_state exists and is fresh/valid:
   Use it.
   Do not overwrite it just because broker_qty exists.

2. target_position_state missing + broker/local flat:
   Initialize target_position_state to target=0.
   Do not submit an order.

3. target_position_state missing + broker/local has executable BTC:
   Adopt current broker exposure as the target.
   Initialize target_position_state to current broker exposure.
   Record target origin as adopted_existing_position.
   Do not submit an order on startup/HOLD.
   Continue running.
   The next strategy SELL should set target=0 and then sell.

4. target_position_state missing + broker BTC below exchange minimum:
   Classify as true_dust.
   Initialize target=0 or a dust-flat target state.
   Do not submit an order.

5. Strategy BUY:
   Set target_exposure_krw = TARGET_EXPOSURE_KRW if configured, otherwise MAX_ORDER_KRW.
   Execute delta from current broker position to target.

6. Strategy SELL:
   Set target_exposure_krw = 0.
   Execute delta from current broker position to zero.

7. Explicit operator closeout command:
   Only this should force target=0 outside strategy SELL.
   This must be separate from normal startup adoption.
```

## Key distinction

This must be enforced in code:

```text
Strategy SELL:
  Strategy explicitly says SELL.
  Target becomes 0.
  target_delta may sell.

Startup adoption:
  No target exists, but broker BTC exists.
  Target becomes current exposure.
  target_delta must not sell.

Operator closeout:
  Operator explicitly asks to close current position.
  Target becomes 0.
  target_delta may sell.
```

## Required implementation

### 1. Add target origin / adoption metadata

Extend target position persistence if needed.

At minimum, target state should be able to distinguish:

```text
strategy_buy
strategy_sell
adopted_existing_position
flat_start
true_dust_flat
operator_closeout
```

If schema migration is too large, use a compatible metadata column or adjacent table. Do not break existing DBs.

Recommended fields:

```text
target_origin
adoption_reason
adopted_broker_qty
adopted_broker_exposure_krw
created_from_signal
updated_ts
```

If you add fields, add migration and tests.

### 2. Implement startup adoption resolver

Add a focused function, for example:

```python
resolve_startup_target_position_policy(...)
```

It should take existing target state, readiness payload, broker position evidence, order rules, reference price, and execution settings.

It should return a structured decision:

```text
policy_action:
  use_existing_target
  initialize_flat_target
  adopt_existing_broker_position
  initialize_true_dust_flat
  block_unsafe_state

target_exposure_krw
target_qty
target_origin
would_submit_on_startup
block_reason
position_truth_state
dust_classification
```

Critical rule:

```text
adopt_existing_broker_position must never submit an order.
```

### 3. Wire adoption into run_loop

In `engine.py`, before building target_delta execution summary for a missing target state, apply startup adoption policy.

Desired flow:

```text
load target_position_state

if EXECUTION_ENGINE=target_delta:
  if target state missing:
    resolve startup adoption policy

    if safe flat:
      persist target=0 origin=flat_start

    if executable broker position:
      persist target=current broker exposure origin=adopted_existing_position
      continue with target equal to current position
      do not sell on startup/HOLD

    if true dust:
      persist target=0 origin=true_dust_flat
      do not sell

    if unsafe:
      block submit and surface clear reason

build_execution_decision_summary with previous_target_exposure_krw from resolved target
```

Do not make missing target state automatically mean target=0 when broker has executable BTC. That is the bug.

### 4. Keep target changes strategy-driven

Target should change to zero only when:

```text
raw_signal == SELL
```

or when an explicit operator closeout command is executed.

HOLD should maintain target.

BUY should set target to configured exposure.

Startup adoption should set target to current exposure, not zero.

### 5. Separate explicit closeout command

If the repo already has flatten/closeout commands, clarify that they are operator closeout operations.

If needed, add a new explicit command such as:

```text
bithumb-bot target-closeout
```

or wire existing flatten/closeout behavior so that:

```text
operator closeout:
  persists target=0 with target_origin=operator_closeout
  may execute target_delta SELL if armed
```

Normal startup must not do this.

## Acceptance criteria

This task is complete only if:

```text
1. Missing target state no longer implies target=0 when executable broker BTC exists.
2. Existing executable broker BTC is adopted as current target on safe startup.
3. Startup adoption never submits an order.
4. HOLD after adoption never closes the position.
5. Strategy SELL after adoption sets target=0 and may sell.
6. Strategy BUY after adoption adjusts target to configured exposure and buys only delta.
7. Explicit closeout remains possible but is separate from startup adoption.
8. target_delta remains the execution model for delta orders.
9. lot-native remains available for accounting/audit/PnL/recovery diagnostics, not primary execution truth in target_delta mode.
10. true_dust means only below exchange minimum delta.
11. Tests prove the previous accidental closeout cannot happen again.
12. Existing live submit and recovery tests still pass.
```

## Non-goals

Do not add another residual sell mode.

Do not add another dust class as the main fix.

Do not remove lot-native accounting.

Do not remove recovery gates.

Do not make broker balance blindly authoritative without convergence checks.

Do not force target=0 on startup just because target state is missing.

Do not treat startup adoption as strategy BUY.

Do not treat startup adoption as strategy SELL.

Do not close positions unless strategy SELL or explicit operator closeout says to.
