## Required implementation goals

### 1) Introduce a canonical fill-level incident verdict or equivalent shared projection

The same fill should not be classified differently by:

* fee-pending repair preview,
* accounting replay / audit-ledger,
* and operator surfaces.

If a fill already exists in `fills` with final fee applied, then a matching stale `broker_fill_observations.accounting_status='fee_pending'` must not continue to count as an unresolved active fee issue.

### 2) Separate raw evidence from active unresolved incident state

Keep raw diagnostic events for traceability, but do not let raw append-only rows directly define active unresolved state.

In particular:

* append-only `broker_fill_observations` should remain available as evidence
* unresolved fee-pending classification should come from canonical incident state, not raw row presence alone
* “already accounted but stale observation exists” should become a first-class representable state if needed

### 3) Reuse the same matching logic everywhere

The matcher currently used by repair preview (`fill_id`, or fallback on `client_order_id + fill_ts + price + qty`) should likely be promoted into shared logic and reused by:

* unresolved fee-pending classification
* audit/audit-ledger summaries
* any fill-observation incident projection
* possibly health/recovery surfaces if they surface related signals

Do not let repair preview and audit replay use different notions of “same accounted fill.”

### 4) Preserve live safety while improving observability

Do not weaken resume gates.

If the repo intentionally distinguishes:

* operational readiness
* versus accounting diagnostics

then encode that distinction explicitly in surface outputs and tests, so operators and LLMs can tell:

* “active blocker”
* from “historical/stale evidence”
* from “already-accounted observation mismatch”
