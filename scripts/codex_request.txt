Continue working directly in the GitHub repository `WG-14/bithumb-bot`.

Do not treat this as a generic cleanup, a profitability task, or a new strategy feature. The research system already has the right broad direction: manifest-based research, deterministic artifacts, rolling walk-forward, parameter stability scoring, strategy registry, promotion artifacts, and a research-to-paper runbook.

Now strengthen the promotion step so a promoted candidate proves both:

```text
backtest / OOS validation evidence
+ rolling walk-forward validation evidence
```

as one bound validation bundle.

The core invariant is:

```text
A promoted research candidate must be the same strategy candidate that passed backtest/OOS validation and rolling walk-forward validation, with both evidence artifacts hash-verified and explicitly recorded in the promotion artifact.
```

## Current concern

Inspect:

```text
src/bithumb_bot/research/promotion_gate.py
tests/test_research_promotion_gate.py
tests/test_research_walk_forward.py
```

Current risk:

```text
promote_candidate finds the candidate in backtest_report.json
→ if walk_forward_required is true, it calls _walk_forward_candidate_for_promotion(...)
→ the walk-forward candidate is returned and effectively replaces the backtest candidate
→ promotion evaluation then happens primarily on the walk-forward candidate
```

This is not strong enough for auditability.

Promotion must not mean “the walk-forward candidate passed.” It must mean:

```text
the backtest/OOS candidate passed
AND its candidate_profile_hash is valid
AND the walk-forward candidate for the same experiment/manifest/parameters/cost profile passed
AND its candidate_profile_hash is valid
AND both evidence sources are bound into the promotion artifact
```

## Preserve priority order

Use this priority order:

```text
P0: Live safety, correctness, live/paper isolation, wrong-order prevention, duplicate-order prevention, ledger consistency, restart/recovery behavior
P1: Reproducibility, deterministic artifacts, promotion evidence binding, path policy, operator observability, recoverability
P2: Strategy tuning, profitability, parameter search convenience
```

Any P2 change that weakens P0 or P1 is unacceptable.

## Required design change

Refactor promotion validation into explicit stages.

Add or adapt helper functions similar to:

```python
validate_backtest_candidate_for_promotion(...)

```

The exact names may differ, but the behavior must be clear and tested.

## Backtest candidate validation requirements

The backtest candidate from:

```text
DATA_ROOT/<mode>/reports/research/<experiment_id>/backtest_report.json
```

must pass all of the following before walk-forward evidence can replace or supplement anything:

```text
candidate exists
acceptance_gate_result == "PASS"
validation_metrics exists and is a dict
candidate_profile_hash exists
sha256_prefixed(build_candidate_profile(backtest_candidate)) == backtest_candidate["candidate_profile_hash"]
```

If any of these fail, refuse promotion before using walk-forward evidence.

Use explicit reasons:

```text
backtest_candidate_not_found
backtest_acceptance_gate_not_passed
backtest_validation_oos_evidence_missing
backtest_validation_trade_count_missing
backtest_candidate_profile_hash_missing
backtest_candidate_profile_hash_mismatch
```

If existing reason names differ, keep backward compatibility where practical, but make the new source-specific reasons visible.

## Walk-forward candidate validation requirements

If `backtest_candidate["walk_forward_required"]` is true, then the walk-forward candidate from:

```text
DATA_ROOT/<mode>/reports/research/<experiment_id>/walk_forward_report.json
```

must pass all of the following:

```text
walk_forward_report.json exists
walk_forward report experiment_id matches
walk_forward report contains candidates
candidate exists
strategy_name matches backtest candidate
parameter_candidate_id matches backtest candidate
parameter_values matches backtest candidate
cost_model matches backtest candidate
manifest_hash matches backtest candidate
walk_forward_gate_result == "PASS"
walk_forward_metrics exists and is a dict
candidate_profile_hash exists
sha256_prefixed(build_candidate_profile(walk_forward_candidate)) == walk_forward_candidate["candidate_profile_hash"]
```

Use explicit refusal reasons:

```text
walk_forward_missing
walk_forward_report_experiment_id_mismatch
walk_forward_report_candidates_missing
walk_forward_candidate_mismatch
walk_forward_gate_not_passed
walk_forward_metrics_missing
walk_forward_candidate_profile_hash_missing
walk_forward_candidate_profile_hash_mismatch
```

Do not let a passing walk-forward candidate rescue a failed or tampered backtest candidate.

## Promotion artifact requirements

When promotion succeeds, the artifact must explicitly record both evidence sources.

At minimum include:

```json
{
  "strategy_name": "...",
  "strategy_profile_id": "...",
  "strategy_profile_source_experiment": "...",
  "strategy_profile_hash": "sha256:...",
  "candidate_id": "...",
  "manifest_hash": "sha256:...",
  "dataset_snapshot_id": "...",
  "dataset_content_hash": "sha256:...",
  "candidate_profile": {},
  "gate_result": "PASS",

  "validation_evidence_source": "backtest_report.json",
  "backtest_candidate_profile_hash": "sha256:...",
  "backtest_candidate_profile_verified": true,

  "walk_forward_required": true,
  "walk_forward_evidence_source": "walk_forward_report.json",
  "walk_forward_candidate_profile_hash": "sha256:...",
  "walk_forward_candidate_profile_verified": true,

  "operator_next_step": "Review this artifact before manual paper env/profile consideration."
}
```

If walk-forward is not required, include:

```json
{
  "walk_forward_required": false,
  "walk_forward_evidence_source": null,
  "walk_forward_candidate_profile_hash": null,
  "walk_forward_candidate_profile_verified": false
}
```

Keep deterministic `content_hash` behavior. `generated_at` must not make the deterministic content hash unstable if the existing hash helper excludes it.

Do not write env files. Do not mutate live or paper runtime config.

## Required tests

Add or update focused tests in:

```text
tests/test_research_promotion_gate.py
```

Required test cases:

```text
test_promotion_refuses_backtest_candidate_hash_mismatch_even_when_walk_forward_exists
test_promotion_refuses_backtest_gate_failure_even_when_walk_forward_passes
test_promotion_refuses_walk_forward_candidate_profile_hash_mismatch
test_promotion_artifact_records_backtest_and_walk_forward_evidence_hashes
test_promotion_artifact_records_no_walk_forward_evidence_when_not_required
```

Test expectations:

1. If the backtest candidate hash is tampered but the walk-forward report has a passing matching candidate, promotion must fail with `backtest_candidate_profile_hash_mismatch`.

2. If the backtest candidate has `acceptance_gate_result="FAIL"` but the walk-forward candidate passes, promotion must fail with `backtest_acceptance_gate_not_passed`.

3. If the walk-forward candidate hash is tampered, promotion must fail with `walk_forward_candidate_profile_hash_mismatch`.

4. If both backtest and walk-forward evidence pass, the promotion artifact must include:

   * `validation_evidence_source`
   * `backtest_candidate_profile_hash`
   * `backtest_candidate_profile_verified`
   * `walk_forward_required`
   * `walk_forward_evidence_source`
   * `walk_forward_candidate_profile_hash`
   * `walk_forward_candidate_profile_verified`

5. If walk-forward is not required, promotion should still pass for a valid backtest candidate and explicitly record that walk-forward evidence was not required.
