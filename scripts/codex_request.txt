## Incident summary to reproduce and keep in scope

This produced a `2.000000 KRW` cash delta and a startup gate blocker around fee/accounting debt. The suspected symptom is that the small fill’s 2.00 KRW fee was not applied even though order-level `paid_fee=27.43` was available.

## Core problem statement to preserve

Do not reduce this to “add the missing 2 KRW fee.” The structural concern is:

```text
The system appears to aggregate fee as a numeric value while losing fee authority, validation state, and provenance across the accounting boundary. A temporary or non-authoritative fee value such as 0.0 can therefore be treated as a real zero fee and merged into a canonical aggregate fill/trade.
```

The fix should prevent the whole class of failures:

```text
partial fills
order-level-only paid_fee
missing trade-level fee
late fee finalization
ambiguous zero-fee assumptions
aggregate projection drift
manual repair loops
startup gate blocked by accounting debt that was actually resolvable from retained evidence
```

## Required design direction

Prefer design changes that prevent the entire problem family.

### 1. Treat fee evidence as stateful evidence, not just a float

A fill fee must carry at least:

```text
fee value
fee_status
fee_source
fee_confidence
fee_provenance
fee_validation_reason
fee_validation_checks
accounting_status / accounting decision
```

Do not allow a non-authoritative `0.0` to cross into canonical accounting as a real finalized zero fee.

### 2. Apply order-level paid_fee allocation whenever current fill fee evidence is not accounting-complete

The allocation path should not be gated only on `fee is None`.

Expected policy:

```python
if multi_fill_order_level_allocation is not None:
    allocated = allocation.evaluations_by_fill_id.get(fill_id)
    current = classify_fee_evaluation(...)
    if allocated is not None and current.accounting_status != "accounting_complete":
        use allocated
```

This must include cases such as:

```text
assumed_zero_non_material
missing
empty
zero_reported
unparseable
invalid
order_level_candidate
fee_pending
```

But do not blindly override truly authoritative trade-level fees. If trade-level fee is already accounting-complete and does not violate order-level invariants, preserve it.

### 3. Redefine or constrain `assumed_zero_non_material`

`assumed_zero_non_material` must not mean “authoritative real zero fee.”

Safer options:

```text
fee=None, fee_status=assumed_zero_non_material, accounting_status=fee_pending
```

or, if `fee=0.0` remains for backward compatibility:

```text
fee=0.0 with explicit non_authoritative_assumption authority that aggregation/apply cannot finalize
```

Choose the least invasive approach that protects accounting boundaries and passes existing tests.

### 4. Aggregation must preserve or respect fee authority

If `_aggregate_fills_for_apply()` remains, it must not aggregate away fee authority. At minimum:

```text
If any component fill is fee_pending / not accounting_complete:
  - do not produce an aggregate fill that appears fee_finalized/complete/authoritative
  - either block strict aggregation and transition to recovery
  - or return an aggregate pending-principal/pending-fee representation that downstream apply preserves
```

Do not let `25.43 + non-authoritative 0.0` become a finalized aggregate fee of `25.43`.

Longer-term, prefer accounting from exchange fill-level postings and treat aggregate fills as reporting/projection only.

### 5. Add invariants around order-level fee allocation

When order-level `paid_fee` exists and complete fill set is available, enforce or report:

```text
sum(fill.qty) == order.executed_volume within tolerance
sum(fill.funds) == order.executed_funds within tolerance
sum(applied_fill_fee) == order.paid_fee within KRW fee rounding tolerance
no canonical finalized fill/trade if fee evidence is incomplete
principal must not be double-applied during recovery/replay
```

If invariant fails, emit a structured diagnostic and block trading safely.
