## Incident context

On EC2, live trading repeatedly entered this failure cycle after real Bithumb BUY/SELL fills:

```text
Bithumb order fills successfully
→ local fee validation classifies the fill as fee_pending
→ local fills/trades/portfolio/lot accounting becomes incomplete or inconsistent
→ broker/local portfolio mismatch appears
→ startup/resume gates block live
→ recovery suggests flatten or manual repair
→ flatten itself creates another real fill
→ that fill goes through the same fee/accounting failure path
→ operator manually repairs fee/principal/lot state and sometimes bot_health
```

Observed runtime facts from the incident:

```text
Actual Bithumb mobile-app fee rate: 0.04% = 0.0004 = ~4 bps

DB-observed examples:
flatten SELL fee_bps = 3.999784
latest BUY fee_bps   = 3.999902
previous SELL        = 3.998178
previous BUY         = 3.999148

Resolved setting during the incident:
LIVE_FEE_RATE_ESTIMATE = 0.0025
LIVE_FILL_FEE_ALERT_MIN_NOTIONAL_KRW = 10000.0
```

The likely direct trigger was:

```text
configured expected live fee rate = 0.0025
actual Bithumb paid fee rate      = 0.0004
→ expected_fee_rate_mismatch
→ order_level_candidate
→ fee_pending
→ accounting/recovery cascade
```

But this must not be reduced to “just set LIVE_FEE_RATE_ESTIMATE=0.0004.” That is operational hygiene, not a sufficient repository-level fix.

The higher-priority structural issue is:

```text
The system must not let a local estimated fee rate override coherent exchange-provided paid_fee evidence as a live accounting gate.

For a single-fill order where exchange order-level paid_fee is present, non-negative, non-zero for material notional, and the executed volume/funds/identifiers match, the fill should be accounting-complete even if the configured fee estimate is wrong.

The expected fee rate mismatch must remain visible as diagnostics/warning/anomaly, but it must not by itself force fee_pending or destabilize portfolio/lot accounting.
```

## Required design priorities

Prioritize problem-class prevention over symptom patches.

### 1. Fee authority must be explicit and ordered correctly

If the current `FeeEvaluation` still relies on a loose combination of `fee_status`, `fee_source`, `fee_confidence`, `provenance`, and `reason`, improve it where feasible without over-scoping.

Preferred direction:

```python
fee_authority = "exchange_order_paid_fee_single_fill"
accounting_decision = "finalize"
diagnostic_flags = ["expected_fee_rate_mismatch"]
```

At minimum, make sure the semantics are unambiguous in code and tests:

```text
exchange trade-level fee                 → authoritative
exchange order-level paid_fee single-fill → authoritative/validated if qty/funds/ids match
operator_confirmed fee                   → authoritative
configured fee estimate                  → planning/diagnostic only, not live settlement authority
unknown/missing fee                      → pending/blocked depending on safety policy
```

### 2. Expected fee-rate mismatch must be diagnostics, not settlement veto

Preserve this invariant across all ingestion/application paths:

```text
single-fill exchange paid_fee + matching qty/funds/ids
→ accounting_complete

expected fee-rate mismatch
→ warning/anomaly/diagnostic
→ not fee_pending by itself
```

Unsafe cases must remain pending/blocked:

```text
missing/unparseable paid_fee on material notional
negative fee
material zero paid_fee
multi-fill order-level fee ambiguity
executed_volume mismatch
executed_funds mismatch
identifier mismatch, if current policy requires identifiers
```

### 3. Live fee-rate drift must be observable

Even after the fee-authority fix, `LIVE_FEE_RATE_ESTIMATE=0.0025` while actual fees are `0.0004` is still operationally dangerous for strategy planning, cost estimates, and diagnostics.

Add or improve operator-facing output in `fee-diagnostics`, `health`, `recovery-report`, and/or `restart-checklist` so the operator can see:

```text
configured_fee_rate
configured_fee_bps
observed_fee_bps_median_10 or similar
observed_fee_sample_count
fee_rate_deviation_pct
expected_fee_rate_warning_count
fee_pending_count
fee_pending_accounting_repair_count
position_authority_repair_count
whether drift is only diagnostic or startup-blocking
```

If a hard startup block would be too disruptive, use a strong warning. But do not keep this invisible.

### 4. Recovery must not recommend flatten as a default response to accounting-root-cause incidents

The incident showed that flatten can make accounting incidents worse by creating another real fill that goes through the same broken fee path.

Modify recovery/reporting logic if needed so that when the dominant issue is unresolved fee/accounting authority drift, the primary recommendation is forensic diagnosis or repair-plan generation, not flatten.

Desired policy:

```text
If accounting/fee-authority root cause is unresolved:
  - suppress flatten as the primary recommendation
  - prevent new orders unless safety gates explicitly allow
  - recommend forensic fee/accounting diagnosis
  - collect broker order/fill evidence
  - propose auditable repair plan if needed

If actual executable market exposure exists and accounting evidence is reliable:
  - flatten may be recommended as market-risk mitigation
```

### 5. Repairability must move away from manual DB edits

Do not add another opaque manual DB workaround.

If repair behavior needs improvement, make it auditable and idempotent. Prefer moving toward:

```bash
bithumb-bot doctor --live --explain
bithumb-bot repair-plan --live
bithumb-bot repair-apply --plan-id ...
```

A full repair-plan engine may be too large for this task. But any repair-related change should move in that direction.

Each repair plan or preview should expose:

```text
preconditions
tables touched
before/after portfolio
before/after lot projection
idempotency key
rollback/backup path
expected audit-ledger result
why it is safe
```

### 6. Treat open_position_lots as a rebuildable projection, not unquestioned truth

Do not fully rewrite lot accounting unless necessary. But verify that `open_position_lots` is treated as a deterministic materialized projection over canonical accounting evidence where possible.

Desired direction:

```text
canonical truth:
  fills + trades + fee finalization/adjustment events + external adjustments

projection:
  open_position_lots, rebuildable deterministically under safe preconditions
```

Make sure the fee-authority fix does not leave a path where one fee warning can drift lot projection and force repeated `rebuild-position-authority`.
