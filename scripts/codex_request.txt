## Root-cause hypotheses to preserve and test

Keep hypotheses as hypotheses until verified. For each, either verify, falsify, or mark inconclusive with evidence.

### Hypothesis A: target-delta execution does not receive authoritative order rules

`build_execution_decision_summary()` likely passes order rules from loose payload keys only. If the live run loop does not place `min_qty` or `residual_proof_min_qty` at top level, target-delta blocks with `missing_order_rule_min_qty` even though env/DB/live order rules exist.

Verification:

* Find all `build_target_position_decision(...)` call sites.
* Inspect what is passed to `order_rules`.
* Confirm whether run loop payloads are expected to contain `min_qty` / `residual_proof_min_qty`.
* Add a regression test reproducing the EC2 condition: `EXECUTION_ENGINE=target_delta`, flat broker qty, BUY signal, target exposure 70,000 KRW, market price around 113,428,000, no `min_qty` in payload, settings/env contain min qty and min notional.
* The current code should fail or demonstrate the defect before the fix.

Falsification condition:

* If authoritative rules are already injected into target-delta in all live paths, then the bug is in serialization, `_as_float`, stale state, or a different call path.

### Hypothesis B: the repo lacks a single authoritative order-rule domain object

Order-rule values appear in env/config, DB snapshots, live chance refresh, decision payloads, residual proof payloads, and diagnostics. The system may be relying on implicit dict keys instead of a typed execution rule authority.

Verification:

* List order-rule sources and consumers.
* Check whether any single function/object is responsible for merging, validating, annotating freshness, and delivering rules to execution engines.
* Check whether ops-report and execution use the same object or divergent sources.

Falsification condition:

* If a single order-rule authority already exists but is not wired into target-delta, scope the fix to wiring plus tests.

### Hypothesis C: tests validate target-position math but not the live integration path

Existing tests may call `build_target_position_decision()` with explicit `_rules()` or add `residual_proof_min_qty` into readiness payload. That validates pure target math, but not the actual run-loop/execution-service path where rules may be absent from payload.

Verification:

* Inspect target-position and execution-service tests.
* Identify missing integration tests that would have caught EC2’s `target_order_rule_min_qty=null` condition.
* Add those tests.

### Hypothesis D: operational reporting separates “tradeability allowed” from “actual submit proof failed”

Ops reports can show `new_entry_allowed=1`, `why_not=none`, while latest decisions show `BLOCK_TARGET_DELTA` due to missing order rules. This makes operator diagnosis misleading.

Verification:

* Inspect ops-report/recovery-report output generation.
* Check whether top-level reports surface:

  * `target_delta_submit_allowed`
  * `target_delta_block_reason`
  * `target_order_rule_min_qty`
  * `order_rule_authority_source`
  * mismatch between snapshot rules and target-delta rules.

### Hypothesis E: fee/accounting issues are a separate recurrent failure class, not the active cause of this incident

Historical logs showed `LiveFillFeeValidationError`, `FillFeeStrictModeError`, `fee_status=order_level_candidate/missing`, and fee-pending repairs. Current diagnostics reportedly showed active fee issue counts at zero. Treat fee/accounting as a separate structural risk, not the direct current cause unless repo evidence contradicts this.

Verification:

* Keep the order-rule fix separate from fee/accounting fixes.
* Do not conflate pre-submit target-delta proof failure with post-fill accounting failure.

## Implementation priorities

Prioritize design changes that prevent the whole class of bug over one-off symptom suppression.

### Priority 1: restore safe operation with a minimal but defensible fix

Implement a fix so target-delta execution receives non-null order rules when they are available from a trusted source.

Acceptable minimum:

* In `execution_service.py`, do not rely only on `payload.get("min_qty")` / `payload.get("residual_proof_min_qty")`.
* Add fallback from settings/config values such as:

  * `settings.LIVE_MIN_ORDER_QTY`
  * `settings.MIN_ORDER_NOTIONAL_KRW`
  * `settings.LIVE_ORDER_QTY_STEP`
* Preserve fail-closed behavior if required execution rules are still unavailable.
* Do not silently submit with invalid or zero min rules.

Better:

* Introduce a small typed helper or function that resolves execution order rules from payload plus settings, with source annotation.
* Use that helper in the target-delta call path.
* Include the resolved values in target decision audit fields.

Do not simply remove `missing_order_rule_min_qty` checks.

### Priority 2: introduce an explicit execution order-rule abstraction

If feasible within the change, introduce a typed model such as `ExecutionOrderRules` or `OrderRuleAuthority`. It should centralize at least:

* `market`
* `min_qty`
* `qty_step`
* `min_notional_krw`
* optional `bid_min_total_krw`
* supported bid/ask order types if available
* source/source_mode
* stale/degraded/fallback flags if available

It must have validation for target-delta and live submission.

Target direction:

* Execution engines should not infer execution-critical order rules from arbitrary `decision_context` dicts.
* `decision_context` should be treated as audit/observability, not as the authoritative execution input.
* If a full refactor is too large, create a clearly named interim resolver, with tests and TODOs pointing toward full `OrderRuleAuthority`.

### Priority 3: add regression tests that reproduce the EC2 failure

Add or update tests so this cannot regress.

Required tests:

1. EC2 reproduction test:

   * `EXECUTION_ENGINE=target_delta`
   * BUY signal
   * flat broker qty
   * target exposure 70,000 KRW
   * market price around 113,428,000
   * readiness payload does not include `min_qty` or `residual_proof_min_qty`
   * settings contain min qty and min notional
   * expected: target-delta receives `target_order_rule_min_qty=0.0001`, `target_order_rule_min_notional_krw=5000`, `would_submit=True`, `target_delta_side=BUY`, no `missing_order_rule_min_qty`.

2. Fail-closed test:

   * No payload rules and no settings/default rules available.
   * Expected: no submit, explicit block reason for missing order rule, no accidental order.

3. Source/audit test:

   * Target decision/audit output includes the rule values used and preferably the source (`payload`, `settings`, `snapshot`, etc.).

4. Ops/reporting test if report code is changed:

   * Report must surface the difference between general entry gate and actual target-delta submit proof.
   * A case with `new_entry_allowed=1` but `target_delta_submit_allowed=0` must be visible and actionable.

### Priority 4: improve observability

Add or update reporting so the operator can immediately distinguish:

* strategy did not signal BUY
* strategy signaled BUY but entry gate blocked
* target-delta proof blocked
* order-rule authority missing/stale
* broker submit failed
* broker accepted but fill accounting blocked
* active recovery/fee issue blocked

At minimum, include in ops-report/recovery-report or latest strategy flow:

```text
target_delta_submit_allowed=<0|1>
target_delta_block_reason=<reason>
target_order_rule_min_qty=<value|null>
target_order_rule_min_notional_krw=<value|null>
order_rule_authority_source=<source>
order_rule_snapshot_min_qty=<value|null>
order_rule_settings_min_qty=<value|null>
entry_gate_allowed=<0|1>
new_entry_allowed=<0|1>
```

If there is a mismatch such as snapshot/settings have min_qty but target-delta has null, emit a clear diagnostic like:

```text
diagnosis=order_rule_not_propagated_to_target_delta
```

### Priority 5: keep fee/accounting as a separate follow-up area

Do not solve fee/accounting in the same patch unless required. But add comments or issue notes if you find the reporting conflates it with pre-submit blocks.

Longer-term recommendation to include in the final report:

* Separate principal quantity accounting from fee finalization.
* Model fee status as a state machine: observed → candidate → estimated → authoritative → repaired.
* Allow policy-based handling of `order_level_candidate`.
* Surface fee pending as post-fill accounting, not as submit failure.
