
Do not weaken the lot-native strategy SELL authority.

Do not make `dust_tracking_qty`, aggregate residual quantity, or residual lots pretend to be canonical strategy executable lots.

Keep these separate:

```text
strategy_sell_candidate:
  authority = lot-native strategy position
  source = sellable_executable_lot_count / sellable_executable_qty
  used for normal strategy exits

residual_sell_candidate:
  authority = residual inventory policy
  source = residual inventory model
  used only through explicit residual policy + final pre-submit proof
  must use standard order submission, OMS dedup, order recording, and recovery semantics
```

Do not manually flatten the live account as a “fix.” Do not special-case `0.00049980 BTC`.

### H1: residual SELL enabled path bypasses the standard live submit pipeline

Hypothesis:

```text
LiveSignalExecutionService.execute() handles residual SELL by directly calling:

  self.broker.place_order(...)

instead of using the existing standard live submit pipeline / OMS / order events / recovery machinery.
```

Why this matters:

```text
Direct broker.place_order risks bypassing:
- orders table PENDING_SUBMIT/NEW/FILLED lifecycle
- order_events / submit_attempt recording
- order_intent_dedup
- submit_unknown / timeout / recovery_required handling
- standard fill application / ledger accounting
- existing live submit contract/profile logic
```

Verify:

* Trace residual SELL `enabled` mode from `ExecutionSubmitPlan` to actual broker call.
* Compare it against normal live BUY/SELL submission in `broker/live.py`, including `run_standard_submit_pipeline`, `execute_live_submission_and_application`, `record_order_if_missing`, `claim_order_intent_dedup`, `record_status_transition`, and related helpers.

If confirmed, fix it. Residual SELL must not call low-level `broker.place_order` directly except through the same standard pipeline abstraction used by normal live orders.

### H2: residual SELL idempotency is not integrated with OMS dedup

Why this matters:

```text
A retry or repeated loop for the same SELL signal/candle could submit multiple residual SELLs if the only uniqueness control is a new UUID.
```

Verify:

* Inspect whether residual SELL enabled mode calls `build_order_intent_key`, `claim_order_intent_dedup`, and `update_order_intent_dedup`, or goes through a standard pipeline that does.
* Confirm intent fields are stable for the same residual decision.

## Required design changes

### 1. Residual SELL must use the standard live submit pipeline

Do not submit residual SELL by directly calling `broker.place_order`.

Refactor so residual SELL enabled mode uses the same standard pipeline and accounting/recovery machinery as normal live orders.

Acceptable implementation options:

```text
Option A:
  Extend the existing live submit pipeline to accept an explicit ExecutionSubmitPlan.

Option B:
  Add a residual-specific adapter that converts ExecutionSubmitPlan into the existing StandardSubmitPipelineRequest and then calls run_standard_submit_pipeline.

Option C:
  If the pipeline cannot currently accept this cleanly, extract a shared submit orchestration function used by both canonical strategy orders and residual orders.
```

The residual submit must preserve:

```text
side=SELL
qty=residual_submit_plan.qty
source=residual_inventory
authority=residual_inventory_policy
decision_reason_code=CLOSE_RESIDUAL_CANDIDATE
intent_type=residual_close or residual_sell
strategy_context=residual_inventory_policy
```

The residual path must use or preserve existing behavior for:

```text
- order intent dedup
- client_order_id length/shape rules
- PENDING_SUBMIT / NEW / PARTIAL / FILLED / FAILED / SUBMIT_UNKNOWN / RECOVERY_REQUIRED transitions
- submit_attempt / order_events recording
- timeout/ambiguous submit handling
- fill application / ledger accounting
- broker response confirmation
- order-rule min qty / min notional / qty step / decimals normalization
- live dry-run and armed checks
```

Do not duplicate a parallel mini submit pipeline if the standard one exists.

### 2. Residual SELL idempotency must be stable and tested

Add or reuse a stable intent key for residual SELL:

```text
intent_type=residual_close or residual_sell
symbol=settings.PAIR
side=SELL
intent_ts=decision ts or candle ts
strategy_context=residual_inventory_policy
qty=residual_sell_candidate.qty
source=residual_inventory
authority=residual_inventory_policy
```

The same residual decision should not create a second live order if retried while an unresolved/pending/active intent exists.

Use existing OMS primitives such as:

```text
build_order_intent_key
claim_order_intent_dedup
update_order_intent_dedup
```

or ensure the standard submit pipeline uses them.

Tests must prove duplicate residual SELL intent is suppressed or deduped.

### 3. Residual-aware BUY delta must affect actual submit sizing in delta mode

For `RESIDUAL_BUY_SIZING_MODE=delta`, wire the computed delta into actual BUY submit sizing.

Expected model:

```text
target_exposure_krw = configured strategy position notional
current_effective_exposure_krw = strategy open exposure + tracked residual exposure
buy_delta_krw = target_exposure_krw - current_effective_exposure_krw
```

Expected behavior:

```text
if buy_delta_krw <= 0:
  no BUY submit
  final_action=HOLD_TARGET_ALREADY_COVERED
  block_reason=tracked_residual_exposure_covers_target

if 0 < buy_delta_krw < min_notional_krw:
  no BUY submit
  final_action=BLOCK_ORDER_RULE
  block_reason=buy_delta_below_min_notional

if buy_delta_krw >= min_notional_krw:
  actual BUY submit notional is buy_delta_krw
  not full MAX_ORDER_KRW
```

Implementation should make the actual sizing source explicit, for example:

```text
sizing_source=residual_inventory_delta
target_exposure_krw=...
current_effective_exposure_krw=...
tracked_residual_exposure_krw=...
buy_delta_krw=...
```

This must be tested against fake broker/executor or sizing builder. Do not stop at telemetry.

### 4. Remove unsafe accounting_projection_ok fallback

In residual SELL proof:

```text
accounting_projection_ok missing -> fail closed
projection_converged=true must not imply accounting_projection_ok=true
```

Do not do:

```python
payload["accounting_projection_ok"] = bool(readiness_payload.get("projection_converged"))
```

Instead:

```text
if accounting_projection_ok not provided:
  proof fails with missing_accounting_projection_ok
```

Tests must include:

```text
projection_converged=true
accounting_projection_ok missing
=> residual pre-submit proof fails with missing_accounting_projection_ok
```

and:

```text
projection_converged=true
accounting_projection_ok=false
=> residual pre-submit proof fails with accounting_projection_not_ok
```

### 5. Make dry_run a meaningful standard-submit dry run

`RESIDUAL_LIVE_SELL_MODE=dry_run` should not submit, but it should show enough evidence that enabled mode would use the standard path.

Operator telemetry should include:

```text
residual_submit_plan.source=residual_inventory
residual_submit_plan.authority=residual_inventory_policy
residual_submit_plan.qty=...
residual_submit_plan.submit_expected=false
residual_submit_plan.block_reason=residual_live_sell_mode_dry_run
would_submit_pipeline=standard
would_intent_key=<stable or redacted>
would_client_order_id_shape=<shape>
would_order_rule_normalized_qty=<if available>
```

If this is too large, at minimum add tests proving dry_run builds the same plan as enabled mode but stops before broker dispatch and before live order mutation.
