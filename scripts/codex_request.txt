## Hypotheses to verify or falsify

### Hypothesis 1

`open_position_lots` is modeled as a rebuildable materialized projection, but in practice stale rows can persist across repairs and become current startup authority.

Verification:

* Inspect code and docs for the intended role of `open_position_lots`.
* Confirm whether startup readiness depends directly on `SUM(open_position_lots.qty_open)`.
* Determine whether there is a deterministic full rebuild path from canonical event/accounting/broker/portfolio state.
* Determine whether stale historical fragments can survive legitimate repair flows.

Falsification condition:

* There is already a correct, transactional, full-projection rebuild path that can rebuild `open_position_lots` from canonical truth and converge to portfolio/broker holdings, and the observed incident would be impossible unless EC2 was running stale code or had local-only corruption.

### Hypothesis 2

Current repair logic is too target-trade centric and does not treat historical fragmentation as a first-class incident class.

Verification:

* Inspect whether `build_position_authority_assessment()` chooses the latest BUY / target trade and classifies incidents around that target.
* Inspect whether `apply_position_authority_rebuild()` deletes/replays only rows for one `entry_trade_id` in key paths.
* Check whether `other_active_qty` and historical fragmentation are only diagnostic, or can drive a safe full rebuild action.

Falsification condition:

* The current implementation already promotes historical fragmentation to a safe full-projection rebuild mode and has tests proving it.

### Hypothesis 3

Accounting replay and lot-authority projection are separate truth layers, but there is no strong enough invariant tying them together in audit, recovery, and restart readiness.

Verification:

* Inspect `audit-ledger`, `health`, `recovery-report`, `restart-checklist`, and runtime readiness logic.
* Determine whether `audit-ledger OK` can coexist with lot projection non-convergence.
* Determine whether operator-facing reports clearly separate:

  * accounting projection health
  * broker/portfolio health
  * lot authority projection health
  * live readiness

Falsification condition:

* Audit/restart tooling already treats lot projection convergence as a first-class invariant and reports it in a way that prevents operator confusion.

### Hypothesis 4

Repair events and current-state attestations are conceptually separated in the code, but the operational flow does not enforce this distinction strongly enough.

Verification:

* Inspect `position_authority_repairs` and `position_authority_projection_publications`.
* Confirm whether a historical repair event can exist while current `open_position_lots` is non-converged.
* Determine whether post-repair state is attested and revalidated as current state, not just logged as historical repair evidence.

Falsification condition:

* Every repair event is atomically paired with current-state projection publication and postcondition validation, and no later operation can invalidate it without a clear incident class.

### Hypothesis 5

External/manual state changes are not modeled strongly enough as first-class events that atomically update both accounting and lot-authority projection.

Verification:

* Inspect manual flat repair, external position adjustment, fee pending repair, fee gap repair, and any broker reconciliation paths.
* Determine whether these flows update accounting/portfolio without always rebuilding or converging lot-authority projection.
* Determine whether a fee/manual/external repair can leave `open_position_lots` stale.

Falsification condition:

* All such flows atomically update accounting, portfolio, lot projection, repair evidence, and projection publication, with postcondition rollback on failure.

## Design priority

Do not start with a symptom patch such as “delete the 14 rows.” That is not acceptable as the primary solution.

The preferred design direction is:

> Make `open_position_lots` explicitly disposable, deterministic, and rebuildable as a materialized projection from canonical evidence plus broker/portfolio anchoring, rather than allowing it to become a long-lived independent source of truth.

Implement a structural repair capability if missing.

## Required design change

Add or refactor a safe full-projection rebuild path. The exact CLI name is up to you, but prefer something explicit such as:

```bash
MODE=live uv run bithumb-bot rebuild-position-authority --full-projection-rebuild
MODE=live uv run bithumb-bot rebuild-position-authority --full-projection-rebuild --apply --yes
```

The dry-run must produce a clear repair plan. The apply mode must be transactional.

Minimum safety gates for apply:

```text
broker_qty_known = true
broker_qty == portfolio_qty within existing asset tolerance
remote_open_order_count = 0
unresolved_open_order_count = 0
recovery_required_count = 0
pending_submit = 0
submit_unknown = 0
no unresolved fee-pending accounting issue
no active broker/portfolio mismatch
```

The implementation must:

1. Start a DB transaction.
2. Snapshot current `open_position_lots`, relevant `trade_lifecycles`, current convergence state, readiness state, broker/portfolio evidence, and repair history summary into the repair basis.
3. Rebuild `open_position_lots` as a full materialized projection, not as a target-trade-only repair.
4. Use portfolio/broker quantity as a hard anchor when safety gates prove they agree.
5. Ensure the rebuilt rows satisfy existing `open_position_lots` trigger constraints.
6. Preserve or regenerate enough lot-native authority for strategy execution and dust handling.
7. Record a `position_authority_repair` with a complete repair basis.
8. Record a `position_authority_projection_publication` as current-state attestation.
9. Run `build_lot_projection_convergence()` as a postcondition.
10. Roll back on any postcondition failure.
11. Make the operation idempotent.

A successful postcondition must include:

```text
projection_converged=true
projected_total_qty == portfolio_qty
broker_qty == portfolio_qty
open_position_lots rows satisfy trigger constraints
startup gate no longer blocked by POSITION_AUTHORITY_PROJECTION_CONVERGENCE_REQUIRED
```

If the safest design requires adding a new module instead of expanding the existing repair function, do that. Do not force the fix into the current target-trade repair flow if that makes the design weaker.
