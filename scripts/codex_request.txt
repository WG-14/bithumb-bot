Do not assume the current diagnosis is final. Treat it as a set of hypotheses to verify or falsify in the repo.

Operationally, the bot appeared “stuck” because BUY and SELL signals could both be converted to HOLD/no-submit under `dust_only_remainder`, even though the observed residual BTC appears exchange-sellable by min quantity and min notional. Also, an early stale-lock explanation was plausible from `/proc/<pid>` and lock-file observations, but systemd journal later showed the prior process was actually running until a user stop and that the bigger issue was residual-policy no-submit behavior. Do not overfit to stale lock as the primary root cause.

## Core problem statement

The suspected structural problem is:

```text
The bot has become good at detecting and safely blocking risky states, but residual inventory is not yet consistently modeled as first-class exposure/inventory through the live run loop, strategy decision, sizing, execution policy, submit path, telemetry, and tests.

As a result, a sellable residual BTC balance can be treated as dust-only/non-executable for strategy authority, blocking both entry and exit. This creates an operational deadlock even if the behavior is technically consistent with some existing conservative policy.
```

Do not “fix” this by simply merging `dust_tracking` into `open_exposure` or by making all residual balances canonical strategy SELL inventory. Preserve the lot-native safety contract: strategy SELL authority must remain based on the proper executable strategy lots. Residual inventory needs its own explicit authority, policy, candidate, and pre-submit proof.

## Hypotheses to verify or falsify

Keep these as hypotheses. Do not treat them as facts until the code and tests support them.

### H1: stale lock was an operational symptom, not the primary root cause

Hypothesis:

```text
The bot was perceived as stopped partly because file logs and systemd journal were not treated as one canonical live log source. The process may have been running and making HOLD/no-submit decisions while older file logs made it look stale. Stale lock handling may still need hardening, but it should not be over-classified as the root cause unless evidence supports that.
```

Verify in repo:

* Inspect systemd service definition, deployment docs, runbooks, lock implementation, stale lock detection, and log routing.
* Check whether systemd mode writes only to journal while file logs are still documented as canonical.
* Check whether lock cleanup on graceful stop and stale-owner detection are tested.

Implement only if needed:

* Improve runbook/docs to make canonical live log source explicit.
* Add or improve stale-lock regression tests if lock cleanup or stale-owner behavior is under-tested.
* Do not make stale lock the central fix for the residual deadlock.

### H2: residual inventory is detected/readiness-tracked but not consistently connected to live execution policy

Hypothesis:

```text
RESIDUAL_INVENTORY_MODE=track and RESIDUAL_INVENTORY_TRACKED may exist in readiness/reporting, but the live run loop and strategy decision path still convert BUY/SELL into HOLD under dust_only_remainder instead of producing explicit residual-aware execution actions.
```

Verify:

* Find where `raw_signal` becomes `final_signal`.
* Find whether `ResidualSellCandidate` or equivalent is actually created in the decision context when `raw_signal=SELL` and residual inventory is exchange-sellable.
* Find whether residual-aware BUY sizing is applied when `raw_signal=BUY`.
* Find whether telemetry/DB decision rows preserve this distinction.

Expected direction:

* `raw_signal=SELL` with no strategy lots but with exchange-sellable, explainable, broker/local/projection-converged residual should not silently collapse to generic HOLD.
* It should produce an explicit residual-aware action/candidate, even if final live submit remains gated by pre-submit proof.

### H3: `dust_tracking` semantics are too strong in execution policy

Hypothesis:

```text
A row/state used for accounting/operator evidence, such as dust_tracking, may be treated as permanently non-executable in execution policy. This is safe but can be operationally wrong when aggregate residual inventory is exchange-sellable and reconciled.
```

Verify:

* Inspect DB lot state semantics, open-position lot calculations, dust/residual classification, and position authority tests.
* Confirm the existing lot-native contract: do not mix `dust_tracking` into canonical strategy `open_exposure`.
* Determine whether there is an explicit separate residual inventory authority for sellable residuals.

Expected direction:

* Preserve strategy lot-native SELL authority.
* Add or complete separate `residual_sell_candidate` / residual authority path instead of changing strategy lot authority.
* Rename or split reason codes so exchange-sellable residual is not reported as generic `dust_only_remainder`.

### H4: BUY/SELL are still treated too much like order commands instead of target exposure transitions

Hypothesis:

```text
The current engine may interpret BUY as “open a new position” and SELL as “close canonical strategy lots”. This makes the system fragile to partial fills, fees, step rounding, and residual inventory.
```

Verify:

* Inspect sizing logic.
* Look for target exposure, current effective exposure, residual-adjusted delta sizing.
* Check whether residual inventory contributes to current exposure under track mode.

Expected direction:

* Move toward this model, or prepare an incremental step toward it:

```text
StrategySignal -> TargetExposure
TargetExposure + CurrentEffectiveExposure -> ExecutionDelta
ExecutionDelta + OrderRules + RiskPolicy -> SubmitPlan
SubmitPlan + PreSubmitProof -> BrokerOrder
```

Incremental acceptable outcome:

* Under `RESIDUAL_INVENTORY_MODE=track`, BUY sizing should account for tracked residual exposure and not blindly block or blindly overbuy.
* SELL should target zero exposure and consider both strategy sell candidates and residual sell candidates through separate authorities.

### H5: readiness, position truth, residual inventory, execution policy, and order submission are not cleanly separated

Hypothesis:

```text
Runtime readiness/reporting, residual classification, strategy signal handling, execution decision, and broker submit proof are too intertwined. This causes final_signal=HOLD to hide materially different cases: strategic HOLD, blocked recovery, unsellable dust, sellable residual pending policy, and actual submit failure.
```

### H6: observability is insufficient for live operations

Hypothesis:


## Design requirements

Prioritize design changes that prevent the whole problem family over one-off symptom patches.

### Requirement 1: Do not weaken the lot-native safety contract

Do not make `dust_tracking` or aggregate residual quantity canonical strategy SELL inventory.

Keep separate:

```text
strategy_sell_candidate:
  source = canonical executable strategy lots
  authority = lot-native strategy position

residual_sell_candidate:
  source = residual inventory model
  authority = residual inventory policy
  requires final pre-submit proof
```

### Requirement 2: Introduce or complete explicit execution actions

Do not collapse all no-submit outcomes into `final_signal=HOLD`.

Add or complete fields/concepts equivalent to:

```text
raw_signal
position_truth_state
residual_inventory_state
strategy_sell_candidate
residual_sell_candidate
target_exposure_krw
current_effective_exposure_krw
final_action
submit_expected
pre_submit_proof_status
block_reason
```

At minimum, distinguish:

```text
STRATEGY_HOLD
ENTER_STRATEGY_POSITION
EXIT_STRATEGY_POSITION
CLOSE_RESIDUAL_CANDIDATE
HOLD_TRACKED_DUST
BLOCK_RECOVERY
BLOCK_UNRESOLVED_RESIDUAL
BLOCK_ORDER_RULE
```

Use the repo’s existing naming style if different.

### Requirement 3: Split residual taxonomy

Do not use a single dust-only state for all residual inventory.

Represent at least these distinctions, using repo-appropriate names:

```text
UNSELLABLE_DUST:
  qty < min_qty OR notional < min_notional
  no sell candidate

SELLABLE_RESIDUAL:
  qty >= min_qty
  notional >= min_notional
  broker/local/projection converged
  no unresolved open orders
  no active submit_unknown/recovery blockers
  fresh broker evidence
  residual policy allows tracking
  residual sell candidate can be built

TRACKED_RESIDUAL_EXPOSURE:
  residual contributes to current effective exposure under track mode

UNRESOLVED_RESIDUAL:
  broker/local/projection mismatch, stale evidence, open order, fee/accounting blocker, or submit_unknown
  fail closed
```

### Requirement 4: Residual-aware BUY sizing

Under `RESIDUAL_INVENTORY_MODE=track`, tracked residual inventory should count toward current effective exposure.

Expected behavior:

```text
BUY signal:
  target_exposure_krw = configured strategy position notional
  current_effective_exposure_krw includes tracked residual notional
  buy_delta_krw = target_exposure_krw - current_effective_exposure_krw

If buy_delta is below min_notional or <= 0:
  no BUY submit
  final_action should explain insufficient delta / residual already covers target exposure

If buy_delta is valid:
  submit only the delta, subject to existing risk/order-rule gates
```

If the repo already has a partial implementation, complete it and add tests. If this is too large for one patch, implement the telemetry and classification first, then isolate the sizing change behind a clearly named policy flag. Do not leave the deadlock silent.

### Requirement 5: Residual-aware SELL policy

Under `RESIDUAL_INVENTORY_MODE=track`, when a SELL signal occurs and canonical strategy lots are absent but exchange-sellable residual inventory exists:

```text
raw_signal=SELL
strategy_sell_candidate=absent
residual_sell_candidate=present
final_action=CLOSE_RESIDUAL_CANDIDATE or equivalent
submit_expected=true only if final pre-submit proof passes
```

Final live submit must remain guarded by hard pre-submit proof.

Required proof checks:

```text
fresh broker snapshot
broker_qty >= residual sell qty
broker_qty == portfolio_qty == projected_total_qty within configured tolerance
projection_converged=true
accounting_projection_ok=true
open_order_count=0
unresolved_open_order_count=0
recovery_required_count=0
submit_unknown_count=0
active fee/accounting blocker=false
exchange_sellable=true
qty >= min_qty
notional >= min_notional
idempotency key / client order id uniqueness is preserved
```

If proof fails:

```text
final_action=BLOCK_UNRESOLVED_RESIDUAL or equivalent
submit_expected=false
block_reason must identify the failed proof condition
```
