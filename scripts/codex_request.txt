### H2: Typed evidence numeric validation does not consistently reject non-finite values with `EvidenceValidationError`

Inspect `evidence_chain.py`.

The likely issue is:

* `_number()` may reject `NaN` via `parsed != parsed`.
* `_number()` may not reject `inf` or `-inf`.
* `compute_evidence_content_hash()` / `validate_evidence_content_hash()` may fail on non-finite floats because canonical JSON uses `allow_nan=False`, but that failure may surface as a generic `ValueError`, not a stable `EvidenceValidationError`.
* Operator-facing profile promotion failures may therefore be less recoverable and less reason-coded than the rest of the evidence contract.

### H3: String forms like `"Infinity"` may bypass JSON strictness and only fail if `_number()` is finite-aware

Python `json.dumps(..., allow_nan=False)` rejects non-finite float values, but a string such as `"Infinity"` is valid JSON. If `_number("Infinity", ...)` converts to `float("inf")`, it must be rejected as non-finite with the supplied schema reason.

### H4: Current tests may cover strict JSON in research artifacts but not typed paper/live evidence validation

Inspect tests. If no test pins `validate_profile_transition_evidence()` behavior for non-finite evidence fields, add one.

## Design goal

Do not add a one-off check only for `fee_drag_ratio`.

Patch the typed evidence validation boundary so all numeric fields parsed through the evidence schema are finite and reason-coded. This should cover:

* `gross_pnl`
* `fee_total`
* `net_pnl`
* `expectancy_per_trade`
* `profit_factor`
* `fee_drag_ratio`
* observation duration
* thresholds
* any other numeric field passed through `_number()`

The solution should be small but structural:

1. `_number()` must reject `NaN`, `Infinity`, and `-Infinity`.
2. Content-hash validation should convert canonical JSON non-finite failures into `EvidenceValidationError` with a stable reason.
3. Tests should prove both raw float non-finite values and string non-finite values fail with operator-readable reason codes.

## Implementation requirements

### 1. Harden `_number()`

In `src/bithumb_bot/evidence_chain.py`, update `_number()` to use `math.isfinite()`.

Expected shape:

```python
import math

def _number(value: object, reason: str) -> float:
    if isinstance(value, bool):
        raise EvidenceValidationError(reason)
    try:
        parsed = float(value)
    except (TypeError, ValueError) as exc:
        raise EvidenceValidationError(reason) from exc
    if not math.isfinite(parsed):
        raise EvidenceValidationError(reason)
    return parsed
```

Preserve existing behavior for valid integers/floats/numeric strings.

Do not accept booleans.

### 2. Harden content-hash validation boundary

Inspect `compute_evidence_content_hash()` and `validate_evidence_content_hash()`.

If `sha256_prefixed(...)` or canonical JSON serialization can raise `ValueError` on non-finite payload values, catch it and raise `EvidenceValidationError` with a stable reason.

Recommended reason code:

```text
<label>_schema_invalid:non_finite_json
```

or, if the codebase already prefers a different style, use the closest existing schema-invalid pattern. Keep it stable and test it.

Suggested shape:

```python
def validate_evidence_content_hash(payload: dict[str, Any], *, label: str) -> str:
    expected = payload.get(EVIDENCE_HASH_FIELD)
    if not isinstance(expected, str) or not expected.startswith("sha256:"):
        raise EvidenceValidationError(f"{label}_schema_invalid:content_hash_missing")
    try:
        actual = compute_evidence_content_hash(payload)
    except ValueError as exc:
        raise EvidenceValidationError(f"{label}_schema_invalid:non_finite_json") from exc
    if actual != expected:
        raise EvidenceValidationError(f"{label}_content_hash_mismatch")
    return actual
```

If `compute_evidence_content_hash()` is used elsewhere and should also fail with `EvidenceValidationError`, consider adding a helper rather than duplicating behavior. Avoid broad exception swallowing.

### 3. Preserve strict JSON behavior

Do not weaken `allow_nan=False` in hashing or storage. If anything, keep it strict.

Do not allow non-finite values into persisted artifacts just to make validation pass.
