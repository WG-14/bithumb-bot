You are working in the `WG-14/bithumb-bot` repository.

The recent patch moved in the right direction: startup position adoption policy now exists in the target-position model, and the core `resolve_startup_target_position_policy()` logic appears to be present. However, do not consider this complete until the run-loop integration is explicitly proven. The previous real live issue was not a sizing bug; it was a policy bug: missing or forced `target=0` allowed target-delta to close broker BTC even when the strategy signal was HOLD.

Your task is to verify and harden the integration so that the accidental closeout cannot happen again.

## Core problem to preserve

Do not weaken this into a generic test cleanup.

The root cause is:

```text
The repo has multiple competing meanings of position:
1. strategy position
2. lot-native position
3. broker/target position

The desired target-delta model is:
  strategy changes target
  execution converges broker current position to target
  startup adopts existing broker position unless strategy SELL or explicit closeout says otherwise
```

The key new policy must be:

```text
If target_position_state is missing and broker/local safely show executable BTC,
the bot must adopt that broker BTC as the current target and submit no order.
```

It must not do this:

```text
missing target + broker BTC + HOLD
→ target=0
→ REBALANCE_TO_TARGET
→ SELL
```

That is the accidental closeout we are preventing.


## Required verification and code review

### 1. Verify target_position.py

Confirm that `resolve_startup_target_position_policy()` behaves as follows:

```text
existing target state:
  policy_action=use_existing_target
  would_submit_on_startup=false

missing target + broker_qty=0:
  policy_action=initialize_flat_target
  target=0
  would_submit_on_startup=false

missing target + broker_qty executable:
  policy_action=adopt_existing_broker_position
  target_exposure_krw=broker_qty * reference_price
  target_qty=broker_qty
  target_origin=adopted_existing_position
  would_submit_on_startup=false

missing target + broker qty below min:
  policy_action=initialize_true_dust_flat
  target=0 or dust-flat
  would_submit_on_startup=false

unsafe readiness:
  policy_action=block_unsafe_state
  no submit
```

If this is already correct, do not rewrite it.

### 3. Verify engine.py run_loop integration

The real run-loop path must be:

```text
if EXECUTION_ENGINE=target_delta:
  load target_position_state
  if missing:
    call resolve_startup_target_position_policy()
    if safe executable broker qty:
      persist adopted target
      use adopted target exposure as previous_target_exposure_krw
  then build_execution_decision_summary(previous_target_exposure_krw=resolved_exposure)
```

Do not let the run loop build the target-delta summary from:

```text
previous_target_exposure_krw=None
```

when broker has executable BTC and startup adoption should apply.

If `_load_previous_target_exposure_for_run_loop()` is still used in the real target-delta run loop path, replace that usage with `_resolve_target_position_state_for_run_loop()`.

### 4. Verify execution_service.py after adoption

After adoption, for `raw_signal=HOLD` and `broker_qty == adopted target qty`:

Expected:

```text
final_action should not be REBALANCE_TO_TARGET
submit_expected=false
target_delta_side=NONE
target_dust_classification=true_dust or converged/noop classification, but not executable_delta SELL
no target_submit_plan with side=SELL and submit_expected=true
```

If the current logic still labels zero delta as `delta_below_exchange_min/true_dust`, that is acceptable only if it does not submit and observability is clear. Prefer a clearer classification such as `target_converged` if already consistent with the code style, but do not make a broad refactor just for naming.

## Acceptance criteria

This task is complete only if all are true:

```text
1. Missing target + safe executable broker BTC adopts existing position.
2. Startup adoption persists target_position_state.
3. Startup adoption never submits an order.
```
