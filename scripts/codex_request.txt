## Hypotheses to verify or falsify

Keep these as hypotheses unless and until you prove them in code/tests:

### Hypothesis A — triggering cause only

The missing-fee incident was the trigger, but not the deepest structural cause.

### Hypothesis B — repair path lacks convergence guarantees

`portfolio_anchored_projection_repair` or the broader `apply_position_authority_rebuild()` flow can mark a repair as safe/applicable and persist a repaired state **without proving post-repair convergence** between:

* portfolio holdings
* projected lot totals
* executable vs residue classification

### Hypothesis C — canonical state and event residue are mixed

`open_position_lots` is acting partly as canonical current state and partly as accumulated event residue/history fragments, which allows stale or duplicated residue to survive as current authority-bearing state.

### Hypothesis D — policy is not the main bug, materialization is

The tracked-dust / reentry boundary policy may be correct in isolation, but it is being fed a wrongly materialized lot state, so the policy is blocking on corrupted inputs rather than on a real canonical residue state.

### Hypothesis E — missing end-to-end incident contract

The repository has contract tests for parts of lot-native behavior, but lacks a strong end-to-end replay contract for the real incident chain:
`missing fee -> fee pending repair -> position authority repair -> resume -> running but not tradable`

### 2. Add explicit post-repair convergence invariants

If a repair path mutates persisted lot/projection state, add postconditions that verify the resulting state is semantically coherent.

At minimum verify, with safe tolerances where appropriate:

* `sum(open_position_lots.qty_open)` vs portfolio asset qty

* that “repair applied” is not treated as equivalent to “state converged”
* that a post-repair non-converged state fails closed and is surfaced as unresolved, not silently normalized into an apparently valid projection

Do not add a superficial assertion in tests only. Put the protection in the relevant runtime repair path or projector boundary where it belongs.

### 3. Prefer deterministic projection rebuild over ad hoc mutation

If the current architecture allows repair code to mutate lot rows directly in a way that can preserve stale projection residue, redesign the touched path toward:

* source-of-truth inputs: trades, fills, trade lifecycles, portfolio, and explicit repair/evidence records
* deterministic projection rebuild from truth
* replacement of persisted projection only after invariants pass

Do not rewrite the whole system if it is not necessary, but do move the changed path in this direction. The resulting design should reduce future drift, not just patch this one case.

### 4. Keep lot-native authority intact

The fix must **not**:

* restore qty-first executable authority
* merge `open_exposure` and `dust_tracking` into one executable inventory
* allow SELL or tradability authority to be recovered merely from aggregate qty
* weaken fail-closed semantics

Any compatibility/reporting fields may remain derived only.

### 5. Improve operator observability

If the code currently allows `resume` / process-running state to look healthy while tradeability is still blocked for projection-convergence reasons, improve the operator surface so that this distinction is explicit and unambiguous.

You do not need a large observability refactor, but the resulting state should clearly distinguish:

* process loop running/stopped
* resume/recovery gate blocked/unblocked
* tradability allowed/blocked
* projection converged/non-converged

Prefer changes to existing reporting/health/readiness surfaces over adding random new outputs.
