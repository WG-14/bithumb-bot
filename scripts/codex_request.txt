You are working in the GitHub repository `WG-14/bithumb-bot`.

Your task is to verify and improve the repository’s research-to-runtime promotion flow so that it becomes a more robust, reproducible, and operable automated trading experiment system. Do not treat the initial diagnosis as proven. Start from the hypotheses below, verify or falsify them directly against the repo, and implement design-level improvements where the code confirms the gap.

## Core problem statement

The repository intentionally keeps research separated from live strategy execution. This is good and must be preserved. Do **not** implement automatic propagation from research results into live trading.

The suspected structural issue is different:

> The approved configuration contract between `research → promotion artifact → paper profile → live profile → runtime execution` may not yet be a first-class, fully verifiable artifact chain.

The risk is that backtested and promoted candidate values are manually copied into paper/live runtime configuration, creating room for parameter drift, hash mismatch, missing regime policy, wrong market/interval, wrong cost assumptions, or paper/live profile confusion.

Your goal is to replace fragile manual copying with a verified manual approval flow:

```text
research registry / reports
  ↓ evidence only
promotion artifact
  ↓ operator review
approved paper profile
  ↓ profile/env diff + paper observation
approved live-dry-run profile
  ↓ live readiness + explicit arming
small-live profile
```

The fix must **not** weaken existing live safety boundaries, arming checks, ledger/reconciliation protections, SELL authority behavior, or research/live separation.

## Hypotheses to verify or falsify

Treat these as hypotheses, not facts.

### Hypothesis A

There is no complete approved diff/apply contract between research promotion artifacts and paper/live runtime configuration.

Verify whether the repo already has a mechanism that compares promotion/candidate profile values against the actual runtime strategy settings, including:

* strategy name
* market
* interval
* all strategy parameters
* cost model / fee / slippage assumptions
* manifest hash
* dataset content hash
* candidate profile hash
* regime policy presence and validity
* profile mode compatibility: paper vs live

Falsify this hypothesis if startup/preflight already reads the promoted artifact/profile, recomputes relevant hashes, compares the complete runtime configuration, and fails closed on mismatch with tests.

### Hypothesis B

Runtime strategy parameter authority may still be env-centered, while the promotion artifact is only evidence.

Check whether env variables such as `STRATEGY_NAME`, `SMA_SHORT`, `SMA_LONG`, `SMA_FILTER_*`, `SMA_COST_EDGE_*`, `ENTRY_EDGE_BUFFER_RATIO`, `STRATEGY_MIN_EXPECTED_EDGE_RATIO`, and `STRATEGY_EXIT_*` can diverge from the promoted candidate/profile without startup failure or BUY fail-closed behavior.

Falsify this if the existing profile loader already makes the approved profile the authoritative source for runtime strategy construction and prevents mismatching env overrides.

### Hypothesis C

The research/live separation is intentional and correct, but the manual approval lifecycle lacks explicit state objects and transition rules.

Check whether the following are represented as distinct schemas/artifacts with enforced transitions:

* research candidate
* promotion artifact
* approved paper profile
* paper observation/validation report
* approved live-dry-run profile
* live readiness report
* small-live/live-armed profile

Falsify this if the repo already has enforced schemas and CLI flows for these transitions.

### Hypothesis D

Regime policy may have started moving toward a fail-closed runtime gate, but the same pattern may not yet cover the full candidate/profile contract.

Check whether missing/invalid regime policy is blocked in live/replay BUY entry paths, and then verify whether the same level of verification exists for the rest of the promoted candidate contract.

Falsify this if all promoted candidate fields, not only regime policy, are already verified before paper/live runtime execution.

## Required design direction

Prioritize structural prevention over small patches.

Implement a verified manual approval flow, not automatic live updates.

### 1. Add or strengthen an approved profile schema

Introduce or extend an immutable approved profile artifact that can represent at least:

* profile schema version
* profile mode: `paper`, `live_dry_run`, `small_live`, or equivalent existing terminology
* source promotion artifact path or source id
* source promotion content hash
* candidate profile hash
* manifest hash
* dataset content hash
* repository version / commit
* strategy name
* market
* interval
* strategy parameter values
* cost model / fee / slippage assumptions
* regime policy, including classifier/version compatibility where available
* generated_at for operator context
* parent profile hash, if promoted from a previous profile
* profile content hash

Follow the repo’s existing deterministic hashing conventions. If existing research reports exclude `generated_at` from deterministic `content_hash`, keep that principle for approved profiles as well.

### 2. Add CLI support for profile generation, diff, and verification

Add commands using the repo’s existing CLI style. Names can be adjusted to match the codebase, but the functionality must exist.

Target commands:

```bash
uv run bithumb-bot profile-generate \
  --promotion DATA_ROOT/paper/reports/research/<experiment>/promotion_<candidate>.json \
  --mode paper \
  --out DATA_ROOT/paper/profiles/<profile_id>.json
```

```bash
uv run bithumb-bot profile-diff \
  --profile DATA_ROOT/paper/profiles/<profile_id>.json \
  --target-env BITHUMB_ENV_FILE_PAPER
```

```bash
uv run bithumb-bot profile-verify \
  --profile DATA_ROOT/paper/profiles/<profile_id>.json \
  --env BITHUMB_ENV_FILE_PAPER
```

Requirements:

* `profile-generate` must not modify `.env`, `BITHUMB_ENV_FILE_PAPER`, `BITHUMB_ENV_FILE_LIVE`, or secrets.
* `profile-diff` must emit both operator-readable output and machine-readable JSON, if the repo has an established JSON output convention.
* `profile-verify` must fail non-zero on hash mismatch, schema mismatch, missing required fields, profile mode mismatch, or strategy/runtime-relevant env mismatch.
* The CLI must be safe to run in CI without exchange credentials.

### 3. Make the approved profile the runtime verification anchor

Do not necessarily remove existing env support immediately, but change the model so env selects an approved profile rather than silently becoming the strategy parameter authority.

At runtime/preflight, when a strategy candidate/profile path is configured:

* load the approved profile or promotion artifact
* validate schema
* recompute content/profile hashes
* compare runtime strategy settings against profile values
* verify market and interval
* verify cost model assumptions
* verify regime policy presence and compatibility where applicable
* record profile hash, promotion hash, manifest hash, and dataset hash in startup audit/decision telemetry
* fail closed on live armed mismatch

For live armed mode:

* missing approved live-compatible profile must prevent live BUY submission
* profile/env mismatch must fail startup or fail closed before BUY submission
* paper profile must not be accepted as a live armed profile unless an explicit promotion step exists and records parent hash
* invalid or missing regime policy must fail closed for BUY
* SELL/exit authority must not be blocked solely because candidate regime policy is missing or invalid, unless existing repo safety rules already require that

For paper/replay mode:

* prefer the same verification path for reproducibility
* if the repo intentionally allows warnings instead of hard failure, document that policy and ensure mismatch appears in decision context/audit output

### 4. Add transition rules instead of direct auto-apply

Implement explicit state transitions where missing.

Examples:

```text
promotion artifact → approved paper profile
approved paper profile + paper validation evidence → live-dry-run profile
live-dry-run profile + live readiness evidence → small-live profile
```

Each transition should:

* preserve parent hash
* recompute child profile hash
* verify parent schema and mode
* reject mode skipping unless explicitly allowed by existing runbook policy
* require paper validation evidence before live profile creation, if compatible with current repo design
* produce an operator-reviewable artifact

Do not automatically arm live trading.

## Required tests

Add unit/integration tests consistent with the repo’s test style. At minimum, cover:

1. Tampering with `candidate_profile_hash` causes profile generation or verification to fail.
2. Promotion artifact with mismatched backtest/walk-forward parameter values is rejected, if this path exists.
3. `profile-diff` detects env drift, e.g. profile has one `SMA_SHORT` and env has another.
4. `profile-verify` exits/fails on strategy parameter mismatch.
5. Live armed startup or BUY preflight fails closed when approved profile is missing.
6. Live armed startup or BUY preflight fails closed when profile/env mismatch exists.
7. Missing or invalid regime policy fails closed for BUY where the repo’s policy requires it.
8. SELL/exit behavior is not blocked solely by candidate regime policy failure.
9. Paper profile cannot be used directly as a live armed profile without explicit promotion.
10. Live profile creation requires paper validation evidence or an explicit documented exception.
11. Startup audit or decision telemetry includes profile hash, promotion hash, manifest hash, and dataset hash.
12. Existing live preflight, arming, run lock, reconciliation, duplicate order, ledger, and storage-root safety tests still pass.

Also add regression tests for old artifacts if the repo supports historical artifact formats:

* old artifact missing regime policy
* old artifact missing profile hash
* old artifact with `generated_at` changed but deterministic hash unchanged
* corrupted artifact content hash

## Observability requirements

Add or strengthen audit/telemetry output so an operator can answer:

* Which approved profile is the runtime using?
* Which promotion artifact produced it?
* Which candidate profile hash is active?
* Which manifest hash and dataset content hash are active?
* Which strategy parameters are active?
* Did runtime env match the approved profile?
* If not, which fields differ?
* Was BUY blocked because of profile mismatch, mode mismatch, hash mismatch, missing regime policy, or another reason?

Use the repo’s existing logging and decision-context conventions. Avoid leaking secrets.

## Recovery and rollback requirements

Provide a safe recovery path:

* If profile verification fails, runtime must stop before live BUY or enter existing safe no-buy mode.
* Operators must be able to revert to the previous approved profile by changing only the profile path or equivalent non-secret selector.
* Failed profile generation must leave no partial approved artifact, or must write atomically.
* CLI errors must be deterministic and machine-readable enough for CI.
* The implementation must not mutate live env files automatically.

## Scope constraints

Do not:

* connect the research registry directly to live execution
* auto-copy research parameters into live env files
* auto-arm live trading
* weaken live arming checks
* weaken DB/storage-root isolation
* weaken lot-native SELL authority or ledger/reconciliation protections
* require exchange credentials for profile diff/generation/verification tests
* introduce nondeterministic hashes that include timestamps unless explicitly separated from content hash

Do:

* preserve the research/live separation
* make promotion artifacts and approved profiles first-class verification anchors
* prefer fail-closed behavior for live armed paths
* keep paper/replay reproducible
* update docs/runbooks to describe the new workflow exactly
