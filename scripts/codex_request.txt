## Mission

Do **not** solve this by flattening the user’s current BTC position, bypassing halt gates, using `resume --force`, deleting lots, or adding another manual DB repair path.

The goal is to evolve the bot into a more robust, reproducible, and operational automated trading experiment system that can safely resume while carrying an already-accounted, broker-confirmed open position.

The core principle for this patch:

```text
Open exposure present
≠ unsafe state
≠ must flatten
≠ cannot run
```

A normal automated trading bot must be able to restart with an existing canonical position and continue managing that position. It may block additional BUY entries if needed, but it should not require manual flattening before it can run.

### Hypothesis A — stale `RISK_STATE_MISMATCH` halt latch is not revalidated after successful repair

The EC2 state suggests the bot still carries an old `RISK_STATE_MISMATCH` halt reason even after the underlying mismatch was repaired.

The repository appears to have stale halt clearing paths for some reconcile failures, startup safety gate cases, and live broker errors, but it may not safely clear `RISK_STATE_MISMATCH` after verified accounting/broker/projection convergence.

Verification target:

```text
Inspect engine.py and runtime_state.py.
Find where halt_state_unresolved is set and cleared.
Find whether RISK_STATE_MISMATCH can be cleared after:
  - last_reconcile_status=ok
  - last_reconcile_reason_code=POSITION_AUTHORITY_REBUILD_COMPLETED
  - startup_gate_reason=None
  - accounting_projection_ok=true
  - unresolved_fee_state=false
  - unresolved_open_order_count=0
  - recovery_required_count=0
  - broker_portfolio_converged=true
  - lot_projection_converged=true
  - broker snapshot fresh/current
  - open exposure exists but is canonical/accounted
```

Expected if hypothesis is confirmed:

```text
RISK_STATE_MISMATCH remains non-clearable even when the current evidence proves the mismatch is gone.
```

Falsification condition:

```text
If a current broker/portfolio mismatch still exists, active fee pending exists, unresolved orders exist, projection drift exists, broker evidence is stale/missing, or risk loss limits are active, then the halt must remain.
```

### Hypothesis B — `OPEN_EXECUTABLE` is treated only as a resume blocker, not as a valid position-management state

An automated trading bot normally creates open exposure after a BUY. Open exposure is not inherently a broken state.

The current behavior appears to collapse:

```text
→ flatten recommended or implied
```

instead of distinguishing:


Expected if hypothesis is confirmed:

Falsification condition:

```text
If the strategy intentionally forbids position carry across restart, that policy must be explicit, configurable, documented, and tested. Otherwise this is a runtime policy bug.
```

### Hypothesis C — operator guidance conflates normal SELL/exit management with emergency flattening

## Required design changes

### 1. Add stale `RISK_STATE_MISMATCH` halt revalidation and safe clear policy

`RISK_STATE_MISMATCH` must not be an irreversible latch once the underlying mismatch has been repaired.

Add or modify a safe clear path so that a stale `RISK_STATE_MISMATCH` can be cleared when current evidence proves the system is coherent.

Candidate clear conditions:

```text
halt_reason_code == RISK_STATE_MISMATCH
halt_state_unresolved == true
last_reconcile_status == ok
last_reconcile_reason_code in {
  POSITION_AUTHORITY_REBUILD_COMPLETED,
  FEE_PENDING_ACCOUNTING_REPAIR_COMPLETED,
  MANUAL_RECOVERY_COMPLETED,
  RECONCILE_OK
}
startup_gate_reason is None
accounting_projection_ok == true
unresolved_fee_state == false
unresolved_open_order_count == 0
recovery_required_count == 0
broker_portfolio_converged == true
lot_projection_converged == true
broker snapshot is fresh/current
```

Important: canonical open exposure must **not** by itself prevent stale halt clearing.

Continue to block if any of these are true:

```text
current broker/portfolio mismatch
active fee pending
unresolved order
remote open order
projection drift
broker snapshot missing/stale
risk loss limit breach
kill switch
operator panic stop
source conflict
submit unknown
```

Expected result:

```text
A stale RISK_STATE_MISMATCH from an earlier mismatch is cleared after verified repair/rebuild convergence.
```

Do not clear true risk halts such as kill switch, daily loss limit, position loss limit, panic stop, or currently unresolved broker mismatch.

### 2. Add explicit position-management resume mode for canonical open exposure

Introduce or expose a policy state equivalent to:

```text
OPEN_POSITION_READY
```

For clean canonical open exposure:

```text
canonical_state=OPEN_EXECUTABLE
accounting_projection_ok=true
broker_portfolio_converged=true
lot_projection_converged=true
unresolved_fee_state=false
open_orders=0
recovery_required=0
broker snapshot fresh/current
no active risk breach
```

Expected behavior:

```text
run_loop_allowed=true
position_management_allowed=true
exit_allowed=true
new_entry_allowed=false or strategy/risk-controlled
flatten_primary_recommendation=false
recommended_action=resume_position_management
recommended_command=uv run python bot.py resume
```

The exact naming can follow the repository’s style, but the semantics must be clear.

Do not treat open exposure itself as broken when it is:

```text
broker-confirmed
portfolio/replay-confirmed
lot-projection-confirmed
fee-clean
order-clean
risk-clean
```

### 3. Separate new-entry permission from existing-position management

The bot must be able to run to manage an existing position even if new BUY entries are blocked.

Implement or clarify separate flags:

```text
run_loop_allowed
position_management_allowed
exit_allowed
new_entry_allowed
emergency_flatten_allowed
flatten_primary_recommendation
```

Expected for canonical open exposure:

```text
run_loop_allowed=true
position_management_allowed=true
exit_allowed=true
new_entry_allowed=false unless strategy/risk policy permits pyramiding
emergency_flatten_allowed=true but not primary
flatten_primary_recommendation=false
```

Expected for flat clean state:

```text
run_loop_allowed=true
position_management_allowed=false or irrelevant
new_entry_allowed=true
exit_allowed=false or irrelevant
```

Expected for forensic blocked state:

```text
run_loop_allowed=false
position_management_allowed=false

```

### 4. Demote `flatten-position` to emergency-only primary recommendation

`flatten-position` must not be the primary recommendation merely because canonical open exposure exists.

It may be primary only for cases such as:

```text
kill switch
position loss limit
daily loss risk policy
explicit operator flatten request
panic/emergency exposure reduction
true unsafe exposure where normal position management cannot run and emergency reduction is required
```

It must not be primary for:

```text
canonical open exposure
accounting evidence reliable
broker/portfolio/projection converged
no active risk breach
no unresolved order
no active fee issue
```

Operator guidance should say something like:

```text
Canonical open exposure is present and accounted.
Resume in position-management mode is allowed.
New entries are blocked until the existing position is closed or risk policy allows additional entries.
Emergency flatten is available but not recommended as the primary action.
```

If blocked, guidance must explain the real blocker:

```text
position-management resume is blocked because:
  stale halt latch unresolved
  missing broker snapshot
  active fee issue
  unresolved order
  current broker mismatch
  active risk breach
```

not merely:

```text
open exposure exists
```

### 5. Keep safety gates strict for truly unsafe open exposure

Do not loosen gates globally.

The bot must still block run/resume if open exposure is accompanied by:

```text
broker/portfolio mismatch
active fee pending
unresolved local order
stray remote open order
submit unknown
projection drift
broker snapshot stale/missing
risk loss limit breach
kill switch
panic stop
source conflict
```

In these cases, `recovery-report`, `repair-plan`, `restart-checklist`, and `health` should point to diagnosis/repair, not position-management resume.
