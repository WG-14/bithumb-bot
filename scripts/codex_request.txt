## Incident context

On EC2, live trading repeatedly entered this failure cycle after real Bithumb BUY/SELL fills:

```text
Bithumb order fills successfully
→ local fee validation classifies the fill as fee_pending
→ local fills/trades/portfolio/lot accounting becomes incomplete or inconsistent
→ broker/local portfolio mismatch appears
→ startup/resume gates block live
→ recovery suggests flatten or manual repair
→ flatten itself creates another real fill
→ that fill goes through the same fee/accounting failure path
→ operator manually repairs fee/principal/lot state and sometimes bot_health
```

Runtime evidence from the incident:

```text
Actual Bithumb mobile-app fee rate: 0.04% = 0.0004 = ~4 bps

DB-observed examples:
flatten SELL fee_bps = 3.999784
latest BUY fee_bps   = 3.999902
previous SELL        = 3.998178
previous BUY         = 3.999148

Resolved setting during incident:
LIVE_FEE_RATE_ESTIMATE = 0.0025
LIVE_FILL_FEE_ALERT_MIN_NOTIONAL_KRW = 10000.0
```

The likely direct trigger was:

```text
configured expected live fee rate = 0.0025
actual Bithumb paid fee rate      = 0.0004
→ expected_fee_rate_mismatch
→ old behavior: order_level_candidate / fee_pending
→ accounting/recovery cascade
```

But do **not** reduce the fix to “set `LIVE_FEE_RATE_ESTIMATE=0.0004`.” That is required operational hygiene, not a sufficient repository-level fix.

The higher-priority structural issue is:

```text
The system must not let a local estimated fee rate override coherent exchange-provided paid_fee evidence as a live settlement/accounting gate.

For a single-fill order where exchange order-level paid_fee is present, non-negative, non-zero for material notional, and executed volume/funds/identifiers match, the fill should be accounting-complete even if the configured fee estimate is wrong.

expected_fee_rate_mismatch must remain visible as warning/diagnostic/anomaly, but must not by itself force fee_pending, principal-pending accounting, broker/local drift, lot projection drift, halt state, or flatten recommendations.
```

## Required design priorities

### 1. Make fee authority explicit and correctly ordered

If current `FeeEvaluation` still relies mainly on loose string combinations such as:

```text
fee_status
fee_source
fee_confidence
provenance
reason
checks
```

then improve it where feasible without over-scoping.

Preferred direction:

```python
fee_authority = "exchange_order_paid_fee_single_fill"
accounting_decision = "finalize"
diagnostic_flags = ["expected_fee_rate_mismatch"]
```

At minimum, make this semantic ordering unambiguous in code and tests:

```text
exchange trade-level fee                     → authoritative
exchange order-level paid_fee single-fill     → authoritative/validated if qty/funds/ids match
operator_confirmed fee                       → authoritative
configured fee estimate                      → planning/diagnostic/anomaly baseline only
unknown/missing/unparseable fee              → pending/blocked depending on safety policy
```

Do not over-scope this into a full accounting rewrite unless a small, safe abstraction already exists. But if the code is still forcing reporting/recovery logic to parse `reason` strings to infer authority, improve that.

### 2. Preserve the core invariant across every path

The invariant is:

```text
single-fill exchange paid_fee + matching qty/funds/ids
→ accounting_complete / fee_finalized

expected fee-rate mismatch
→ warning/anomaly/diagnostic
→ not fee_pending by itself
→ not principal_applied_fee_pending by itself
→ not startup/restart blocker by itself
```

Unsafe cases must remain pending/blocked:

```text
missing/unparseable paid_fee on material notional
negative fee
material zero paid_fee
multi-fill order-level fee ambiguity
executed_volume mismatch
executed_funds mismatch
identifier mismatch, if current policy requires identifiers
```

### 3. Make live fee-rate drift visible on operational surfaces

Even after the fee-authority fix, `LIVE_FEE_RATE_ESTIMATE=0.0025` while actual fees are `0.0004` is still operationally dangerous for strategy planning, sizing, diagnostics, and incident analysis.

Add or improve operator-facing output in:

```text
fee-diagnostics
health
recovery-report
restart-checklist
```

At minimum, ensure `fee-diagnostics` contains the full drift summary, and at least one restart/recovery surface points the operator to that drift when present.

Required fields or equivalents:

```text
configured_fee_rate
configured_fee_bps
observed_fee_bps_median
observed_fee_sample_count
configured_minus_observed_bps
fee_rate_deviation_pct
expected_fee_rate_warning_count
fee_pending_count
fee_pending_accounting_repair_count
position_authority_repair_count
diagnostic_only_vs_startup_blocking
startup_impact
```

Use a strong warning if hard-blocking startup would be too disruptive. But do not keep this invisible.

Suggested operator surfaces:

```text

  WARN fee-rate drift: configured=25bps observed≈4bps diagnostic_only no active fee_pending

```

### 4. Recovery must not default to flatten for accounting-root-cause incidents

Flatten made this incident worse by creating additional real fills that went through the same broken fee path.

Modify recovery/reporting guidance if needed:

```text
If accounting/fee-authority root cause is unresolved:
  - suppress flatten as the primary recommendation
  - do not suggest additional orders as the first action
  - recommend forensic fee/accounting diagnosis
  - collect broker order/fill evidence
  - propose an auditable repair plan if needed

If actual executable market exposure exists and accounting evidence is reliable:
  - flatten may be recommended as market-risk mitigation
```

This is a priority. Do not make flatten the default response to accounting drift.

Search explicitly for recovery output and recommendation logic using terms such as:


If no explicit accounting-root-cause mode exists, add one with tests.

### 5. Repairability must move away from manual DB edits

Do not add manual DB update instructions or another opaque repair shortcut.

Any repair-related change should move toward:


### 6. Treat `open_position_lots` as a rebuildable projection

Do not rewrite the full accounting engine unless necessary. But ensure the direction is clear:

```text

  fills + trades + fee finalization/adjustment events + external adjustments

projection:
  open_position_lots, deterministic and rebuildable under safe preconditions
```

Make sure one fee warning cannot drift lot projection and force repeated `rebuild-position-authority`.
