# Primary objective

Convert replay/sweep from “unbounded DB-backed experiment execution” into “bounded, reusable, operator-safe experiment execution.”

This patch should prioritize:

```text
1. explicit candle dataset boundary
2. one-time candle loading for sweep
3. max-candles / allow-full-history guard
4. query/index optimization
5. execution plan observability
```

Do **not** implement PnL/backtest/fill simulation.

# Part 1: Introduce a bounded CandleReplayDataset

In `src/bithumb_bot/strategy_replay.py`, add a dataset boundary.

Suggested type:

```python
@dataclass(frozen=True)
class CandleReplayDataset:
    pair: str
    interval: str
    candles: tuple[tuple[int, float], ...]
    from_ts_ms: int | None
    to_ts_ms: int | None
    through_ts_ms: int | None
    max_candles: int | None
    source: str = "sqlite:candles"
```

Adjust naming if repo style suggests better names.

## Add a loader

Replace or wrap `_load_replay_candles()` with a public/internal loader such as:

```python
def load_replay_candles(
    conn: sqlite3.Connection,
    *,
    pair: str,
    interval: str,
    from_ts_ms: int | None = None,
    to_ts_ms: int | None = None,
    through_ts_ms: int | None = None,
    max_candles: int | None = None,
) -> CandleReplayDataset:
    ...
```

Requirements:

* read only from `candles`
* support `from_ts_ms`
* support `to_ts_ms`
* support `through_ts_ms`
* support `max_candles`
* return candles in ascending timestamp order
* avoid unbounded full-history scans when `max_candles` is provided
* keep behavior compatible for existing replay tests where feasible

## Query design

For bounded latest-N candles, prefer an efficient query shape:

```sql
SELECT ts, close
FROM (
    SELECT ts, close
    FROM candles
    WHERE pair=? AND interval=?
      AND ts <= ?
    ORDER BY ts DESC
    LIMIT ?
)
ORDER BY ts ASC
```

If `from_ts_ms` is supplied, include it in the inner WHERE too.

If no `max_candles` is supplied, you may keep ascending full-window query, but this must be guarded at CLI level in live mode.

Be careful with `to_ts_ms` and `through_ts_ms`; use the stricter upper bound.

# Part 4: Add execution plan / budget observability

Add a small plan type, preferably in `strategy_sweep.py`:

```python
@dataclass(frozen=True)
class StrategySweepExecutionPlan:
    grid_count: int
    candle_count: int
    estimated_operations: int
    max_candles: int | None
    full_history: bool
    allowed: bool
    block_reason: str | None
```

Implement a helper such as:

```python
def build_strategy_sweep_execution_plan(
    *,
    grid: StrategySweepGrid,
    candle_count: int,
    max_candles: int | None,
    from_ts_ms: int | None,
    to_ts_ms: int | None,
    through_ts_ms: int | None,
    mode: str,
    allow_full_history: bool,
) -> StrategySweepExecutionPlan:
    ...
```

The plan should:

* count only valid configs where `short_n < long_n`
* estimate operations as `grid_count * candle_count`
* detect full-history mode when no window and no max-candles are provided
* block unsafe live full-history runs unless `allow_full_history=True`
* be deterministic and serializable

This plan is not a trading recommendation. It is an execution safety/budget object.
